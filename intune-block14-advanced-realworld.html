TOPIC 116
Zero Trust Model and Intune's Role
How Intune is the enforcement engine of a Zero Trust architecture
What Is Zero Trust?

Zero Trust is a security model based on one principle: never trust, always verify. Traditional security assumed everything inside the corporate network was safe. Zero Trust assumes breach — every access request must be verified regardless of where it originates, even from inside the network perimeter.

Microsoft's Zero Trust framework has three pillars: Verify explicitly (always authenticate and authorize based on all available data points), Use least privilege access (limit user access with just-in-time and just-enough-access), and Assume breach (minimize blast radius and segment access).

Intune's Role in Zero Trust

Intune is the device compliance enforcement engine of Zero Trust. It answers the critical question that Conditional Access asks on every sign-in: "Is this device healthy and compliant?" Without Intune, Conditional Access can verify user identity (MFA, risk) but cannot verify device health.

🔐
Identity Verification
Azure AD handles user identity via MFA, risk signals, and Conditional Access policies. Intune provides the device identity component.
💻
Device Health Signal
Intune compliance policies define "healthy device" — BitLocker on, OS patched, no malware, password set. This signal flows to Conditional Access in real time.
📦
App Protection
MAM policies enforce Zero Trust on unmanaged (BYOD) devices — corporate data stays within managed apps even without full device enrollment.
🛡️
Endpoint Security
Intune deploys Defender for Endpoint, BitLocker, ASR rules, and security baselines — hardening devices against the threats Zero Trust assumes are present.
The Zero Trust Access Decision Flow

Every resource access attempt flows through this chain: User authenticates (Azure AD + MFA) → Azure AD checks all Conditional Access policies → CA asks Intune: "Is device compliant?" → Intune responds with compliance state → CA grants or blocks access → Access token issued with appropriate scope. Intune's compliance answer is the device gate in this chain.

🌐
Real-World Zero Trust Implementation

Company: 2,000-employee financial services firm. After a ransomware incident, CISO mandated Zero Trust within 6 months.

Intune's role: (1) Enrolled all Windows/Mac/mobile devices. (2) Compliance policies requiring BitLocker, Defender AV active, OS patched within 30 days, screen lock PIN. (3) MDE integration — devices with active threats automatically marked non-compliant. (4) CA policy: all Microsoft 365 access requires compliant device. BYOD users get MAM-only access to Outlook/Teams with app protection policies. (5) Security baselines deployed to harden all endpoints.

Result: Any device compromised by malware is automatically flagged as non-compliant by MDE → Intune marks it non-compliant → CA blocks the device from all M365 services within minutes — without any manual intervention.

TOPIC 117
Intune + Azure AD Privileged Identity Management
Just-in-time admin access for Intune roles — removing standing privileges
What Is PIM?

Azure AD Privileged Identity Management (PIM) provides just-in-time (JIT) privileged access — instead of permanently assigning admin roles, users are eligible for a role and must activate it when needed. Activation requires MFA, approval, and is time-limited (e.g., 4 hours). After the window expires, admin rights are automatically revoked.

Why PIM Matters for Intune Administration

Permanent admin standing is one of the highest-risk configurations in any tenant. An Intune Administrator account with standing access, if compromised, lets an attacker immediately wipe all devices, push malicious scripts, or exfiltrate data. PIM eliminates standing Intune admin access — forcing elevation only when needed, with audit trail and approval.

Configuring PIM for Intune Roles
  1. 1
    Open Azure AD PIM: Azure AD → Identity Governance → Privileged Identity Management → Azure AD Roles
  2. 2
    Find the Intune Administrator role. Click it → Settings. Configure: maximum activation duration (recommend 4–8 hours), require MFA on activation, require justification, require approval if needed for high-risk roles.
  3. 3
    Add eligible assignments. Under Assignments → Add assignments → select Intune admins as "Eligible" (not "Active"). They now have zero standing permissions.
  4. 4
    Admin activation flow: Admin goes to PIM portal → My Roles → activates Intune Administrator → provides MFA + justification → gets 4-hour Intune admin access → role expires automatically.
  5. 5
    Configure alerts and notifications. PIM sends email when a role is activated. Set up alerts for unusual activation patterns (outside business hours, unexpected users).
PIM for Intune RBAC Custom Roles

PIM works for Azure AD roles (like Intune Administrator) but not natively for Intune's internal RBAC custom roles. For Intune-specific roles (Application Manager, Help Desk Operator), use Azure AD Groups with PIM: put the group as the role assignment in Intune, then make users eligible for the group via PIM Group Management. When they activate PIM group membership, they get the Intune custom role access.

🎤 Interview Questions — Topic 117
What is PIM and why would you use it with Intune?
PIM is Azure AD's just-in-time privileged access management. Instead of permanently assigning admin roles, you make users "eligible" — they activate the role only when needed, with MFA, justification, and a time limit. For Intune, it means no one has standing Intune Administrator access. If an Intune admin account is compromised, the attacker gets zero privileges until they can also complete PIM activation — which requires MFA and triggers immediate alerts. It dramatically reduces the blast radius of a compromised admin account.
How would you apply PIM to Intune's internal RBAC custom roles?
PIM natively covers Azure AD roles but not Intune's internal custom roles directly. The workaround is to use PIM-managed groups: assign the Intune custom role to an Azure AD group rather than directly to users. Then use PIM for Groups to make users eligible for that group. When they activate via PIM, they join the group temporarily and get the Intune role access. When activation expires, they're removed from the group and lose the role automatically.
TOPIC 118
Intune in Hybrid Environments — Edge Cases
The real-world complexity of running Intune alongside on-premises AD, SCCM, and GPO
Hybrid Environment Reality

Most enterprise Intune deployments aren't greenfield cloud-only — they coexist with on-premises Active Directory, Group Policy, SCCM/ConfigMgr, and legacy on-prem apps. This creates edge cases that don't appear in clean lab environments. Understanding these is what distinguishes real enterprise engineers.

Edge Case 1: GPO and MDM Policy Conflict

When a device is Hybrid AADJ, both Group Policy and Intune MDM write to the device. For the same registry key, the last writer wins — and GPO often wins because it applies at logon/background refresh, which may happen after Intune sync. The Intune portal shows "Succeeded" (MDM sent the setting successfully) but the device enforces the GPO value.

⚠️ Critical Nuance

Use MDMWinsOverGP CSP policy to make MDM settings take precedence over GPO for supported settings. Deploy via OMA-URI: ./Device/Vendor/MSFT/Policy/Config/ControlPolicyConflict/MDMWinsOverGP with value Integer 1. This resolves the conflict in Intune's favor for Settings Catalog policies.

Edge Case 2: Hybrid AADJ Requires Line-of-Sight for Initial Join

Hybrid Azure AD Join requires the device to reach an on-premises Domain Controller during the join process to complete the computer object sync. Devices that have never been on the corporate network (or VPN) since being prepared cannot complete Hybrid AADJ. This is a common failure mode for remote workers receiving new laptops by mail.

⚠️ Edge Case

Remote workers with new laptops cannot complete Hybrid AADJ without VPN or direct DC line-of-sight. Solution: either deploy Always On VPN before the join completes (complex), or switch to Azure AD Join only — which has no on-premises dependency. Many organizations use this edge case as justification to move from Hybrid AADJ to pure AADJ.

Edge Case 3: Stale Objects in Both AD and Azure AD

In hybrid environments, device objects exist in both on-premises AD and Azure AD. When a device is wiped and re-enrolled, a new Azure AD object is created but the old one may persist. Similarly, the on-premises AD computer object from the original Hybrid AADJ may conflict with the new one. This causes duplicate device records, policy assignment confusion, and BitLocker key mismatches.

Fix: Implement a device offboarding process that deletes objects from both AD and Azure AD when devices are retired. Use Intune's retire/wipe action AND delete the Azure AD object AND delete the on-premises AD computer object.

Edge Case 4: Intune Policies Don't Apply Until User Logs In (Hybrid AADJ)

On Hybrid AADJ devices, some Intune user-targeted policies only apply after the user logs in with their Azure AD credentials for the first time. Before that first login, the device only has the computer object token — not the user token that Intune uses for user-targeted policies. Device-targeted policies apply fine; user-targeted policies are delayed.

Edge Case 5: Certificate Autoenrollment Conflict

In hybrid environments, Group Policy Certificate Autoenrollment (from on-premises CA) and Intune SCEP/PKCS certificate profiles may both try to enroll certificates for the same template. This can result in duplicate certificates, unexpected certificate replacements, or enrollment failures. Audit which certificate enrollment mechanism owns which template before deploying Intune cert profiles.

🎤 Interview Questions — Topic 118
A configuration profile shows "Succeeded" in Intune but the setting isn't applied on Hybrid AADJ devices. What's happening?
Most likely a GPO is overriding the Intune MDM setting. On Hybrid AADJ devices, both Group Policy and MDM can write the same registry value. GPO often applies after Intune sync and overwrites it — Intune shows "Succeeded" because MDM successfully delivered the setting, but GPO reverted it. The fix is to deploy the MDMWinsOverGP CSP policy, which instructs the OS to prefer MDM settings over GPO for supported policies. Alternatively, block GPO inheritance for the OU containing cloud-managed devices.
Why might a remote worker's brand new laptop fail to complete Hybrid Azure AD Join?
Hybrid AADJ requires the device to communicate with an on-premises Domain Controller to sync the computer object from AD to Azure AD. A remote worker with no corporate network access and no VPN connection cannot complete this sync — the join process hangs or fails. Solutions: require the device to be on VPN before setup, ship devices to office first, or reconsider the join type and switch to pure Azure AD Join which has no on-premises dependency and works natively for remote workers.

Intune Update Rings — Feature & Quality Updates
Controlling Windows Update deployment at enterprise scale with rings and policies
Why Update Rings?

Deploying updates to 10,000 devices simultaneously is dangerous — a bad update could break every device at once. Update Rings solve this by deploying updates in waves: a small pilot group gets updates first, IT validates, then broader groups receive the update progressively. Problems are caught early and blasted radius is limited.

Types of Windows Updates in Intune
🔄
Update Rings (WUfB)
Control Quality Updates (Patch Tuesday) and Feature Updates (annual Windows versions) via Windows Update for Business policies. Set deferral periods, maintenance windows, and restart behavior.
🪟
Feature Update Policies
Pin devices to a specific Windows version (e.g., 22H2). Prevents automatic feature updates from installing until IT is ready. Requires Windows Update for Business reports for monitoring.
🛡️
Quality Update Policies
Deploy specific cumulative update KB numbers to targeted device groups — useful for emergency patches or testing a specific update on pilot devices before broad rollout.
🔁
Driver Update Policies
Control driver updates separately from OS updates. Approve or pause specific driver updates. Prevents problematic driver updates from auto-installing.
Designing a 4-Ring Update Strategy
RingGroupQuality Update DeferralFeature Update DeferralPurpose
Ring 0 — PilotIT admins, volunteers (1–2% of devices)0 days0 daysFirst to receive everything. IT validates before broader rollout.
Ring 1 — EarlyTech-savvy users, dev teams (10–15%)7 days30 daysEarly adopters. Broader real-world validation.
Ring 2 — StandardGeneral staff (70–80%)14 days60 daysMain fleet. Updates after validation complete.
Ring 3 — CriticalExecutives, VIPs, OT-adjacent (5–10%)21 days90 daysLatest updates. Maximum validation time. Never disrupted during business hours.
Key Update Ring Settings to Know
  • Deferral period: Days to wait after Microsoft releases the update before it installs on devices in this ring. Gives time for Microsoft to fix issues post-release.
  • Maintenance window: Restricts update installation and reboots to specific hours (e.g., 2 AM – 6 AM). Prevents mid-workday disruption.
  • Active hours: Windows won't reboot for updates during defined active hours. Separate from maintenance window — active hours prevent reboot, maintenance window defines when installs can happen.
  • Deadline: Forces installation even if device has been avoiding updates. After deadline days, update installs regardless of active hours. Ensures security patches don't stay pending indefinitely.
  • Grace period: Days after deadline before a forced restart. Gives users warning time to save work before forced reboot.
  • Pause updates: Emergency pause button. If a bad update ships, you can pause it for up to 35 days across all rings without modifying deferral settings.
🔄
Real-World — Handling a Bad Patch Tuesday

Scenario: Microsoft releases KB5034441 in January. Two days later, reports emerge that it causes blue screens on devices with BitLocker enabled.

Response: IT immediately pauses updates on Ring 1, 2, and 3 via Intune (Update ring → Pause quality updates). Ring 0 (IT pilot) has 12 devices already updated — 3 have BSOD issues, confirming the problem. Pause holds for 7 days while Microsoft releases a fix. When Microsoft releases the revised KB, IT resumes Ring 1 deferral. The ring structure meant only 12 devices were affected instead of all 8,000.

Key insight: Ring 0 being IT staff means the people discovering the BSOD are also the people who can immediately pause rollout. This is why IT should always be in Ring 0 — they're the canary and the response team in one.

🎤 Interview Questions — Topic 119
Explain the difference between deferral, deadline, and grace period in Intune Update Rings.
Deferral is how many days after Microsoft releases an update before Intune even offers it to devices — this is your ring staging mechanism. Deadline is how many days after the update becomes available that Intune forces installation regardless of user activity — this ensures security patches don't stay pending forever. Grace period is how many days after the deadline before Intune forces a restart — it gives users warning time to save work. Together: deferral controls when the update arrives, deadline ensures it installs, and grace period manages the reboot.
A critical security patch has just been released. How do you get it deployed urgently while bypassing normal ring deferral?
Two options: first, use a Quality Update Policy to target a specific KB number directly to all device groups, bypassing Update Ring deferral — this is the fastest way to push an emergency patch to all rings simultaneously. Second, temporarily set deferral to 0 days on all rings and set a short deadline (e.g., 1 day). After the emergency patch is deployed, restore the normal deferral settings. Always communicate to users before forcing rapid restarts — especially for the Critical/VIP ring.
TOPIC 120
Windows Hello for Business via Intune
Passwordless authentication — deploying and managing WHFB at enterprise scale
What Is Windows Hello for Business?

Windows Hello for Business (WHFB) replaces passwords with strong two-factor authentication using a combination of device possession (the enrolled device with its TPM-bound key) and a user gesture (PIN, fingerprint, or face recognition). Unlike a password, the credential never leaves the device — it's asymmetric key-based and phishing-resistant by design.

Why WHFB Matters for Security
  • Phishing resistant: The private key never leaves the TPM. Even if a user is phished into entering a PIN on a fake site, the attacker gets nothing — the PIN is only meaningful on that specific device.
  • No password spray or credential stuffing: No shared password secret means these attack vectors are eliminated entirely for WHFB-enrolled devices.
  • MFA in a single gesture: Device possession (hardware key) + user gesture (PIN/biometric) = MFA built into the login flow without a second-step prompt.
  • Zero Trust aligned: WHFB credentials are device-specific — they fulfill the "verified identity + verified device" requirement of Zero Trust simultaneously.
Deploying WHFB via Intune

Method 1 — Intune WHFB Enrollment Policy (Tenant-wide):

Intune → Devices → Windows → Windows enrollment → Windows Hello for Business. This tenant-wide policy configures WHFB for all enrolled Windows devices. Settings: Enable WHFB (Required/Disabled/Not Configured), minimum PIN length (recommend 6+), biometrics (Enabled/Disabled), anti-spoofing (Enhanced), TPM requirement.

Method 2 — Settings Catalog / Identity Protection Profile:

For granular per-group control: Intune → Devices → Configuration profiles → Settings Catalog → Windows Hello for Business. This lets you target WHFB settings to specific device groups rather than tenant-wide.

Hybrid Key TrustHybrid AADJ, access to on-prem resources neededOn-prem AD, AAD Connect, schema update, DRS. More complex but uses WHFB for both cloud and on-prem auth.

Common WHFB Issues
  • TPM not ready: WHFB requires TPM 2.0. Devices with TPM 1.2 or no TPM cannot enroll. Check: Settings → Windows Security → Device Security → Security processor details.
  • Provisioning fails after Autopilot: If the ESP completes before the WHFB provisioning policy is delivered, WHFB setup might be skipped. Ensure the WHFB policy is assigned and synced before ESP completes, or trigger provisioning manually via: dsregcmd /forcerecovery
  • PIN reset not working: If a user forgets their PIN and the WHFB PIN reset service isn't configured, they may be locked out. Enable Microsoft PIN Reset Service in Intune WHFB policy settings.
🎤 Interview Questions — Topic 120
Why is Windows Hello for Business considered phishing resistant?
WHFB uses asymmetric key cryptography where the private key is stored in the device's TPM and never leaves it. The user's PIN or biometric only unlocks the key locally on that device — it's never transmitted to any server. A phishing site that tricks a user into entering their PIN gets nothing useful because the PIN is bound to a specific device's TPM. Without both the physical device AND the correct PIN/biometric gesture, authentication fails. This is fundamentally different from passwords, which are shared secrets that can be captured and replayed from anywhere.
What's the difference between WHFB cloud-only and hybrid key trust deployments?
Cloud-only WHFB is for pure Azure AD Joined devices — it's the simplest deployment, just needs Azure AD P1, Intune, and TPM 2.0. It provides passwordless authentication to cloud resources only. Hybrid key trust extends WHFB to Hybrid AADJ devices and enables passwordless authentication to both cloud AND on-premises resources using Kerberos. It requires on-premises AD, AAD Connect syncing, schema updates, and Azure AD Kerberos or ADFS. The complexity is significantly higher but is necessary when devices need to authenticate to on-prem file servers, printers, or legacy apps.
TOPIC 121
Certificate Authority Integration — NDES and SCEP Deep Dive
How Intune delivers certificates to devices using NDES, SCEP, and PKCS
Why Certificates via Intune?

Certificates enable: Wi-Fi authentication (802.1X EAP-TLS), VPN client authentication, S/MIME email signing/encryption, code signing, and device identity in Zero Trust architectures. Manually deploying certificates to thousands of devices is impossible — Intune automates certificate lifecycle (enrollment, renewal, revocation) at scale.

The Two Certificate Delivery Methods
🔑
SCEP (via NDES)
Simple Certificate Enrollment Protocol. Each device generates its own key pair and requests a certificate from the CA. Private key never leaves the device. Best for device identity certificates. Requires NDES server on-premises.
📜
PKCS (via PFX)
The CA generates the key pair and certificate, encrypts it, and Intune delivers the PFX to the device. Private key is generated centrally. Required for S/MIME (email certificates must be backed up). Requires PFX connector.
SCEP Certificate Delivery Architecture

The SCEP flow involves several components that must all be configured correctly:

  1. 1
    On-premises CA (Microsoft AD CS): Issues certificates based on templates. Must have a certificate template configured for device/user enrollment that Intune can use.
  2. 2
    NDES Server (Network Device Enrollment Service): An IIS-based service that acts as the SCEP endpoint. Intune connector is installed here. Must be internet-reachable (or accessible via proxy) for Intune-managed devices to reach it. Critical: NDES must use an SSL certificate from a public or device-trusted CA — not a self-signed cert.
  3. 3
    Intune Certificate Connector: Installed on the NDES server. Authenticates to Intune, receives SCEP challenge passwords, and brokers the certificate request between the device and the CA.
  4. 4
    Trusted Root Certificate Profile: Must be deployed to devices BEFORE the SCEP profile. Devices need to trust the CA root before they can validate and accept certificates issued by it.
  5. 5
    SCEP Certificate Profile: Configured in Intune with the NDES URL, certificate template name, key usage, SAN, and validity period. Assigned to device/user groups. Intune orchestrates the SCEP enrollment on the device.

🎤 Interview Questions — Topic 121
What's the difference between SCEP and PKCS certificate delivery in Intune?
With SCEP, the device generates its own key pair locally and sends a certificate signing request to the CA via the NDES server — the private key never leaves the device, which is more secure. With PKCS, the CA generates both the private key and the certificate, encrypts the PFX package, and Intune delivers it to the device. PKCS is required for S/MIME email certificates because the private key needs to be backed up (for email decryption if the device is lost). For device identity certificates — Wi-Fi, VPN, device auth — SCEP is preferred because the private key never leaves the TPM.
Walk through the components needed for SCEP certificate deployment via Intune.
Five components: first, an on-premises Microsoft AD CS Certificate Authority with a configured device/user certificate template. Second, an NDES server with IIS that acts as the SCEP endpoint — must have a valid SSL cert and be reachable by devices and Intune. Third, the Intune Certificate Connector installed on the NDES server. Fourth, a Trusted Root Certificate Profile deployed to devices — devices must trust the CA root before accepting its certificates. Fifth, the SCEP Certificate Profile in Intune with the NDES URL and template settings, assigned to device groups. The Trusted Root profile must be deployed and applied before the SCEP profile is processed.
TOPIC 122
Network Access Control (NAC) Integration
Using Intune compliance as a gate for network access via NAC solutions
What Is NAC and Why Does It Matter?

Network Access Control (NAC) ensures that only compliant, trusted devices can connect to the corporate network — wired or wireless. When integrated with Intune, NAC solutions can query Intune's real-time compliance state for a device before granting network access. Non-compliant devices are quarantined to a remediation VLAN instead of the corporate network.

How Intune NAC Integration Works

Intune exposes a NAC API endpoint that NAC solutions can query with a device ID. The query returns: device enrollment status, compliance state, and last compliance check time. The NAC solution (Cisco ISE, Forescout, Aruba ClearPass) receives this response and enforces network access policy based on the compliance state.

  1. 1
    Enable NAC in Intune: Intune → Tenant administration → Connectors → Network Access Control. Note: The NAC vendor registers an app in your Azure AD tenant which gets permission to query Intune's NAC API.
  2. 2
    Configure NAC vendor (e.g., Cisco ISE): Add Microsoft Intune as an MDM server in ISE. Configure the Azure AD application credentials for API queries.
  3. 3
    802.1X RADIUS policy on switches/APs: Configure your network infrastructure to require 802.1X authentication. The RADIUS server (ISE) handles authentication and queries Intune for compliance.
  4. 4
    Test enforcement: A non-compliant device connects to corporate Wi-Fi → RADIUS authenticates the device → queries Intune API → receives "Non-Compliant" → device placed in quarantine VLAN with internet-only access and remediation instructions.
Supported NAC Partners
  • Cisco Identity Services Engine (ISE): Most common in enterprise. Deep integration with Intune via RADIUS and API.
  • Forescout: Agentless network visibility combined with Intune compliance queries for full device posture.
  • Aruba ClearPass: Popular in education and healthcare. Intune compliance integration for 802.1X wired and wireless.
  • Portnox: Cloud-native NAC with native Intune integration — no on-premises RADIUS infrastructure required.
💡 NAC vs Conditional Access

Conditional Access blocks cloud app access based on compliance. NAC blocks network access based on compliance. They're complementary — CA protects cloud resources, NAC protects network resources. In a Zero Trust architecture, both are deployed: CA for cloud, NAC for network, creating layered enforcement that non-compliant devices cannot bypass through any path.

🎤 Interview Questions — Topic 122
How does Intune integrate with NAC solutions like Cisco ISE?
Intune exposes a NAC API that third-party NAC solutions can query with a device identifier to get the device's real-time compliance state. When a device connects to the network via 802.1X, the RADIUS server (ISE) authenticates the device certificate and then queries Intune's NAC API with the device ID to check compliance. If Intune returns "Non-Compliant", ISE places the device in a quarantine VLAN. The integration requires enabling NAC in Intune's connector settings, registering the NAC vendor's app in Azure AD, and configuring ISE with the API credentials.
What's the difference between Conditional Access and NAC for enforcement?
Conditional Access operates at the identity/application layer — it controls access to cloud applications and resources by evaluating compliance at the time of Azure AD authentication. NAC operates at the network layer — it controls whether a device can even connect to the network. They protect different layers: CA stops a non-compliant device from accessing SharePoint or Teams; NAC stops that same device from getting an IP address on the corporate VLAN. In a Zero Trust architecture, both are used together because a non-compliant device shouldn't reach the network or the cloud.
TOPIC 123
Intune and Third-Party Integrations
Jamf, ServiceNow, Lookout, and partner ecosystem integrations
Why Third-Party Integrations?

Intune doesn't operate in isolation. Enterprise environments include macOS fleets (where Jamf is often preferred), ITSM platforms for ticket automation, mobile threat defense solutions for advanced mobile security, and identity governance tools. Intune's open API and connector ecosystem lets it feed into and receive data from these platforms.

Jamf + Intune Integration (macOS)

Many organizations use Jamf Pro for macOS management (better native macOS support, richer Mac-specific features) while using Intune for Windows/mobile. The Jamf + Intune integration bridges the two: Jamf-managed Macs can participate in Azure AD Conditional Access using Intune's compliance framework.

How it works: Jamf registers managed Macs in Azure AD. Jamf sends device compliance data to Intune via the Jamf Cloud Connector. Intune uses this compliance signal in Conditional Access decisions — so a Mac managed by Jamf can be required to be "compliant" (per Jamf's definition) before accessing M365 resources.

🍎 Jamf Integration Requirements

Azure AD P1 license · Jamf Pro 10.36 or later · Jamf Cloud Connector configured in both Jamf and Intune (Intune → Tenant administration → Connectors → Partner compliance management → Jamf Pro) · macOS devices enrolled in Jamf must also be Azure AD registered

ServiceNow + Intune

ServiceNow integration enables: automatic ITSM ticket creation from Intune alerts, device inventory sync to ServiceNow CMDB, and Intune remediation actions triggered from ServiceNow workflows. Configuration is via the Microsoft Intune ServiceNow connector or through ServiceNow's Microsoft 365 spoke using Graph API calls.

  • Auto-ticket creation: When Intune detects a non-compliant device, ServiceNow auto-creates an incident assigned to the device owner's manager's queue.
  • CMDB sync: Intune device inventory exported to ServiceNow CMDB via Graph API, keeping asset records current automatically.
  • Remote wipe from ITSM: IT agents can trigger Intune device wipe/retire directly from the ServiceNow incident, without needing Intune portal access.

🎤 Interview Questions — Topic 123
An organization uses Jamf for Mac management. How can they include Macs in Conditional Access compliance requirements?
Through the Jamf + Intune integration via the Jamf Cloud Connector. Jamf registers managed Macs in Azure AD and sends compliance status to Intune. Intune then uses this as a compliance signal in Conditional Access — the CA policy sees the Mac as "Intune-managed" (via Jamf) and checks its compliance state. This way, Macs managed by Jamf can be required to be compliant before accessing M365 resources, even though Intune isn't managing them directly. It requires Azure AD P1, Jamf Pro 10.36+, and configuring the Jamf connector in both platforms.
What is Mobile Threat Defense and how does it work with Intune?
MTD solutions add threat detection capabilities for iOS and Android beyond what's built into Intune. They analyze the device for malicious apps, network-based attacks, jailbreak/root, and OS exploits, then send a risk score (Clear/Low/Medium/High) to Intune. Intune compliance policies can include an MTD condition — for example, "device threat level must be Low or Clear." If the MTD partner reports a High threat (e.g., device is connecting to a malicious Wi-Fi network), Intune marks the device non-compliant, and Conditional Access blocks corporate access automatically. Microsoft Defender for Endpoint on iOS/Android is the native Microsoft MTD option.
TOPIC 124
Intune Migration — From Other MDMs to Intune
Planning and executing a migration from MobileIron, VMware Workspace ONE, or Jamf to Intune
Why Migrate to Intune?

Organizations migrate to Intune from legacy MDMs for several reasons: Microsoft licensing consolidation (Intune included in M365 E3/E5 — no separate MDM cost), deep integration with Azure AD and Microsoft 365, unified endpoint management across Windows, iOS, Android, and macOS, and Copilot/AI features in the Microsoft security stack that don't extend to third-party MDMs.

Migration Phases
  1. 1
    Discovery and inventory: Export the full device inventory, policy list, app list, and certificate profiles from the existing MDM. Document: how many devices per platform, which apps are deployed, what compliance requirements exist, which certificates are in use, and what configurations are critical to operations.
  2. 2
    Intune environment setup: Configure the Intune tenant before migrating any devices — MDM enrollment scope, Apple MDM push cert, Android Enterprise binding, enrollment restrictions, naming conventions, and Azure AD groups.
  3. 3
    Policy translation: Recreate every policy from the old MDM in Intune. Don't assume 1:1 feature mapping exists — some old MDM policies may need to be achieved differently in Intune (e.g., custom OMA-URI instead of a GUI setting). Test every policy on pilot devices.
  4. 4
    App migration: Re-package and re-upload all Win32 apps. For managed iOS/Android apps, approve them in Managed Google Play / Apple VPP. Test all app deployments independently before migrating users.
  5. 5
    Pilot migration (5–10% of devices): Select a diverse pilot group (different platforms, departments, locations). Unenroll from old MDM → enroll in Intune → validate all policies and apps apply correctly → document issues and fix them before proceeding.
  6. 6
    Wave-based production migration: Migrate in waves by department or location. Maintain parallel operation during migration — old MDM manages unenrolled devices, Intune manages enrolled ones. Do not decommission old MDM until 100% migration is validated.
  7. 7
    Certificate migration: Most complex part. Certificates issued by the old MDM will expire — plan for SCEP/PKCS infrastructure in Intune before migrating devices that depend on certificates for Wi-Fi or VPN. Stagger migration so certificate-dependent devices migrate after the Intune PKI is validated.
The Double-Enrollment Problem

You cannot have two MDMs simultaneously managing a device. During migration, the device must be unenrolled from the old MDM before enrolling in Intune. For corporate-owned devices, this means a brief period of no MDM management. Plan the unenroll-to-enroll window carefully — preferably automate it so the gap is under 5 minutes.

🎤 Interview Questions — Topic 124
How would you plan a migration from VMware Workspace ONE to Intune for 5,000 devices?
I'd structure it in phases: first a full discovery — inventory all devices, policies, apps, and certificates from Workspace ONE. Then set up the Intune environment — configure enrollment, build all policies and apps in Intune, test on lab devices. Run a pilot with 200–300 diverse devices, validate everything, fix issues. Then wave-based migration by department — 500–1,000 devices per wave over 4–6 weeks. The trickiest part is certificates: any device using SCEP certs for Wi-Fi or VPN must have the Intune PKI validated and working before that device migrates, or they lose network access. I'd keep Workspace ONE running in parallel until the last device is confirmed enrolled in Intune and validated.
What's the double-enrollment problem in MDM migrations and how do you handle it?
A device can only be actively managed by one MDM at a time. During migration you must unenroll from the old MDM before enrolling in Intune — creating a brief unmanaged gap. For corporate devices, I minimize this gap by scripting or automating the unenroll and immediate re-enroll sequence, targeting sub-5-minute windows. For ADE iOS devices, I reassign them in Apple Business Manager to Intune and wipe them so Setup Assistant auto-enrolls them into Intune — that's the cleanest approach with no unmanaged period. The key is planning and communicating the timing so it happens during low-usage periods.
TOPIC 125
Real-World Enterprise Deployment — End to End
Bringing it all together: deploying Intune in a 3,000-device enterprise from zero to production

This final topic is a capstone — a complete walkthrough of deploying Intune in a real enterprise environment, integrating every block of this course into a coherent implementation. Follow this as a reference architecture.

The Scenario

Organization: 3,000-employee manufacturing company with offices in 5 countries. Currently using on-premises SCCM for Windows, no MDM for mobile, no cloud identity. Moving to Microsoft 365 E3. Goal: full Intune deployment in 6 months, Zero Trust aligned, with compliance reporting for cyber insurance.

Phase 1: Foundation (Weeks 1–4)
  • Azure AD setup: Configure Azure AD Connect to sync on-premises AD. Enable password hash sync. Configure MFA (Authenticator app) for all users.
  • Intune tenant setup: Set MDM authority, configure MDM user scope (All), set enrollment restrictions (block personal Windows devices, allow personal mobile), create naming conventions for groups and policies.
  • Azure AD groups: Create device groups by platform (All-Windows, All-iOS, All-Android), by ring (Pilot-Ring0, Standard-Ring2), by criticality (VIP-Devices). Dynamic group rules where possible.
  • Licensing: Assign Microsoft 365 E3 licenses (includes Intune P1) to all users. Assign Azure AD P2 to IT admins (for PIM).
  • PIM: Configure PIM for Intune Administrator, Global Administrator roles. All IT admins become eligible — no standing admin roles.
Phase 2: Windows Enrollment & Configuration (Weeks 5–10)
  • Hybrid AADJ: Configure Hybrid Azure AD Join via AAD Connect for existing domain-joined devices. Enable automatic MDM enrollment via Azure AD Mobility settings.
  • Autopilot: Register hardware hashes for all new devices from OEM. Build Autopilot profile (AADJ, user-driven, hide OOBE steps). Configure ESP with 60-minute timeout, required apps blocking.
  • Configuration profiles: Deploy security baseline (Microsoft Security Baseline for Windows 11). Configure BitLocker silent encryption with Azure AD key escrow. Deploy Wi-Fi and VPN profiles. Restrict removable media (USB blocking). Configure LAPS for local admin passwords.
  • Update rings: Create 4-ring strategy. Ring 0 (IT) — 0-day deferral. Ring 1 (Early) — 7 days. Ring 2 (Standard) — 14 days. Ring 3 (VIP) — 21 days. Feature updates pinned to current production Windows build.
  • Apps: Package and deploy: Microsoft 365 Apps (via built-in Intune installer), Chrome Enterprise (Win32), VPN client (Win32), LOB ERP application (Win32). Configure supersedence chains for legacy app versions.
Phase 3: Mobile Enrollment (Weeks 8–12)
  • iOS/iPadOS: Set up Apple Business Manager, connect to Intune, obtain Volume Purchase Program token. Configure ADE enrollment profile for corporate iPhones. Deploy: Outlook, Teams, Company Portal, VPN client via VPP. App protection policies for corporate email/data on BYOD.
  • Android: Configure Android Enterprise with Managed Google Play. AE Fully Managed for corporate Androids. AE Work Profile for BYOD. Deploy core apps from Managed Google Play. App protection policies for MAM.
Phase 4: Security & Compliance (Weeks 10–16)
  • Compliance policies: Windows: BitLocker enabled, Defender AV active, OS within 30 days of latest patch, minimum OS version, complex password. iOS: passcode required, not jailbroken. Android: not rooted, Play Protect enabled.
  • MDE integration: Connect Defender for Endpoint to Intune. Compliance policy includes MDE risk requirement (Low or Clear). Deploy MDE onboarding profile to all Windows devices.
  • Conditional Access: Require compliant device for all M365 apps. Require MFA for all cloud apps. Block legacy authentication. Named Locations configured for offices. Risk-based CA using Azure AD Identity Protection.
  • SCEP certificates: Deploy NDES infrastructure. Configure Intune Certificate Connector. Deploy Trusted Root → SCEP certificate profiles for Wi-Fi authentication. Test 802.1X connectivity on pilot devices before broad rollout.
  • Windows Hello for Business: Enable WHFB policy for all AADJ and Hybrid AADJ devices. Configure PIN complexity. Enable PIN reset service. Target all users to move to passwordless over 60 days.
Phase 5: Monitoring, Automation & Reporting (Weeks 14–20)
  • Proactive Remediations: Auto-fix stale DNS cache, repair Windows Update issues, detect and re-enable Defender Real-Time Protection, alert on low disk space before it becomes critical.
  • Endpoint Analytics: Monitor startup performance. Create baseline. Identify the 200 slowest-booting devices for hardware refresh prioritization.
  • Log Analytics: Stream Intune audit logs and device compliance logs to Azure Monitor Log Analytics. Build KQL queries for non-compliance trends, enrollment failures, app install failure rates.
  • Power BI dashboard: Connect to Intune Data Warehouse. Build CISO dashboard: compliance rate (target: 95%), BitLocker coverage, MDE onboarding rate, OS currency, enrollment growth. Auto-refresh daily. Share with CISO and Head of IT.
  • Graph API automation: PowerShell script run weekly via Azure Automation: export non-compliant device list, stale devices (no check-in 14+ days), devices approaching OS end-of-support. Email report to IT team leads.
Phase 6: Validation and Handover (Weeks 20–24)
  • Compliance audit: Run full compliance report across all 3,000 devices. Target ≥95% compliant before sign-off. Document remaining non-compliant devices with remediation plans and owners.
  • Documentation: Architecture diagram, runbook for common operations (enrollment, offboarding, app packaging, certificate renewal), escalation procedures, and incident response plan for Intune-related outages.
  • Handover training: Train L1 helpdesk on Troubleshooting blade usage. Train L2 on log analysis and policy troubleshooting. Train IT management on Power BI dashboard interpretation.
  • Co-management decision: Review remaining SCCM workloads. Migrate: Compliance policies, Device configuration, Endpoint Protection, Windows Update to Intune. Keep SCCM for: legacy application deployment (if needed), OSD for specific scenarios.

Key Lessons from Real Enterprise Deployments

Certificate complexity is always underestimated. Budget 2x the time you think certificate infrastructure will take. NDES, SCEP, and 802.1X issues are the most common cause of deployment delays.

Communication is half the job. Users who understand WHY Intune is being deployed (security, enabling remote work, BYOD access) comply far better than users who receive a mandated IT change with no explanation. Send clear communication: what Intune can and cannot see on their device.

Never decommission the old MDM until 100% migration is validated. Parallel operation feels wasteful but saves you when a wave migration fails and devices need to re-enroll in the old system while you fix Intune issues.

PIM from day one. Standing Intune admin accounts are the biggest security risk in your deployment. Configure PIM before the first device enrolls — not as an afterthought 6 months later.

🎤 Interview Questions — Topic 125
You're hired to deploy Intune for a 3,000-device organization. Where do you start?
I start with discovery — understand the current state: what devices exist, what platforms, what the security requirements are, what licensing is in place, and what the timeline is. Then I set up the foundation before touching any devices: Azure AD groups, MDM scope, enrollment restrictions, naming conventions, and PIM for admin roles. I build a pilot group (50–100 devices covering all platforms and use cases) and validate every policy, app, and compliance requirement end-to-end before any production migration. Only after the pilot validates cleanly do I begin wave-based production deployment — by department or location, with communication and rollback plans at each wave.
What's the most challenging part of an enterprise Intune deployment in your experience?
Certificate infrastructure — specifically getting SCEP via NDES working reliably for Wi-Fi and VPN authentication. It involves the most components (CA, NDES, connector, firewall rules, certificate templates, Intune profiles) and the most potential failure points. The second challenge is hybrid environment complexity — GPO conflicts, Hybrid AADJ line-of-sight requirements, and co-management workload decisions require deep knowledge of both SCCM and Intune. The organizational challenge is user communication — Intune deployments that fail technically can usually be fixed, but deployments that fail because users resist enrollment or compliance are much harder to recover.
How would you measure success of an Intune deployment?
I'd define success metrics upfront: enrollment rate (target: 100% of corporate devices enrolled within timeline), compliance rate (target: ≥95% compliant), app deployment success rate (target: ≥98% of required apps installed successfully), MDE onboarding rate (target: 100% of Windows devices), BitLocker encryption coverage (target: 100% of laptops), and WHFB adoption rate (target: 80% within 3 months of deployment). I'd track these in a Power BI dashboard refreshed daily and report weekly to IT leadership during the deployment. Post-deployment, I'd measure helpdesk ticket volume for Intune-related issues as a proxy for operational stability.
