TOPIC 76
Intune RBAC — Built-in Roles Explained
The right access for the right people — understanding every built-in role
What Is RBAC in Intune?

Role-Based Access Control (RBAC) in Intune determines what actions a signed-in admin can perform and which objects they can see in the Intune portal. Without RBAC, every admin would need to be a Global Admin — a serious security risk. RBAC lets you give people exactly the permissions they need and nothing more — the Principle of Least Privilege.

Intune RBAC works independently from Azure AD roles. A user needs both: an Azure AD role (like Intune Administrator) to access Intune at all, and an Intune RBAC role assignment to control what they can do once inside. Exception: Global Admin and Intune Administrator (Azure AD roles) bypass all Intune RBAC restrictions and have full access.

Built-in Intune Roles — Complete Reference
Role NameWhat They Can DoWhat They Cannot DoWho Gets This
Intune Role Administrator Create, modify, delete, and assign Intune roles. Manage scope tags. View all Intune configuration. Cannot manage devices, apps, or policies directly. Senior IT admin responsible for role governance. Usually just 1–2 people.
Help Desk Operator View device/user details. Perform device remote actions: wipe, retire, sync, remote lock, reset passcode, restart. View app install status. View compliance status. Cannot create or modify policies, compliance rules, apps, or enrollment configurations. Cannot view BitLocker keys (without additional permission). L1/L2 helpdesk staff who handle day-to-day user support tickets.
Application Manager Create, modify, assign, and delete apps. Manage app configuration policies. Manage app protection policies (MAM). View device and user information (read-only). Cannot manage device compliance, configuration profiles, enrollment settings, or endpoint security policies. App deployment team — manages the app catalog without access to device policy management.
Policy and Profile Manager Create, modify, assign, and delete configuration profiles, compliance policies, terms of use. Run scripts. View device and user information. Cannot manage apps, endpoint security (Defender, BitLocker, ASR), or enrollment configurations. Cannot perform device remote actions. Configuration management team — handles policy deployment but not apps or security hardening.
Endpoint Security Manager Manage all Endpoint Security policies (AV, BitLocker, Firewall, ASR, EDR, Account Protection). Manage Security Baselines. View Secure Score. Manage Conditional Access in Intune context. Cannot manage general configuration profiles, apps, or enrollment settings. Security team / SOC analyst who focuses on hardening without touching general device management.
Read Only Operator View everything in Intune — all devices, apps, policies, reports, compliance status. Cannot modify anything. Cannot create, modify, or delete any Intune object. Cannot perform remote actions on devices. Auditors, managers, NOC staff, compliance officers who need visibility without change capability.
School Administrator Manage groups, apps, and devices within Intune for Education context. Cannot manage enterprise Intune features outside the education scope. Education-specific — IT staff in schools using Intune for Education.
RBAC Permission Categories

Each Intune role is built from a combination of permissions across these categories:

HIGH Device Actions
Remote wipe, factory reset, retire, remote lock, reset passcode, BitLocker key rotation, collect diagnostics. High impact — can destroy data or lock users out.
HIGH Policy Management
Create/edit/delete configuration profiles, compliance policies, endpoint security policies, scripts. Changes affect thousands of devices.
MED App Management
Deploy and remove apps, manage app protection policies, app configuration. Can install or remove software across the fleet.
MED Enrollment
Configure enrollment methods, enrollment restrictions, Autopilot profiles, CNAME settings. Controls how new devices join Intune.
LOW Read Permissions
View devices, users, policies, reports, app status, compliance. No change capability. Safe to grant broadly for monitoring and troubleshooting.
LOW Remote Sync
Trigger device sync to refresh policies. Low risk — just accelerates the normal check-in cycle. Good to grant helpdesk for troubleshooting.
How RBAC Assignment Works
  1. 1
    A Role defines what actions are allowed (the permission set).
  2. 2
    A Role Assignment links that role to specific members (who has the role) and specific scope groups (which devices/users they can manage).
  3. 3
    The Members are Azure AD groups containing the admin users (e.g., "HelpDesk Staff" group).
  4. 4
    The Scope (Groups) are Azure AD groups of devices or users that this role assignment can manage (e.g., "India Devices" group).
  5. 5
    Scope Tags further restrict visibility — the admin only sees Intune objects tagged with their scope tag (covered in Topic 78).
💡 Role Assignment vs Scope Tags — Two Layers

Role Assignment scope groups control which devices and users the admin can take action on (e.g., can only wipe India devices).
Scope Tags control which Intune objects (policies, apps, profiles) the admin can see and manage.
Both work together — an admin scoped to India can only: (a) act on India devices, AND (b) see India-tagged policies. Full isolation requires configuring both correctly.

🎤 Interview Questions — Topic 76
What is the difference between the Help Desk Operator and Policy and Profile Manager roles in Intune?
Help Desk Operator can perform device remote actions (wipe, sync, remote lock, restart) and view device/user information, but cannot create or modify policies. Policy and Profile Manager can create and manage configuration profiles, compliance policies, and scripts, but cannot perform device remote actions or manage apps. Helpdesk handles day-to-day user support; Policy Manager handles configuration management.
Does a user with the Intune Role Administrator role in Intune bypass all RBAC restrictions?
No. The Intune Role Administrator is a built-in Intune RBAC role that only allows managing roles and scope tags — it doesn't bypass other restrictions. What bypasses Intune RBAC entirely is having the Global Administrator or Intune Administrator Azure AD role — these Azure AD roles have full Intune access regardless of Intune RBAC assignments. The Intune Role Administrator role is specifically for governing who can assign roles within Intune.
An auditor needs to review all Intune configurations but must not be able to change anything. Which role do you assign?
Read Only Operator. This built-in role provides full read access to all Intune objects — devices, apps, policies, compliance reports, enrollment configurations — with absolutely no ability to create, modify, delete, or perform remote actions. It's the ideal role for auditors, compliance officers, and managers who need visibility without change capability.
TOPIC 77
Custom Roles — Creating & Assigning
When built-in roles don't fit — building purpose-built permission sets
When Built-in Roles Aren't Enough

Built-in roles cover common scenarios well, but real enterprise IT teams often have specialized needs that don't map cleanly to built-in roles. Custom roles let you combine exactly the permissions you need — no more, no less — for specific teams or workflows.

Real Scenarios Where Custom Roles Are Needed
🚑
Healthcare IT Helpdesk
Needs: view device details, remote lock lost tablets, collect diagnostics, view compliance. Does NOT need: wipe (HIPAA concern — accidental wipe of clinical device is catastrophic), create policies, manage apps. Built-in Help Desk Operator has wipe permission — too dangerous. Custom role removes wipe, retains everything else.
📊
Security Compliance Reporter
Needs: read all compliance reports, read endpoint security policy status, view BitLocker recovery keys for audit. Does NOT need: create/modify anything. Read Only Operator doesn't include BitLocker key viewing. Custom role: Read Only + BitLocker key read permission.
🔄
Autopilot Deployment Team
Needs: manage Autopilot device registrations, create enrollment profiles, manage device categories. Does NOT need: access to compliance policies, endpoint security, apps. A narrow custom role focused only on enrollment/provisioning tasks.
📱
Mobile App Deployer
Needs: deploy iOS and Android apps via Managed Google Play and VPP. View mobile device status. Does NOT need: Windows management, policy creation, compliance configuration. A mobile-only App Manager custom role.
Creating a Custom Role — Step by Step
  1. 1
    Intune → Tenant Administration → Roles → All roles → Create
  2. 2
    Enter a Name (descriptive: "Healthcare Helpdesk - No Wipe") and Description (explain what this role is for and why — helps future admins understand the intent).
  3. 3
    Under Permissions, browse the permission categories:
    • Check each permission category (e.g., "Managed Devices")
    • For each category, enable specific permissions: Read, Create, Update, Delete, or action-specific (Wipe, Retire, Sync, etc.)
    • Grant only what's needed — review each permission description carefully
  4. 4
    Click Next → Scope Tags tab: assign scope tags to restrict the role to specific administrative units (covered in Topic 78).
  5. 5
    Review and Create the role.
  6. 6
    Assign the role: Find the new role in All Roles → Assignments → Add assignment → Specify members (Azure AD group of admin users) and scope groups (Azure AD group of devices/users they'll manage).

Duplicating a Built-in Role

The fastest way to create a custom role is to duplicate a built-in role and then modify it. This way you start with a tested permission set and only adjust what needs changing.

  1. 1
    Intune → Tenant Administration → Roles → Find the built-in role closest to what you need (e.g., Help Desk Operator)
  2. 2
    Click on the role → Duplicate
  3. 3
    Rename the duplicate to your custom role name. Edit permissions to add or remove what's needed.
  4. 4
    Save → Assign as normal.
💡
Document Your Custom Roles: Create a role design document that lists every custom role, its purpose, which built-in role it was based on, what permissions were added/removed, and which groups it's assigned to. Without this documentation, the next admin to look at your tenant won't understand why the roles exist. Include a review date — custom roles should be audited annually to verify they still reflect the principle of least privilege.
🎤 Interview Questions — Topic 77
When would you create a custom Intune role instead of using a built-in role?
When no built-in role precisely matches the permission needs of a team. Common scenarios: a helpdesk that needs device remote actions but not wipe (remove wipe from Help Desk Operator), a security auditor who needs Read Only access plus BitLocker key viewing, an Autopilot team that only needs enrollment management, or any team whose responsibilities span multiple built-in roles in a way that combining them would grant too many permissions. Custom roles implement the principle of least privilege more precisely.
Walk me through creating a custom role for a helpdesk team that can remote lock and sync devices but not wipe them.
Duplicate the Help Desk Operator role. Rename it "Helpdesk - No Wipe." In the Permissions tab, expand "Managed Devices" permissions. Remove the Wipe and Retire checkboxes while keeping Read, Remote Lock, Sync, and Restart. Save the role. Create a role assignment: assign the role to the "Helpdesk Staff" Azure AD group as members, and specify the "All Devices" (or relevant regional device group) as the scope group. Helpdesk can now remote lock and sync but cannot wipe or retire any device.
TOPIC 78
Scope Tags — What They Are & How to Use Them
Isolating Intune objects by region, department, or business unit
The Problem Scope Tags Solve

Imagine a global company with Intune managing devices in India, USA, and UK — all in one tenant. The India IT team should only see and manage India devices and India-specific policies. The US team should only see US objects. Without Scope Tags, every admin with Intune access sees and can accidentally modify ALL devices and ALL policies — regardless of geography or team. One fat-finger from the India admin can wipe a device in New York.

Scope Tags are labels you attach to Intune objects (devices, policies, apps, enrollment profiles) and to role assignments. When an admin is scoped to a specific tag, they only see and manage objects carrying that tag. Objects without their tag are invisible to them.

How Scope Tags Work — The Mechanics
  1. 1
    Create Scope Tags: Intune → Tenant Administration → Roles → Scope tags → Create → Name: "India", "USA", "UK", "Finance", "Retail"
  2. 2
    Tag Intune objects: Each policy, app, configuration profile, enrollment profile, and device can be assigned one or more scope tags. You tag objects when creating or editing them — there's a "Scope tags" step in the creation wizard.
  3. 3
    Assign scope tags to role assignments: When assigning a role to a team, specify which scope tags that team can see. The India Helpdesk role assignment → select scope tag "India".
  4. 4
    Result: When the India Helpdesk admin logs into Intune, they only see devices, policies, and apps tagged "India". All USA and UK objects are completely invisible to them.
Scope Tags — Regional Isolation Example
🇮🇳 India Scope Tag
Tagged objects: India devices, India Wi-Fi profiles, India VPN config, India helpdesk enrollment profile, India-specific apps.

India IT team members: Can only see and manage these objects.
🇺🇸 USA Scope Tag
Tagged objects: USA devices, USA compliance policies, USA firewall rules, USA endpoint security configs.

USA IT team members: Can only see and manage these objects.
🇬🇧 UK Scope Tag
Tagged objects: UK devices, UK GDPR-specific policies, UK app deployments.

UK IT team members: Can only see and manage these objects.
🌐 Default (No Tag / All)
Global admins (Intune Administrator Azure AD role) see ALL objects regardless of scope tags. Regional admins cannot see untagged objects unless explicitly assigned.
Scope Tags vs Role Assignment Scope Groups
MechanismControlsWhat Admin Sees/Does
Role Assignment → Scope GroupsWhich USERS and DEVICES the admin can take action onThe India admin can only wipe/sync/view devices that are members of the "India Devices" Azure AD group
Scope TagsWhich INTUNE OBJECTS (policies, apps, profiles) the admin can see and manageThe India admin can only see policies, apps, and profiles that carry the "India" scope tag
Both TogetherComplete isolation of both device management AND policy managementTrue administrative isolation — India admin manages India devices with India policies only
Default Scope Tag

Every Intune tenant has a Default scope tag. Objects created without explicitly assigning a scope tag automatically get the Default tag. Admins with access to the Default scope tag can see all untagged objects. Global Admins and Intune Administrators always see everything — scope tags don't restrict Azure AD admin roles.

⚠️ Common Scope Tag Mistake

Creating scope tags but forgetting to tag existing policies and devices. If you set up scope tags for a new regional admin and forget to tag the objects they need to manage, they'll see nothing in Intune and file a support ticket saying "I can't see any devices." Whenever you create a new scope tag for a new region/team, go through existing Intune objects and tag the relevant ones. Also tag new devices at enrollment using enrollment profiles or device categories.

🌏
Real-World Multi-Region Scope Tag Setup

Company: IT outsourcing firm managing Intune for 3 clients — Client A (retail), Client B (finance), Client C (healthcare) — all in one Intune tenant (Multi-Tenant Management scenario using Lighthouse, or a shared tenant approach).
Setup:
Create scope tags: "ClientA-Retail", "ClientB-Finance", "ClientC-Health"
Tag all Client A devices, policies, apps → "ClientA-Retail"
Create role assignments for each client's dedicated IT admin group → scoped to their respective tag
Result: Client A's IT admin sees only Client A devices and policies. Client B's admin sees only Client B. Zero cross-client visibility. A Client A admin cannot accidentally modify a compliance policy affecting Client B's financial devices. Regulatory segregation maintained within a single Intune tenant.

🎤 Interview Questions — Topic 78
What are Scope Tags in Intune and what problem do they solve?
Scope Tags are labels assigned to Intune objects (devices, policies, apps) and role assignments to create administrative isolation within a single tenant. Without them, every Intune admin sees and can manage all objects across all regions and departments. With Scope Tags, an admin scoped to "India" only sees India-tagged devices and policies — USA and UK objects are completely invisible. This prevents accidental cross-region changes and supports multi-team, multi-region, or multi-client deployments in a single tenant.
What is the difference between Scope Tags and Role Assignment Scope Groups?
Scope Groups in a role assignment control which users and devices the admin can take action on — they can only wipe, sync, or view devices that are in the specified Azure AD group. Scope Tags control which Intune objects (policies, apps, profiles) the admin can see and edit in the portal. Both are needed for complete isolation: scope groups restrict device actions, scope tags restrict object visibility. Using only scope groups means an admin can still see and modify all policies even if they can't apply them to out-of-scope devices.
Can Scope Tags restrict what a Global Administrator can see in Intune?
No. Scope Tags do not restrict Global Administrators or users with the Intune Administrator Azure AD role. These Azure AD admin roles bypass all Intune RBAC controls including Scope Tags — they always see everything in the tenant. Scope Tags only restrict custom and built-in Intune RBAC roles. This is why Global Admin accounts should be used sparingly and only for break-glass scenarios — they bypass all administrative boundaries.

Delegated Administration Scenarios
Real-world RBAC design patterns — putting it all together
Why Delegation Design Matters

RBAC theory is simple. Real enterprise delegation is complex. IT teams have sub-teams, third-party partners, contractors, outsourced helpdesks, and regional offices — all needing different levels of access. This topic covers the most common enterprise delegation patterns and how to implement them correctly using roles, scope groups, and scope tags together.

Delegation Pattern 1 — Tiered IT Support (L1/L2/L3)
TierRoleScope GroupsScope TagsWhat They Can Do
L1 HelpdeskCustom: "L1 Support" (Read + Sync + Remote Lock only)All DevicesDefaultView device info, trigger sync, remote lock lost devices. Cannot wipe, retire, or modify policies.
L2 HelpdeskHelp Desk Operator (built-in)All DevicesDefaultAll L1 actions + wipe, retire, reset passcode, restart, collect diagnostics. Cannot create policies.
L2 ConfigPolicy and Profile Manager (built-in)All DevicesDefaultCreate/modify configuration profiles, compliance policies, scripts. No device actions, no app management.
L2 AppApplication Manager (built-in)All DevicesDefaultDeploy and manage apps, app protection policies. No config profiles, no compliance policies.
L3 Senior / Endpoint AdminIntune Administrator (Azure AD role)All (unrestricted)All (unrestricted)Full Intune access. Used for complex changes and escalations.
Delegation Pattern 2 — Geographic Isolation

For multi-region organizations where each region has its own IT team:

  1. 1
    Create Dynamic Device Groups per region: "India_Devices" → device.country -eq "IN" or based on device name prefix, enrollment profile, or device category.
  2. 2
    Create Scope Tags per region: "India", "USA", "UK"
  3. 3
    Tag all India objects: As India IT creates policies and apps, they tag them "India". Global admin ensures enrollment profiles for India devices automatically apply the "India" scope tag.
  4. 4
    Create Role Assignments: "India Helpdesk" role → Members: India IT group → Scope Groups: India_Devices → Scope Tags: India
  5. 5
    Test: Have an India IT admin log in → Verify they see only India devices in All Devices → Verify they can only see India-tagged policies → Verify they cannot see or act on US/UK devices.
Delegation Pattern 3 — Third-Party Managed Service Provider (MSP)

Many organizations outsource Intune management to an MSP. The MSP needs enough access to manage devices but cannot see sensitive data or other clients' configurations:

  • Create an MSP scope tag: "MSP-Managed". Tag all devices and policies the MSP is responsible for.
  • Create a custom "MSP Operations" role: Based on Help Desk Operator + Application Manager + Policy and Profile Manager, scoped to the MSP tag only.
  • The MSP never gets Global Admin or Intune Administrator — they can't see other clients, billing information, or sensitive tenant settings.
  • Audit log filtering: Set up audit log alerts for any changes made by the MSP accounts — you can verify their work without giving them self-audit capabilities.
Delegation Pattern 4 — Department-Specific Admin

Finance department has its own GDPR-sensitive devices with strict compliance requirements. Finance IT manages their own devices within the corporate tenant:

  • Finance Scope Tag: Applied to all Finance devices during enrollment (via enrollment profile → scope tag). Also applied to Finance-specific compliance policies.
  • Finance IT Admin role: Policy and Profile Manager + Help Desk Operator scoped to Finance Scope Tag and Finance_Devices group.
  • Finance IT can: Create Finance-specific compliance policies with stricter requirements, manage Finance devices, deploy Finance-approved apps.
  • Finance IT cannot: See or manage HR, Engineering, or Sales devices. Cannot see global policies unless tagged Finance.
  • Central IT can: See and manage everything (Global Admin). Apply a global baseline to Finance devices if needed (tagged both Default and Finance).
🏦
Enterprise RBAC Design — Banking Scenario

Bank with 5,000 devices, 3 regions (Mumbai, Delhi, Hyderabad), outsourced L1 helpdesk, in-house L2/L3 team, security compliance team, and external auditors.

Roles designed:
• "L1-Outsourced" (custom): Read + Sync only. No wipe. No policy access. Scoped to "All Devices" with Default tag. Members: Outsourced helpdesk team.
• "Mumbai-L2" (Help Desk Operator duplicate): Full device actions. Scoped to Mumbai_Devices group + Mumbai scope tag. Members: Mumbai IT team.
• "Delhi-L2" / "Hyderabad-L2": Same pattern for other regions.
• "App-Deployer" (Application Manager built-in): Scoped to All Devices + Default tag. Members: App team.
• "Security-Auditor" (custom: Read Only + BitLocker key view): All scope. Members: Internal audit team.
• "External-Auditor" (Read Only): Default scope, 30-day time-limited assignment via Azure AD PIM. Members: External audit firm account.

Result: Mumbai helpdesk can only see Mumbai devices. Outsourced L1 cannot wipe anything. External auditors get temporary access for their quarterly audit. Zero privilege creep. Every action is logged in audit logs.

🎤 Interview Questions — Topic 79
How would you set up Intune for a company with IT teams in India and the UK who should not be able to see each other's devices or policies?
Create two Scope Tags: "India" and "UK". Create dynamic Azure AD device groups based on region (device naming convention, enrollment profile, or country attribute). Tag all India-specific devices, policies, and apps with the India scope tag; do the same for UK. Create role assignments for each regional team — Help Desk Operator or custom role, with Members set to the regional IT group, Scope Groups set to the regional device group, and Scope Tags set to the regional tag. India IT admins now only see India-tagged objects; UK admins only see UK-tagged objects. Central Global Admins retain full visibility.
How would you give an external auditing firm temporary read-only access to Intune for a compliance review?
Create guest accounts in Azure AD for the auditors (or use B2B collaboration). Assign the Read Only Operator role in Intune via a time-limited role assignment. For the most controlled approach, use Azure AD Privileged Identity Management (PIM) to create a time-limited eligible assignment — the auditors activate their access for specific days and it automatically expires after the review period. Scope the assignment appropriately — if they should only see certain regions, add scope tags. Monitor their activity via Intune Audit Logs during and after the review.

Audit Logs — Who Did What & When
The complete change history of your Intune tenant — compliance, forensics, and accountability
What Are Intune Audit Logs?

Every action taken in the Intune portal — creating a policy, deleting an app, wiping a device, changing a role assignment — generates an audit log entry. Audit logs are immutable, tamper-proof records of every change and action in your tenant. They answer the critical compliance questions: Who changed this policy? When was this device wiped? Who viewed this BitLocker key? Did anyone modify enrollment settings last Tuesday?

Accessing Audit Logs
  • Primary location: Intune → Tenant Administration → Audit logs
  • Per-resource logs: Many Intune sections have their own audit log view. E.g., Devices → Monitor → Audit logs shows device-specific actions only.
  • Filtering: Filter by date range, activity type (Create/Delete/Action/Patch), category (Device/App/Policy), and specific user.
  • Retention: Intune audit logs are retained for 1 year by default in the portal.
  • Long-term retention: Export to Azure Monitor Log Analytics or storage account for longer-term retention (required for many compliance frameworks like ISO 27001, SOC2 — which often require 3+ year retention).

Using Audit Logs for Security Investigations

Audit logs are invaluable during security incidents. Common investigation scenarios:

🔍
Unauthorized Policy Change
A compliance policy was modified without a change ticket. Filter audit logs by the policy name and date range. See exactly who changed it, what was changed (old vs new value), their IP address, and the exact timestamp. Correlate with HR records if the account belongs to a former employee.
📱
Unexplained Device Wipe
An employee reports their device was wiped unexpectedly. Filter audit logs by device name and "Wipe" activity. Identify which admin account triggered the wipe, from what IP, at what time. If it wasn't an authorized action, investigate the admin account for compromise.
🔑
BitLocker Key Access Audit
For GDPR or internal audit requirements, you need to demonstrate who accessed BitLocker recovery keys and when. Filter by "Get BitLocker Key" activity. Every key retrieval is logged with the admin's identity, the device, and timestamp — providing a full chain of custody for the encryption key.
👤
Insider Threat Detection
A disgruntled employee with helpdesk access is suspected of unauthorized device actions. Pull all audit log entries for that specific user account across a date range. Review every action they took — what devices they accessed, what they changed, what remote actions they performed.
Exporting and Long-Term Audit Log Storage

The Intune portal retains audit logs for 1 year. Many compliance frameworks require longer retention. Options:

  • CSV Export: Filter the logs as needed → Export to CSV. Manual, point-in-time. Not suitable for ongoing retention.
  • Azure Monitor / Log Analytics: Configure Intune Diagnostic Settings to stream audit logs continuously to an Azure Log Analytics Workspace. Query with KQL. Retain for 90 days to 2 years in Log Analytics.
  • Azure Storage Account: Archive logs to blob storage for long-term, low-cost retention (years). Not directly queryable but meets retention requirements.
  • Azure Event Hub → SIEM: Stream Intune logs to a SIEM (Sentinel, Splunk, QRadar) for real-time alerting and correlation with other security events. The enterprise standard for security operations.
Setting Up Azure Monitor Integration
  1. 1
    Go to Intune → Tenant Administration → Diagnostic settings → Add diagnostic setting
  2. 2
    Select log categories to stream: AuditLogs, OperationalLogs, DeviceComplianceOrg, Intune events, etc.
  3. 3
    Select destination: Log Analytics Workspace (for querying), Storage Account (for archiving), Event Hub (for SIEM integration), or all three.
  4. 4
    Save. Logs begin streaming within minutes.
  5. 5
    In Log Analytics: query with KQL. Example: IntuneAuditLogs | where ActivityResultStatus == "Success" | where OperationName contains "Wipe" | project TimeGenerated, Identity, ResultDescription
Creating Audit Alerts

Don't just store audit logs — alert on critical actions in real-time. Via Azure Monitor Alerts or Microsoft Sentinel, create alert rules for:

  • Device wipe triggered outside business hours → immediate alert to security team
  • Compliance policy modified → alert to change management team (verify against change ticket)
  • New admin role assigned → alert to CISO (privilege escalation detection)
  • Enrollment restriction changed → alert to IT manager (could open door to unauthorized device enrollment)
  • ASR rules disabled or weakened → security team alert (potential attack preparation)

Real Audit Log Investigation

Incident: Monday morning, 3 users report their company iPhones were wiped over the weekend. No wipe was scheduled. IT management is alarmed.
Investigation using Audit Logs:
Step 1: Intune → Tenant Administration → Audit logs → Filter: Activity type = "Wipe", Date = Saturday 00:00 – Sunday 23:59.
Step 2: Three wipe events found at 02:15, 02:17, 02:19 on Saturday night.
Step 3: Actor: svc-intune-automation@contoso.com — a service account.
Step 4: Actor IP: 185.234.219.x — an IP outside the company's known ranges.
Step 5: Check Azure AD Sign-in logs for the service account → this account signed in from that foreign IP using a password spray attack at 02:10 — 5 minutes before the wipes.
Root Cause: Service account was compromised via password spray. The attacker used the service account's Intune permissions to wipe 3 iPhones as a destructive act.
Remediation: Reset service account credentials. Add MFA. Restrict service account to specific IP ranges. Audit all service account permissions. Implement alert rule for wipes outside business hours.

🎤 Interview Questions — Topic 80
What information is captured in an Intune audit log entry?
Each audit log entry captures: the timestamp (UTC), the actor (user or service account that performed the action and their Object ID), the actor's IP address and application used, the activity type (Create, Update, Delete, Action), the target resource name and ID, the result (Success/Failure), and the modified properties (what changed, showing old and new values). This provides complete accountability for every action in the Intune tenant.
How long does Intune retain audit logs and what do you do for longer retention requirements?
Intune retains audit logs for 1 year in the portal. For longer retention (required by many compliance frameworks like ISO 27001, SOC2, HIPAA): configure Diagnostic Settings in Intune to stream logs to Azure Log Analytics (up to 2 years, queryable), Azure Storage Account (years, low-cost archival), or Azure Event Hub (for SIEM integration like Microsoft Sentinel or Splunk). For real-time security alerting, streaming to a SIEM is the enterprise standard, allowing correlation of Intune audit events with other security signals.
You suspect an admin account was used to make unauthorized changes to Intune policies. What's your investigation process using audit logs?
Go to Intune → Tenant Administration → Audit logs. Filter by the suspected admin account's UPN as the actor. Set the date range covering the suspected period. Review all activities that account performed — note any Creates, Updates, or Deletes that were not expected. For each suspicious entry, check the modified properties field to see what changed (old vs new values), the timestamp, and the IP address. Cross-reference the IP with Azure AD Sign-in logs to verify if there were any suspicious sign-in patterns (unusual location, risky sign-in). If the account was compromised, escalate to the security incident response process.
What critical actions would you set up alerts for in Intune audit logs?
High-priority alerts: device wipe triggered outside business hours (potential insider threat or compromised account), new admin role assigned to any user (privilege escalation), compliance or ASR policies modified (security weakening), enrollment restrictions changed (opens enrollment to unauthorized devices), and any action performed by service accounts from unexpected IPs. These alerts are configured via Azure Monitor alert rules on the Log Analytics workspace receiving the streamed audit logs, or via Microsoft Sentinel analytics rules for SIEM integration.
