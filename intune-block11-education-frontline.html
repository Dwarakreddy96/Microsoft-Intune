Intune for Education — Differences & Use Cases
How Intune works in schools — student devices, shared iPads, and education-specific features
What Is Intune for Education?

Intune for Education is a simplified, education-optimized version of Microsoft Intune accessible at intuneeducation.microsoft.com. It's not a separate product — it's the same Intune backend with a streamlined interface designed for school IT admins who may not be enterprise endpoint management specialists. The full Intune portal is also available for advanced configurations.

It's designed to handle the unique challenges schools face: shared devices used by multiple students, bulk provisioning of hundreds of iPads or Chromebook-like Windows devices, simplified app deployment for educational software, and age-appropriate content restrictions.

School Data Sync (SDS) — Auto-Importing School Rosters

School Data Sync is a free Microsoft service that automatically imports school roster data (students, teachers, classes, groups) from your Student Information System (SIS) — like PowerSchool, Infinite Campus, Skyward — into Azure AD. Instead of manually creating 1,000 student accounts, SDS does it automatically at the start of each school year.

  • Automatic group creation: SDS creates Azure AD groups for each class — "Year 10 Science Period 3" — which Intune uses for targeted app and policy deployment.
  • Annual refresh: At the start of each school year, SDS updates groups — new students added, graduated students removed, class rosters updated.
  • Teacher/Student distinction: SDS marks accounts with education roles. Teacher devices and student devices get different Intune policies automatically.
  • Integration: Set up at sds.microsoft.com. Connects to your SIS via CSV file export or direct API integration depending on the SIS vendor.

Real Education Deployment Scenario

School: 2,000-student secondary school deploying 600 shared iPads and 200 Windows laptops for teachers.
Setup:
1. Apple School Manager set up — 600 iPads pre-registered via ASM at purchase
2. School Data Sync imports 2,000 students and 150 teachers from PowerSchool → Azure AD groups created per class
3. Intune for Education portal → Express Configuration creates baseline policies in minutes
4. Shared iPad mode enabled for iPads → students log in with Managed Apple ID → their profile loads
5. Student iPad policies: web content filter (block social media, adult content), no App Store access, approved educational apps only (Khan Academy, Duolingo, Google Classroom)
6. Teacher Windows laptops: assigned devices, M365 desktop apps, content creation tools, Microsoft Classroom enabled
7. Take a Test profile configured → IT can lock exam iPads to a single test URL with one policy during exam periods
Result: 600 iPads shared among 2,000 students. Each student gets their personal experience on any iPad. Teachers have dedicated managed laptops. Zero physical device pre-configuration needed.

What is Intune for Education and how does it differ from standard Intune?
Intune for Education is an education-optimized portal at intuneeducation.microsoft.com providing a simplified interface on top of the standard Intune backend. Key differences: Express Configuration wizard for rapid setup, built-in education group templates, School Data Sync integration for automatic student/teacher roster import from SIS systems, Apple School Manager (ASM) support for shared iPad scenarios, and the Take a Test secure assessment configuration. Standard Intune has full depth and complexity; Intune for Education is simplified for school IT admins who may not be endpoint management specialists.
What is School Data Sync and why is it valuable for schools using Intune?
School Data Sync automatically imports school roster data — students, teachers, class groups — from the school's Student Information System into Azure AD. Without SDS, IT would manually create thousands of student accounts and class groups every year. With SDS, the sync runs automatically at the start of each school year: new students get accounts, graduated students are removed, class rosters are updated. These Azure AD groups are then used by Intune to target policies and apps to the right students and teachers automatically.
What is Shared iPad for Education and how does it work with Intune?
Shared iPad is an Apple feature that allows multiple students to use the same iPad with individual user profiles. Each student signs in with their Managed Apple ID (provided via Apple School Manager), and their personal apps, settings, documents, and progress load. When they sign out, the next student's profile loads. IT manages the shared device centrally via Intune — pushing approved apps (which all users see), enforcing web content filters, and controlling device settings. This makes one iPad serve multiple students throughout the day rather than requiring a 1:1 device-to-student ratio.
TOPIC 93
Shared Device Mode — iOS & Android
One device, many users — enterprise shared device configuration for shift workers
The Shared Device Problem

In many industries — retail, healthcare, hospitality, logistics — workers don't have personal assigned devices. Instead, they pick up a shared tablet or phone at the start of their shift, use it for work, then return it. The next shift worker picks it up and uses it. Shared Device Mode makes this work securely: each user signs in, gets their apps and data, then signs out — and the next user starts fresh with zero data leakage between users.

Shared Device Mode — iOS

Azure AD Shared Device Mode for iOS requires the device to be enrolled via ADE (supervised) and uses the Microsoft Authenticator app as the broker. When Shared Device Mode is enabled via Intune:

  • Single sign-in: Worker signs in once through Authenticator — all Microsoft apps (Teams, Outlook, SharePoint) immediately authenticate with their identity. No repeated sign-in per app.
  • Global sign-out: Worker signs out of Authenticator once — ALL apps on the device are signed out simultaneously. The next worker starts with a clean authenticated state.
  • No personal use: Device is company property. Personal Apple IDs and personal app downloads blocked. Worker doesn't mix personal and work on this device.
  • Data cleared on sign-out: Apps built with the MSAL (Microsoft Authentication Library) SDK automatically clear their corporate data when global sign-out occurs.
Configuring iOS Shared Device Mode via Intune
  1. 1
    Enroll device via ADE (supervised iOS device — required for shared device mode)
  2. 2
    Deploy Microsoft Authenticator as a required app via VPP to the device group
  3. 3
    Create an iOS App Configuration Policy for Microsoft Authenticator → Enable Shared Device Mode:
    Key: IntuneMAMUPN
    Value: {{userprincipalname}}
    Key: SharedDeviceMode
    Value: true
  4. 4
    Configure device restrictions: Block App Store (no personal apps), block Apple ID sign-in (no personal iCloud), enable auto-lock (clear screen when shift ends)
  5. 5
    Worker experience: Pick up device → Open Authenticator → Sign in with work credentials → All work apps immediately authenticate → End of shift: sign out of Authenticator → All apps signed out, data cleared → Device ready for next worker
Shared Device Mode — Android Enterprise

On Android, Shared Device Mode works through Android Enterprise Dedicated Device mode + the Microsoft Authenticator app as broker. The concept is the same: single sign-in/sign-out that affects all apps simultaneously.

  • Enrollment: Android Enterprise Dedicated Device (Fully Managed kiosk variant) — QR code, Zero Touch, or Knox enrollment
  • App deployment: Deploy Authenticator and all work apps via Managed Google Play as Required
  • Kiosk launcher: Use Managed Home Screen app for a controlled launcher showing only work apps
  • Sign-in experience: Worker signs into Authenticator → all MSAL-enabled apps authenticate → Authenticator sign-out signs out all apps and clears data
  • App compatibility: Apps must integrate Microsoft's MSAL library for global sign-out to work. Microsoft's own apps (Teams, Outlook, SharePoint) all support this. Custom apps need to integrate MSAL.

Hospital Shared Device Scenario

Hospital: 200 nurses across 3 shifts using 50 shared iPhones for patient communication, EHR access, and task management.
Challenge: If Nurse A leaves patient data visible and Nurse B picks up the same phone, HIPAA is violated.
Solution: ADE-enrolled iPhones + Azure AD Shared Device Mode.
Nurse A starts shift → Signs into Authenticator (Face ID or PIN) → All work apps (Teams, Epic EHR app, patient task app) immediately authenticate with Nurse A's credentials and show her data → End of shift: one tap on "Sign out" in Authenticator → ALL apps sign out simultaneously → Nurse A's patient lists, messages, and data cleared → Phone reset to sign-in screen → Nurse B picks up same phone, signs in, sees only her own patient assignments.
Result: HIPAA compliance maintained. 50 phones serve 200 nurses. Zero personal data mixing. Centrally managed via Intune.

🎤 Interview Questions — Topic 93
What is Shared Device Mode and how does it differ from Kiosk Mode?
Shared Device Mode allows multiple workers to use the same device with their own identity — each worker signs in, gets their personal work data in apps, then signs out which clears all their data for the next user. It uses Microsoft Authenticator as a broker for single sign-in and global sign-out across all MSAL-enabled apps. Kiosk Mode locks the device to one or a few apps with no user identity — everyone sees the same experience. Shared Device Mode is for shift workers needing personalized work access; Kiosk Mode is for anonymous single-purpose devices like self-service terminals.
What is required for global sign-out to work in Shared Device Mode?
Three things: Microsoft Authenticator app must be installed and configured as the broker for Shared Device Mode, the device must be supervised (iOS via ADE) or enrolled as Android Enterprise Dedicated Device, and each app must integrate Microsoft's MSAL (Microsoft Authentication Library) to receive the global sign-out signal. Microsoft's own apps (Teams, Outlook, SharePoint) all support MSAL global sign-out. Custom line-of-business apps need to integrate MSAL for their data to be cleared on global sign-out — without MSAL integration, an app won't receive the sign-out signal and its data won't be cleared.

Frontline Worker Scenarios — Dedicated Kiosk Devices
Managing purpose-built devices for workers who don't sit at desks
Who Are Frontline Workers?

Frontline workers (also called deskless workers or task workers) are the roughly 2 billion people worldwide who work in physical-world roles — retail associates, warehouse pickers, delivery drivers, nurses, factory operators, hotel staff. They don't sit at a desk with a company laptop. Instead they use purpose-built devices: rugged Android scanners, shared tablets, single-purpose kiosk POS terminals, or basic smartphones for task management.

Intune is increasingly important for this population because organizations are equipping frontline workers with mobile technology — and these devices need the same security and management as knowledge worker laptops, but with completely different configurations.

Frontline Worker Device Scenarios
🛍️ Retail
Shared Android tablets or handheld scanners. Apps: inventory lookup, price check, customer order management. Config: Kiosk mode limiting to retail apps, Managed Home Screen, no camera on trading floor (some stores), auto-wipe overnight.
🏥 Healthcare
Shared iPhones/iPads for nurses, shared workstation tablets for bed-side care. Apps: EHR access, patient communication, medication lookup. Config: ADE supervised, Shared Device Mode, HIPAA-compliant data clearing, screen auto-lock in 30 seconds.
🏭 Manufacturing
Rugged Android scanners (Zebra, Honeywell) for warehouse picking and quality control. Apps: WMS (Warehouse Mgmt System), barcode scanning, production tracking. Config: Dedicated device (single app kiosk), OEMConfig for hardware-specific settings, no camera in secure areas.
🚚 Field Services
Personal or company Android/iOS phones for field technicians or delivery drivers. Apps: route navigation, delivery confirmation, service ticket management. Config: MAM-WE for BYOD or fully managed for company phones, location policies, offline capability.

Managed Home Screen — The Frontline Launcher

Microsoft Managed Home Screen (MHS) is a Microsoft app that provides a controlled launcher for Android Enterprise dedicated devices. Instead of the standard Android home screen, workers see only the apps and shortcuts IT approves — in a branded, organized layout IT controls remotely.

  • Custom branding: Company logo, colors, background image. The device feels like a company-branded tool.
  • Approved apps only: IT configures exactly which apps appear. Workers can't access Settings, Google Play, or any other system functions.
  • Virtual home button: A software home button appears since the physical home button is typically disabled in kiosk mode.
  • Notifications: Configurable — show work notifications but block personal ones.
  • Auto sign-out: Combined with Shared Device Mode, MHS can trigger sign-out after inactivity.
  • Configured via: Intune App Configuration Policy for the Managed Home Screen app.
OEMConfig — Managing Hardware-Specific Features

Many frontline devices are rugged Android devices from OEM vendors like Zebra, Honeywell, Datalogic, or Panasonic. These have hardware features (barcode scanner triggers, gun-form-factor buttons, thermal management) that standard Android MDM can't configure. OEMConfig solves this:

  • What it is: OEM-published Android app + JSON schema that exposes the OEM's hardware-specific settings to any MDM (including Intune).
  • How it works: Download the OEM's OEMConfig app from Managed Google Play. Create an App Configuration Policy in Intune targeting the OEMConfig app. Configure hardware-specific settings in JSON (scanner sensitivity, button remapping, display brightness schedules).
  • Example (Zebra): Configure the Zebra scanner trigger button behavior, set the scanning mode (1D/2D/QR), configure beep and vibration on scan, set display timeout for outdoor brightness.
  • No custom development: OEMConfig means you don't need custom MDM agents or proprietary management consoles — everything through standard Intune.
Microsoft Teams for Frontline Workers

Teams has specific features built for frontline workers accessible through Intune-managed devices:

  • Shifts: Schedule management — workers see their shifts, request time off, swap shifts in Teams mobile app.
  • Walkie Talkie: Push-to-talk communication over internet — replacing physical two-way radios for retail, healthcare, and warehouse teams.
  • Tasks by Planner: Daily task checklists pushed by managers to frontline workers' Teams apps — "Open store checklist," "Clean aisle 7," "Restock shelf 12."
  • Approvals: Simple approval workflows from mobile — manager approves time off request, expense, or procedure exception from Teams.
  • Deploy via Intune: Teams deployed as Required app from Managed Google Play or iOS App Store (VPP). App Configuration Policy pre-configures the tenant, disables personal calling, enforces org-specific features.

🎤 Interview Questions — Topic 94
What is Microsoft Managed Home Screen and when would you use it?
Managed Home Screen is a Microsoft app that replaces the standard Android launcher with a controlled, IT-configured home screen on Android Enterprise Dedicated Devices. IT specifies exactly which apps appear in which layout, adds company branding, and blocks access to system settings, Google Play, and unauthorized apps. Workers interact only with approved apps in a branded environment. It's used for frontline/kiosk scenarios — retail associates, warehouse workers, delivery drivers — where the device should be purpose-built and workers shouldn't be distracted or able to install personal apps.
What is OEMConfig and why is it important for frontline device management?
OEMConfig is a standard that allows device OEMs (Zebra, Honeywell, Datalogic) to expose their hardware-specific settings to any MDM solution including Intune. The OEM publishes an OEMConfig app and a JSON schema defining available settings. In Intune, you deploy the OEMConfig app and create an App Configuration Policy using the OEM's schema to configure hardware-specific features — scanner trigger behavior, button remapping, display settings, device health monitoring. Without OEMConfig, configuring these hardware features would require the OEM's proprietary management console instead of Intune.
What Microsoft 365 license would you recommend for a retail associate who only needs Teams, shift management, and a managed device?
Microsoft 365 F1 at approximately $2.25 per user per month. F1 includes Teams (mobile, with Shifts and Walkie Talkie features), SharePoint, Exchange (without Outlook desktop), and full Intune for device management and Azure AD P1. It doesn't include Office desktop apps or Outlook desktop — but retail associates don't need those. For 8,000 frontline workers, the cost difference between F1 and E3 is enormous: F1 costs ~$18,000/month vs E3 at ~$160,000/month — a 9x cost difference for features that frontline workers don't use.

Microsoft Managed Desktop Overview
The premium "endpoint management as a service" — Microsoft runs your endpoint ops
What Is Microsoft Managed Desktop?

Microsoft Managed Desktop (MMD) is a premium cloud-based endpoint management service where Microsoft itself manages your Windows devices — including the Intune configuration, security policies, Windows Update management, hardware, and 24/7 support. It's "Intune as a managed service" — instead of your IT team configuring and operating Intune, Microsoft's operations team does it on your behalf.

Think of it as the difference between owning a car (Standard Intune) and using a chauffeur service (Microsoft Managed Desktop). You still tell the driver where to go (set business requirements), but you don't drive or maintain the vehicle.

The Four Pillars of Microsoft Managed Desktop
🖥️ Hardware
Microsoft-recommended certified devices (Surface, Dell, HP, Lenovo models validated for MMD). Organizations purchase from these certified models. Hardware is pre-validated for compatibility with MMD's configuration baseline.
🔒 Security
Microsoft pre-configures and maintains security baselines, Defender for Endpoint, BitLocker, Conditional Access, ASR rules. The security posture is Microsoft-managed — updated as the threat landscape evolves. Microsoft monitors for security issues.
🔄 Updates
Microsoft manages all Windows quality updates, feature updates, Microsoft 365 Apps updates, and Edge updates — using deployment rings similar to Windows Autopatch. SLA: 95% of devices updated within 21 days. Identical to what Autopatch provides but as a fully managed service.
🎯 Support
Microsoft provides 24/7 operations support for the managed service. Your IT team focuses on line-of-business app deployment and business requirements. Escalation path: your IT → Microsoft support team (not vice versa).

MMD Deployment Rings

Like Autopatch, MMD uses four deployment rings for update management:

  • Test Ring: 1% of devices — immediate updates, no deferral. Catch-all for immediate issues.
  • First Ring: ~9% — 1 day deferral. IT team and early adopters.
  • Fast Ring: ~20% — 6 day deferral. Tech-comfortable users.
  • Broad Ring: ~70% — 9 day deferral. Everyone else.

The difference from DIY Autopatch: Microsoft's operations team monitors these rings, responds to failures, and manages exceptions — your IT team doesn't have to.

MMD Prerequisites and Requirements
  • License: Microsoft 365 E3, E5, or Business Premium (Intune included). MMD service itself has an additional cost — contact Microsoft for pricing as it varies.
  • Hardware: Must use Microsoft-approved MMD device models. Existing non-approved hardware cannot enroll in MMD.
  • Azure AD: Cloud-only or Hybrid Azure AD Join. Pure on-premises AD devices not supported.
  • Network: Devices must reach all Microsoft cloud endpoints directly (or via approved proxy). WSUS, network-based SSL inspection, and certain firewall configurations must be accommodated or removed.
  • No conflicting policies: Existing Intune policies, SCCM co-management, and third-party security tools that conflict with MMD's baseline cannot be in place.

⚠️ MMD Is NOT for Everyone

MMD is a premium, opinionated service. You accept Microsoft's security baseline and update schedule — you cannot significantly deviate. Organizations with complex compliance requirements (defense contractors, government, heavily regulated healthcare) often find MMD too rigid. Organizations heavily invested in SCCM and co-management can't use MMD simultaneously. And organizations with strong internal Intune expertise often find it more cost-effective to manage Intune themselves using Autopatch for the update management piece. MMD's sweet spot is medium-sized organizations with limited IT staff who want enterprise-grade endpoint management without building that expertise internally.

🎤 Interview Questions — Topic 95
What is Microsoft Managed Desktop and how does it differ from standard Intune?
Microsoft Managed Desktop is a premium cloud-based endpoint management service where Microsoft's operations team manages your Windows devices on your behalf — including Intune configuration, security baselines, Windows Update management via deployment rings, hardware support, and 24/7 monitoring. With standard Intune, your IT team configures and operates everything. With MMD, Microsoft does it for you. The tradeoff is cost and control — MMD costs more and you accept Microsoft's opinionated security baseline with limited customization ability.
What types of organizations are NOT good candidates for Microsoft Managed Desktop?
Organizations that are poor MMD candidates: those with heavy SCCM investment and co-management needs (MMD and SCCM co-management conflict), highly regulated environments requiring custom compliance configurations (government, defense contractors, some healthcare), organizations with large, varied hardware fleets (MMD requires certified hardware models), companies with strong internal Intune expertise who can self-manage cost-effectively, and organizations needing to significantly customize the security baseline. MMD is best for organizations with limited IT endpoint expertise, standardized hardware, and standard enterprise compliance needs.
If a customer asks whether they should use Windows Autopatch or Microsoft Managed Desktop, how do you advise them?
Autopatch handles only the update management lifecycle — deploying Windows, M365 Apps, Edge, and Teams updates through deployment rings automatically. The customer still configures and manages all other Intune policies (security, compliance, apps, enrollment). Managed Desktop goes further — Microsoft manages the entire endpoint configuration including security baseline, Defender, BitLocker, compliance, updates, and hardware support. If the customer needs only automated patching, Autopatch is the right choice (and is included in M365 E3 at no extra cost). If they want to outsource the entire endpoint operations function, MMD is the answer — at a higher cost, with standardized hardware requirements.
