TOPIC 29
What Are Configuration Profiles?
Types, purpose, delivery mechanism — the master overview
The Core Concept

A Configuration Profile is a policy in Intune that pushes specific settings to enrolled devices — the cloud equivalent of a Group Policy Object (GPO), delivered over the internet to any device anywhere, targeted to Azure AD groups instead of OUs. Create it once in the Intune portal, assign it to a group, and every device in that group receives the settings on next check-in.

Profile Types per Platform
🖥️ Windows
Settings Catalog, ADMX Templates, Custom OMA-URI, Device Restrictions, Kiosk, Shared Multi-User, Wired Network, Edition Upgrade
📱 iOS/iPadOS
Device Features, Restrictions, Email, Wi-Fi, VPN, Certificate, Kiosk (Single App), Notifications, Web Content Filter
🤖 Android
Enterprise Restrictions, OEMConfig, Wi-Fi, VPN, Certificate, Managed Home Screen, Dedicated Device (Kiosk)
🍎 macOS
Device Features, Restrictions, Custom (PLIST), Wi-Fi, VPN, Certificate, FileVault, System Extensions, Settings Catalog
🌐 All Platforms
Wi-Fi, VPN, Certificate profiles (SCEP, PKCS, Trusted Root) available across all major platforms
How Profiles Are Delivered
  1. 1
    Admin creates profile in Intune → selects platform → configures settings
  2. 2
    Admin assigns to Azure AD group with optional exclusions and filters
  3. 3
    Device checks in every 8 hours automatically, or when Intune pushes a notification after a policy change
  4. 4
    Intune delivers settings via MDM channel — on Windows, written to registry under MDM CSP paths
  5. 5
    Device reports status back: Succeeded / Failed / Conflict / Pending / Not Applicable — visible per device in portal
⚡ Config Profile vs Compliance Policy vs App — Critical Difference

Configuration Profile: Pushes settings TO the device (enforces Wi-Fi, camera block, VPN). Compliance Policy: Checks if the device meets requirements (evaluates BitLocker, OS version). Non-compliant devices can be blocked via Conditional Access. App: Deploys software. All three work together but do very different jobs — confusing them is a red flag in interviews.

🎤 Interview Questions — Topic 29
What is the difference between a Configuration Profile and a Compliance Policy?
A Configuration Profile pushes settings to configure the device — Wi-Fi, VPN, device restrictions. A Compliance Policy checks whether the device meets security requirements — is BitLocker enabled, is the OS version current. Profiles enforce; compliance policies evaluate. Non-compliant devices can be blocked via Conditional Access, but compliance policies don't configure anything themselves.
How quickly does a device receive a new profile after it's assigned?
When a policy is created or changed, Intune sends a push notification to online devices prompting an early check-in — usually within minutes. Without the push, devices check in every 8 hours. You can force immediate sync from Intune portal (Device → Sync) or from the device Settings → Access work or school → Info → Sync.
TOPIC 30
Windows Settings Catalog — Deep Dive
5,000+ searchable settings — the modern replacement for Device Restrictions
What Is the Settings Catalog?

The Settings Catalog is a searchable interface with 5,000+ Windows (and macOS) settings in Intune. Instead of fixed templates, you search for any setting by name and add only what you need. It replaces the older Device Restrictions profile for Windows and grows with every Windows update — the recommended approach for all new profiles.

Real-World Example

Task: Block USB storage on all employee laptops except IT team; disable Bing in Windows Search.
Settings Catalog profile (assigned to All Employees, excluding IT Admins):
→ Search "Removable storage" → Enable "All Removable Storage classes: Deny all access"
→ Search "Search" → Enable "Don't search the web or display web results in Search"
Two settings, one profile, deployed in 10 minutes — 5,000 devices updated on next check-in.

🎤 Interview Questions — Topic 30
What is the Settings Catalog and how does it differ from Device Restrictions?
Settings Catalog is a searchable repository of 5,000+ Windows settings where you add only the settings you need. Device Restrictions is an older fixed-template with a limited pre-selected set. Settings Catalog is more flexible, has vastly more settings, is continuously updated, and is the recommended approach for all new Windows configuration profiles.
How do you block USB storage via Intune?
Create a Windows Settings Catalog profile. Search "Removable storage." Enable "All Removable Storage classes: Deny all access" (or "Removable Disks: Deny write access" for read-only USB). Assign to the target device group. On next check-in, USB storage is blocked.

Administrative Templates (ADMX-backed policies) bring traditional Group Policy settings to Intune — cloud-delivered, no domain join needed. Over 2,500 settings covering Microsoft Office, Edge, OneDrive, Internet Explorer, and Windows. Familiar to any GPO admin — the same tree structure as GPMC.

🎤 Interview Questions — Topic 31
When would you use Administrative Templates over Settings Catalog?
When configuring Microsoft Office app settings (Settings Catalog has limited Office coverage), OneDrive Known Folder Move, Edge enterprise configurations, or third-party apps via custom ADMX import like Chrome or Firefox. Settings Catalog is better for modern Windows security and device settings.
How do you configure OneDrive Known Folder Move via Intune?
Create an Administrative Templates profile. Under Computer Configuration → OneDrive, enable "Silently move Windows known folders to OneDrive" and enter your tenant ID. Enable "Prevent users from redirecting known folders back to their PC." This silently redirects Desktop, Documents, and Pictures to OneDrive for Business without user interaction.

Endpoint Security Profiles vs Configuration Profiles
Two overlapping security paths — understanding the right tool for each scenario
The Overlap and How to Resolve It

Both Devices → Configuration and Endpoint Security can configure BitLocker, Defender AV, and Firewall settings. Configuring the same setting in both causes conflicts. Here is the definitive guidance.

Security Baselines❌✅ Dedicated concept
MDE integrationLimited✅ Deep integration with Defender for Endpoint
Conflict priorityNormalEndpoint Security generally wins for same settings
Best used byGeneral IT adminSecurity team / SOC
Endpoint Security Profile Types
🦠
Antivirus
Defender AV real-time protection, cloud protection, scan schedules, exclusions, PUA blocking.
🔒
Disk Encryption
BitLocker (Windows) and FileVault (macOS). Silent enablement, recovery key escrow to Azure AD.
🔥
Firewall
Windows Firewall per profile (Domain, Private, Public). Enable/disable, inbound/outbound defaults, specific rules.
🛡️
EDR
Onboard devices to Microsoft Defender for Endpoint. Required before MDE can detect and respond to threats.
⚔️

👤
Account Protection

✅ Decision Rule

Use Configuration Profiles for general config: Wi-Fi, VPN, email, restrictions, ADMX, Windows Update, browser. Use Endpoint Security for hardening: Defender AV, BitLocker, Firewall, ASR, EDR, LAPS. Never configure the same setting in both — document which system owns each setting.

🎤 Interview Questions — Topic 32
Can BitLocker be configured in both places? Which is preferred?
Yes, both can configure BitLocker. Prefer Endpoint Security → Disk Encryption — dedicated interface, integrates with Security dashboard, generally wins in conflicts. Never configure the same BitLocker setting in both places — conflicts cause unpredictable effective values.
What are ASR rules and why use Audit before Block mode?
ASR rules block common attack techniques like Office spawning child processes and LSASS credential theft. Audit mode logs what WOULD be blocked without blocking — letting you identify legitimate processes that would be caught. Review Audit logs for 2–4 weeks, confirm no false positives, then switch to Block confidently.

WPA2-Personal vs WPA2-Enterprise
🔑
WPA2-Personal (PSK)
One shared password for all devices. Simple but risky — if leaked, all devices compromised. Password must be changed organization-wide when someone leaves.
🏆
WPA2-Enterprise (802.1X)
Each device authenticates with a unique certificate (EAP-TLS) or credentials. No shared secret. Individual revocation possible. Required for security-conscious enterprise environments.
📡
Autopilot + Wi-Fi Certificate Scenario

Company deploys laptops via Autopilot. Device starts on employee's home Wi-Fi. Intune pushes: 1) Trusted Root cert → 2) SCEP device cert → 3) Wi-Fi profile (EAP-TLS referencing the cert). When employee walks into office, laptop auto-connects to 802.1X corporate Wi-Fi using its certificate — no password, no IT support, completely seamless.

🎤 Interview Questions — Topic 33
What is the benefit of EAP-TLS over PSK in enterprise Wi-Fi?
EAP-TLS uses a unique device certificate instead of a shared password. No shared secret can be leaked. Certificates can be individually revoked per device — a stolen device's cert can be revoked, blocking corporate Wi-Fi access even without password knowledge. Individual device accountability is maintained per connection.
TOPIC 34
VPN Profiles — Always On VPN
Auto-configured secure access to on-premises resources from anywhere
VPN Types in Intune
🔗
Manual VPN
Intune configures the client (server, auth, credentials) but user triggers connect/disconnect. Most common in enterprises where VPN is optional.
🔄
Always On VPN
Auto-connects whenever device has internet. Zero user action. All corporate traffic tunneled. For compliance-heavy environments needing guaranteed connectivity.
📱
Per-App VPN
VPN only activates for specific managed apps. Personal browsing goes direct to internet. BYOD privacy — only corporate app traffic in VPN.
🌐
Split Tunnel
Corporate traffic → VPN. Internet traffic → direct. Reduces VPN server load dramatically. Standard practice since remote work surge.
Windows Always On VPN Setup

Always On VPN (AOVPN) is Microsoft's modern DirectAccess replacement. Uses native Windows IKEv2. Auto-connects, no separate VPN app needed, configured entirely via Intune.

  1. 1
    Deploy certificates first: Trusted Root CA → SCEP user/device certificates for VPN authentication
  2. 2
    Create VPN profile: Windows 10 → VPN → Type: IKEv2 → Server FQDN → Auth: Certificates → Enable Always On
  3. 3
    Device Tunnel vs User Tunnel: Device tunnel = connects before login (for pre-auth AD). User tunnel = after login (user resources). Both can coexist.
  4. 4
    Split tunnel routes: Add on-prem subnets (e.g., 10.0.0.0/8) through VPN. All other traffic goes direct to internet.
  5. 5
    Assign to device group → Deploy. VPN connects silently on every boot — users never see a connect button.

🎤 Interview Questions — Topic 34
What is Always On VPN vs traditional VPN?
AOVPN auto-connects whenever the device has internet — zero user action. Traditional VPN requires the user to launch a client and initiate the connection. AOVPN supports Device Tunnel (connects before login) and User Tunnel (after login). On Windows, it uses native IKEv2 — no separate app required. Fully managed via Intune profiles.
What is split tunneling and why enable it?
Split tunneling routes only corporate traffic through VPN; internet traffic goes directly to the internet. Without it, ALL traffic — YouTube, news, personal browsing — goes through corporate VPN, creating congestion and performance issues especially for large remote workforces. Split tunneling dramatically reduces VPN server load at the cost of less visibility into internet traffic.

Email profiles automatically configure the native email app (iOS Mail, Android Email, Windows Mail) with Exchange Online settings. Employees receive a new device and corporate email is already working — no server addresses, no manual configuration, no support calls about email setup.

💡 Email Profile vs App Configuration Policy for Outlook

Email Profile configures the NATIVE mail app — simple, no separate app required. App Configuration Policy configures the Outlook app specifically — more powerful with Outlook-specific settings (focused inbox, signature, S/MIME integration). For organizations standardized on Outlook (most enterprises), App Configuration Policy is the better choice.

🎤 Interview Questions — Topic 35
Why use Modern Authentication instead of Basic Auth for email profiles?
Basic Auth sends credentials with every request — vulnerable to credential theft. Microsoft disabled Basic Auth for Exchange Online. Modern Auth uses OAuth 2.0 tokens — credentials exchanged once, subsequent requests use short-lived tokens. Supports MFA and Conditional Access, meaning stolen credentials alone aren't enough to access email.
TOPIC 36
Certificate Profiles — SCEP, PKCS & Trusted Root
Deploying digital certificates for passwordless Wi-Fi, VPN, and email authentication
The Three Certificate Profile Types
🌳
Trusted Certificate
Deploys your Root/Intermediate CA cert to the device's trusted store. ALWAYS deploy first — without it the device won't trust your CA-issued certs. Required before any SCEP or PKCS profile.
📋
SCEP Certificate
Device generates its own key pair and requests a unique cert from your CA. Private key NEVER leaves the device. Most secure. Requires NDES server or Cloud PKI. Each device gets its own unique certificate.
🔑
PKCS Certificate
Admin exports cert from CA; Intune delivers it to devices. Simpler setup (no NDES). Private key handled during export — less secure. Same cert can go to multiple devices.
SCEP vs PKCS
AspectSCEPPKCS
Certificate uniqueness✅ Each device gets unique cert⚠️ Same cert can go to multiple devices
Private key✅ Never leaves device⚠️ Admin handles during export
InfrastructureNDES + Connector, or Cloud PKIIntune Certificate Connector only (simpler)
Revocation✅ Per-device revocationRevoking affects all devices with that cert
Best forWi-Fi/VPN auth, large deploymentsSimple setups, email S/MIME
Certificate Deployment Order — Always Follow This
Step 1
Trusted Root CA
→
Step 2
SCEP or PKCS cert
→
Step 3
Wi-Fi/VPN references cert

Wrong order = Wi-Fi fails silently (cert not trusted yet). Trusted Root must succeed before SCEP; SCEP must succeed before Wi-Fi/VPN.

Microsoft Cloud PKI

Setting up NDES on-premises is complex (Windows Server, IIS, CA, NDES role, Certificate Connector). Cloud PKI (part of Intune Suite) is a fully cloud-hosted CA — create it in the Intune portal and issue SCEP certificates with zero server infrastructure.

🎤 Interview Questions — Topic 36
Difference between SCEP and PKCS certificate profiles?
SCEP: device generates its own key pair and requests a unique certificate — private key never leaves the device. Most secure. PKCS: admin exports a cert from CA and Intune delivers it — private key is admin-handled, same cert can go to multiple devices. SCEP is more secure and scalable; PKCS is simpler but less secure.
Why must Trusted Root be deployed before SCEP?
Trusted Root establishes the device's trust in your Certificate Authority. Without it, the device won't recognize certificates your CA issues as valid. When SCEP delivers the identity cert, or when Wi-Fi presents it for 802.1X, the device validates against trusted CAs — if your root isn't trusted, authentication fails even if the certificate is technically valid.
TOPIC 37
Custom OMA-URI Profiles
Configure any Windows MDM setting not exposed in the Intune UI
What Is OMA-URI?

OMA-URI (Open Mobile Alliance Uniform Resource Identifier) is the path format for Windows MDM CSP settings. Every Intune-configurable setting has an OMA-URI. When a setting isn't in Settings Catalog or ADMX Templates, you configure it directly using a Custom OMA-URI profile.

// OMA-URI path format: ./Device/Vendor/MSFT/PolicyArea/SettingName // Block camera (also in Settings Catalog now): ./Device/Vendor/MSFT/Policy/Config/Camera/AllowCamera // Data type: Integer | Value: 0 = disabled // Enable Credential Guard with UEFI lock: ./Device/Vendor/MSFT/Policy/Config/DeviceGuard/LsaCfgFlags // Data type: Integer | Value: 1 // Disable consumer Microsoft account experience: ./Vendor/MSFT/Policy/Config/Experience/AllowWindowsConsumerFeatures // Data type: Integer | Value: 0 // Reference: docs.microsoft.com/windows/client-management/mdm/policy-csp
OMA-URI Data Types
TypeUsed ForExample
IntegerEnable/disable, numeric values0 = disabled, 1 = enabled
StringText, URLs, domain namesHomepage URL, DNS suffix
BooleanTrue/Falsetrue / false
Base64Certificates, binary dataCertificate .cer in Base64
XMLComplex structuresWi-Fi XML, VPN profile XML
💡
When to use OMA-URI: Setting exists in Windows CSP documentation but isn't in Settings Catalog or ADMX yet. Third-party software MDM settings. Granular firewall rules. Always verify the correct path at docs.microsoft.com/windows/client-management/mdm/policy-csp. Test on a single pilot device first — wrong paths silently fail.
🎤 Interview Questions — Topic 37
What is OMA-URI and when would you use a Custom profile?
OMA-URI is the path format for Windows MDM CSP settings. Use Custom OMA-URI when a setting isn't in Settings Catalog or ADMX Templates but exists in Microsoft's Windows MDM CSP documentation. Common scenarios: advanced Defender settings, granular firewall rules, third-party software MDM settings, or new Windows CSPs not yet surfaced in the Intune UI.

🎤 Interview Questions — Topic 39
How do you prevent copying company data from a managed iOS app to a personal app?
Two layers working together: In iOS Device Restrictions, enable "Block sending corporate data to unmanaged apps." Also configure App Protection Policies (MAM) with "Restrict cut, copy, paste" set to "Policy managed apps only." Device Restrictions handles the OS level; MAM handles the app level — together they create complete DLP coverage.
How do you prevent Android factory reset on company devices?
In the Android Enterprise Device Restrictions profile for Fully Managed (COBO) or COPE mode, enable "Block factory reset." This prevents users from resetting the device from settings. Note this only works for company-owned Android Enterprise modes — on BYOD Work Profile devices, users can always remove the work profile (and effectively their management) since they own the device.

Include + Exclude Rules
  • Exclude always wins: Device/user in BOTH included and excluded group → excluded. No exceptions ever.
  • All except VIPs: Include = All Employees / Exclude = VIP Executives (relaxed policies for executives)
  • Pilot rollout: Include = Pilot Group (50 users) → test 2 weeks → add All Employees → remove pilot separately
  • Cannot mix types: User group assignments can only exclude user groups; device group assignments only exclude device groups
Assignment Filters — Granular Targeting Without New Groups
// Filter: Only Windows 11 devices device.osVersion -startsWith "10.0.22" // Filter: Only Dell laptops device.manufacturer -eq "Dell Inc." // Filter: Only Autopilot enrolled devices device.enrollmentProfileName -startsWith "Autopilot" // Usage: Assign to "All Devices" group + apply filter above // Result: Only Dell Windows 11 Autopilot devices → no new group needed
💡
Filters vs Dynamic Groups: Use Dynamic Groups when reusing the same subset across many policies (create once, reference everywhere). Use Filters for one-off targeting without creating new groups. Filters evaluate at assignment time (nearly instant) vs dynamic groups which can take 24 hours — critical for Autopilot scenarios where policies must apply immediately during first boot.
🎤 Interview Questions — Topic 40
When do you assign to a device group vs a user group?
Device group when the setting is hardware-specific or should apply regardless of who logs in — BitLocker (needs TPM check), kiosk mode, Wi-Fi (device always needs corporate Wi-Fi), trusted root certificates. User group when the setting follows the person across devices — email profiles, user VPN, user certs, OneDrive KFM. If the same user uses multiple devices, user group ensures they get the right settings on every device they use.
Why are Filters faster than dynamic groups for Autopilot?
Filters evaluate at policy assignment time — nearly instant. Dynamic group membership can take up to 24 hours after enrollment. During Autopilot's Enrollment Status Page, the device needs policies immediately. If the device isn't in the dynamic group yet, it won't get the profile during setup. Filters eliminate this timing issue entirely, making them the reliable choice for Autopilot deployments.
TOPIC 41
Profile Conflict Resolution
What happens when two profiles set the same setting differently — and how to fix it
How Intune Handles Conflicts
ScenarioResult
Two profiles set same setting to SAME value✅ No conflict — setting applies normally
Two profiles set same setting to DIFFERENT values⚠️ Conflict — both report Error/Conflict status. Setting may revert to default or remain at last applied value. Unpredictable behavior.
Profile A sets setting X; Profile B doesn't mention X✅ No conflict — Profile A's value applies cleanly
Endpoint Security vs Settings Catalog for same settingEndpoint Security wins — designed to have priority for security settings
Real Conflict Examples
⚔️ Conflict: Password Length
Profile A (All Devices): Minimum password = 6 digits
Profile B (Security Team): Minimum password = 12 characters
Security Team member receives BOTH profiles.
Result: Conflict. Both show Error. Effective password length unpredictable. Fix: Exclude "Security Team" from Profile A, OR create one consolidated profile per scenario — never overlap the same setting.
⚔️ Conflict: Camera Block vs Allow
Profile A (All Employees): Block camera = Yes
Profile B (Design Team): Block camera = No (designers need cameras)
Design Team member receives both.
Fix: In Profile A's assignment, Exclude "Design Team." Now only Profile B reaches Design Team members — clean, no conflict.
✅ No Conflict: Different Settings
Profile A: Blocks camera. Profile B: Configures Wi-Fi. Different settings — no overlap.
Result: Both apply cleanly. Camera blocked AND Wi-Fi configured. Intune merges non-conflicting settings. Only same setting + different values triggers conflicts.
Resolving Conflicts — Step by Step
  1. 1
    Intune → Devices → All Devices → Click affected device → Device configuration tab → Find profiles with "Conflict" or "Error" status
  2. 2
    Click the conflicting profile → identify the specific setting in error
  3. 3
    Find ALL profiles that configure that same setting (review each profile in the tenant)
  4. 4
    Resolution options: (a) Remove setting from one profile, (b) Use Include/Exclude so only one profile reaches the device, (c) Consolidate into one authoritative profile
  5. 5
    Trigger device sync → verify Conflict status resolves to Succeeded in Device configuration tab
💡 Architecture Principle — Avoid Overlapping Profiles

Best prevention: One baseline profile (All Devices) for universal settings. Role-specific profiles for department variations — never duplicate baseline settings. Use exclusions liberally — if a group needs different behavior for one setting, exclude them from the baseline for that setting rather than creating a competing profile. Document which profile "owns" which settings.

What happens when two profiles set the same setting to different values?
A conflict occurs. Both profiles report Conflict or Error status in Intune for that setting. The effective value becomes unpredictable — it may stay at last applied value, revert to default, or fluctuate. Resolution requires removing the setting from one profile, using exclusions so only one profile reaches the device, or consolidating into a single authoritative profile.
How would you architect profiles to minimize conflicts in a 1,000-person org?
Layered approach: one baseline profile assigned to All Devices covering universal settings. Role-specific profiles for department groups — critical rule: never duplicate a baseline setting in a role-specific profile. The Troubleshooting Hierarchy — Always Work in Order
  1. 1
    Is the device enrolled? → Intune → Devices → All Devices → Search. No device = enrollment issue, not a profile issue.
  2. 2
    Is the profile assigned to a group the device/user belongs to? → Open profile → Assignments → Check included groups. Check device/user's group memberships in Azure AD.
  3. 3
    Is the device in an excluded group? → Check Exclude groups in the profile. Exclusion overrides inclusion — if excluded, profile won't apply by design.
  4. 4

  5. 5
    What is the profile status for this device? → Devices → Configuration → Click profile → Device and user check-in status → Find device → read Status column.
  6. 6
    Is the setting applicable to this OS version? → Some Settings Catalog settings require specific Windows versions. Check the setting's documentation for minimum requirements.

Common "Profile Not Applying" Scenarios
🚨 Scenario: Pending for 24+ Hours

Cause: Device is offline, MDM agent is unhealthy, or device recently unenrolled/re-enrolled. Check: Is device online? In Intune → device → try Sync button. On device: Settings → sync manually. If still pending, run MDMDiagReport and check MDM channel health errors in Event Viewer.

⚠️ Scenario: Succeeded Status But Setting Not Applied

Cause 1: Another profile with different value is overriding (check for conflicts). Cause 2: Setting requires a device restart to take effect — reboot the device. Cause 3: CSP path changed in a newer OS version. Check: Verify the registry key directly on device. Check MDMDiagReport for the actual CSP value stored.

⚠️ Scenario: Not Applicable Unexpectedly

Cause 1: An Assignment Filter is excluding this device — review filter rules. Cause 2: Profile platform mismatch (Windows profile assigned via group containing Android devices). Cause 3: Device OS version below the setting's minimum requirement. Always check platform requirements and filter configuration.

Real Troubleshooting Case — Wi-Fi Fails After Adding Certificate Profile

Ticket: "Our Wi-Fi profile worked fine, but after adding a new SCEP certificate profile, devices stopped auto-connecting to corporate Wi-Fi."
Investigation: Wi-Fi profile status = Failed. Error: "Certificate not found." SCEP certificate profile status = Pending on same devices.
Root cause: Both profiles assigned to a device group. Dynamic group membership took 18 hours to populate — devices got the Wi-Fi profile before receiving the SCEP certificate it depends on. Wi-Fi requires the cert but the cert hadn't arrived yet → authentication fails.
Fix: Wait for SCEP profile to show Succeeded → Wi-Fi auto-succeeds on next check-in. Future prevention: Use Assignment Filters (instant evaluation) instead of dynamic groups for Autopilot/certificate-dependent scenarios, ensuring cert arrives before Wi-Fi profile triggers.

🎤 Interview Questions — Topic 42
Walk me through troubleshooting a Wi-Fi profile that isn't connecting devices to corporate Wi-Fi.
Confirm the device is enrolled. Check the Wi-Fi profile assignment — is the device/user's group included, not excluded? Check the profile's per-device status in Intune — Succeeded/Failed/Conflict? If Failed, examine the error detail. If the profile uses 802.1X certificates, verify the Trusted Root and SCEP profiles also show Succeeded — certificate failures cascade to Wi-Fi failures. On Windows, run MDMDiagnosticsTool and check Event Viewer MDM logs. Trigger a manual sync and recheck.
What does "Not Applicable" mean for a Configuration Profile and what are common causes?
Not Applicable means Intune determined the profile was assigned to the device but correctly doesn't apply it. Common causes: an Assignment Filter is excluding this specific device based on its attributes, the device OS is below the profile's minimum requirement, or there's a platform mismatch (e.g., a Windows-only profile somehow reaching an Android device's group). It's not an error — it's Intune correctly not applying an inapplicable policy.
Where in Intune do you check all profiles applied to a specific device?
Devices → All Devices → Find and click the device → Device configuration tab. Shows every assigned profile with its status. Also use Troubleshoot & Support → search the user → see all their devices and profile states from one screen. The per-device view in Device configuration also shows which specific settings within a profile succeeded or failed.
