TOPIC 51
App Types in Intune
Every app format Intune supports — what each is, when to use it
The App Type Landscape

Intune supports many different app formats across platforms. Choosing the wrong app type wastes hours of troubleshooting. Here's the complete map — learn it cold because interviewers love asking about it.

🖥️ Win32 App
Standard Windows .exe or .msi installers, packaged into .intunewin. Most powerful, most complex. Supports detection rules, requirements, dependencies, supersedence. Required for most real enterprise apps (Adobe, AutoCAD, Chrome, custom apps).
🏪 Microsoft Store App
New Microsoft Store integration (WinGet-backed). Modern MSIX/UWP apps from the Store. Auto-update. No packaging needed. Replaces the old deprecated Microsoft Store for Business.
📦 Line-of-Business (LOB)
Upload your own .msi, .msix, .appx, .ipa, .apk directly to Intune. Simpler than Win32 — no IntuneWin packaging. Limited features (no detection rules, no dependencies). Good for simple MSI apps.
🌐 Web App (Link)
Creates a shortcut/bookmark to a URL on the device's desktop or home screen. No installer. Zero bandwidth. Used for web-based tools, SaaS apps, SharePoint sites. Available across all platforms.
📊 Microsoft 365 Apps
Deploy the full M365 suite (Word, Excel, Outlook, Teams) or individual apps. Uses the Office Deployment Tool (ODT) behind the scenes. Configurable via Intune — choose apps, update channel, languages, architecture.
🔧 Built-in App
Pre-vetted Microsoft apps — Company Portal, Edge, Authenticator, OneDrive. One-click deploy, no packaging. Platform-specific. Auto-updated by Microsoft. Always use these for Microsoft's own apps instead of packaging them yourself.
Platform-Specific App Types
PlatformApp Types AvailableNotes

iOS/iPadOSApp Store (VPP), LOB (.ipa), Web clip, Built-in, Custom app (private B2B)VPP (via ABM) for paid/free App Store apps without personal Apple ID

💡 Win32 vs LOB — Which to Use for Windows?

Use Win32 when you need: detection rules to verify install success, requirement rules (minimum disk space, OS version, PowerShell pre-check), dependencies (install app A before app B), or supersedence (replace an old version). Use LOB only for simple .msi or .msix apps where none of these advanced features are needed. In practice, Win32 is the standard for most enterprise app deployments.

🎤 Interview Questions — Topic 51
What is the difference between a Win32 app and a LOB app in Intune?
Win32 apps are packaged into .intunewin format using IntuneWinAppUtil.exe and support advanced features like custom detection rules, requirement rules (PowerShell checks, OS version, disk space), app dependencies, and supersedence. LOB (Line-of-Business) apps are raw .msi/.msix/.ipa files uploaded directly — simpler setup but no detection rules, no dependencies, and limited troubleshooting visibility. Win32 is the enterprise standard for Windows app deployment.
What app type would you use to deploy Google Chrome to 500 Windows laptops via Intune?
Win32 app. Download the Chrome enterprise MSI or EXE installer. Package it with IntuneWinAppUtil.exe into a .intunewin file. Upload to Intune as a Win32 app. Set install command (msiexec /i GoogleChrome.msi /quiet), uninstall command, and a detection rule (registry key or file path). Assign as Required to the All Devices or target device group. Devices receive and silently install Chrome on next check-in.
TOPIC 52
Win32 App Packaging — IntuneWinAppUtil.exe Step by Step
Converting any Windows installer into an Intune-deployable .intunewin package
What Is IntuneWinAppUtil?

IntuneWinAppUtil.exe is Microsoft's free command-line tool that converts any Windows installer (EXE, MSI, scripts) into an .intunewin file — the only format Intune accepts for Win32 app deployment. It zips and encrypts your installer so Intune can securely deliver it to managed devices. The process is simple but must be done correctly.

Download and Setup
# Download IntuneWinAppUtil from Microsoft GitHub: https://github.com/microsoft/Microsoft-Win32-Content-Prep-Tool # Download the latest release .exe # No installation needed — it's a standalone executable # Place it in a working folder, e.g. C:\IntunePackaging\
Step-by-Step Packaging Process
  1. 1
    Prepare your source folder: Create a folder containing ONLY the installer files needed. Example: C:\IntunePackaging\Source\Chrome\ containing GoogleChromeStandaloneEnterprise64.msi
  2. 2
    Run IntuneWinAppUtil:
    # Syntax: IntuneWinAppUtil.exe -c <source_folder> -s <setup_file> -o <output_folder> # Example for Chrome MSI: IntuneWinAppUtil.exe -c "C:\IntunePackaging\Source\Chrome" -s "GoogleChromeStandaloneEnterprise64.msi" -o "C:\IntunePackaging\Output" # Parameters: # -c Source folder containing all installer files # -s The main setup file (the entry point installer) # -o Output folder where .intunewin will be saved
  3. 3
    Output: A file named GoogleChromeStandaloneEnterprise64.intunewin appears in the output folder. This is your upload-ready package.
  4. 4
    Upload to Intune: Apps → Windows → Add → App type: Windows app (Win32) → Select the .intunewin file → Configure install commands, detection rules, requirements → Assign → Publish.
Key IntuneWinAppUtil Flags
FlagPurposeExample
-cSource content folder — all files the installer needs-c "C:\Source\MyApp"
-sSetup file — the main executable or MSI-s "setup.exe"
-oOutput directory for the .intunewin file-o "C:\Output"
-qQuiet mode — suppress prompts for automation-q
What's Inside an .intunewin File?

An .intunewin file is actually a ZIP file containing:

  • IntunePackage.intunewin: An AES-256 encrypted ZIP of all your source files
  • Detection.xml: Metadata about the package (name, version, install command, file size)

You can rename it to .zip and extract it to inspect the contents. The encryption key is generated during packaging and embedded in the detection XML — Intune uses it to decrypt during deployment on the device.

Packaging Best Practices
  • Always test the installer silently first: Before packaging, run setup.msi /quiet /norestart or setup.exe /S manually on a test machine. If it doesn't install silently, it won't work via Intune either.
  • Keep source folders clean: Only include files the installer actually needs. Unnecessary files increase package size and upload time.
  • Document your install/uninstall commands: Write these down when you package so you don't have to figure them out again in 6 months.
  • Version your packages: Name your output folders with version numbers: Chrome_108.0.5359.98\. Makes supersedence easy to manage.
  • Test on a pilot device first: Always assign as Required to a test group, verify it installs correctly, THEN expand to All Devices.
📦
Packaging Example — 7-Zip for Enterprise

Goal: Deploy 7-Zip 23.01 to all Windows company laptops silently.
Step 1: Download 7-Zip enterprise MSI: 7z2301-x64.msi to C:\Packaging\Source\7zip\
Step 2: Test silent install: msiexec /i 7z2301-x64.msi /quiet /norestart → Verify 7-Zip appears in Programs.
Step 3: Package: IntuneWinAppUtil.exe -c "C:\Packaging\Source\7zip" -s "7z2301-x64.msi" -o "C:\Packaging\Output"
Step 4: Upload to Intune → Install command: msiexec /i "7z2301-x64.msi" /quiet /norestart
Uninstall command: msiexec /x "{23170F69-40C1-2702-2301-000001000000}" /quiet /norestart (use the MSI product GUID)
Step 5: Detection rule: MSI Product Code or registry key. Assign to pilot group → verify → assign to All Devices.

🎤 Interview Questions — Topic 52
What is IntuneWinAppUtil and what does it do?
IntuneWinAppUtil.exe is Microsoft's free command-line packaging tool that converts Windows installers (EXE, MSI, scripts) into .intunewin files — the only format accepted by Intune for Win32 app deployment. It zips and AES-256 encrypts all source files so Intune can securely deliver them to managed devices. The tool takes three parameters: source folder, setup file (entry point), and output directory.
Can you use IntuneWinAppUtil with a PowerShell script as the setup file?
Yes. IntuneWinAppUtil accepts any file as the setup file, including PowerShell scripts. Package the .ps1 and any supporting files as the source, specify the .ps1 as the setup file. In Intune, set the install command to: powershell.exe -ExecutionPolicy Bypass -File "script.ps1". This is useful for complex deployments that need pre/post-install tasks, registry modifications, or multi-file operations that a simple MSI can't handle.
TOPIC 53
Win32 App Deployment — Detection Rules, Requirements & Dependencies
The three pillars that make Win32 apps reliable at enterprise scale
The Three Pillars of Win32 Deployment

What separates Win32 from LOB apps is its three advanced features: Detection Rules (how Intune knows if the app is installed), Requirement Rules (prerequisites before installing), and Dependency (other apps that must install first). Together they make Win32 deployments reliable and self-healing.

Pillar 1 — Detection Rules

Detection rules tell Intune how to determine if the app is already installed on the device. Without a correct detection rule, Intune doesn't know if it needs to install the app, or if the app was successfully installed after deploying. This is the most critical Win32 setting.

Detection TypeHow It WorksBest Used ForExample
MSI Product CodeIntune checks if an MSI with this GUID is installed in Windows Installer databaseMSI-based installers — most reliable for MSIsProduct code: {23170F69-40C1-2702-2301-000001000000}
File/FolderCheck if a specific file or folder exists (optionally check version or date)EXE installers, apps that don't register as MSIFile: C:\Program Files\App\app.exe version ≥ 3.2
RegistryCheck if a registry key/value exists (optionally check value data)Most apps — write version to HKLM\Software on installHKLM\SOFTWARE\Google\Chrome: Version = 108.x
PowerShell ScriptRun a custom PS1 script — if it exits with code 0 = detected, non-zero = not detectedComplex detection logic — multiple conditions, custom checksPS1 that checks multiple registry keys AND a file AND a service status
Finding the MSI Product Code
# Method 1: From the MSI file using PowerShell $msi = "C:\Path\To\installer.msi" $shell = New-Object -ComObject WindowsInstaller.Installer $db = $shell.GetType().InvokeMember("OpenDatabase", "InvokeMethod", $null, $shell, @($msi, 0)) $view = $db.GetType().InvokeMember("OpenView", "InvokeMethod", $null, $db, @("SELECT Value FROM Property WHERE Property='ProductCode'")) $view.GetType().InvokeMember("Execute", "InvokeMethod", $null, $view, $null) $record = $view.GetType().InvokeMember("Fetch", "InvokeMethod", $null, $view, $null) $record.GetType().InvokeMember("StringData", "GetProperty", $null, $record, 1) # Method 2: Check registry after manual install # HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\ # Look for the app and note its PSChildName (the product code GUID)
Pillar 2 — Requirement Rules

Requirement rules define prerequisites that must be met before the app can be installed. If a device fails a requirement, Intune won't even attempt the installation — it shows "Not Applicable" status for that device.

Requirement TypeWhat It ChecksExample Use Case
Operating systemMinimum/maximum Windows versionApp requires Windows 11 22H2 minimum — don't push to Windows 10 devices
Disk spaceFree disk space in MBAdobe Acrobat requires 2000MB free — skip devices with full disks
Physical memory (RAM)Minimum RAM in MBCAD software requires 8GB RAM minimum
Processor speed (MHz)Minimum CPU speedEncoding software requires 2GHz minimum
Logical processorsMinimum CPU core countMulti-threaded software needs 4+ cores
PowerShell scriptCustom PS1 — exit 0 = requirement met, exit 1 = not metCheck if a specific software is already installed, or if device is on VPN, or custom hardware detection
RegistryCheck registry key/value existsOnly install plugin if the parent app is already installed
FileCheck file/folder existsOnly install update if base version is present
Pillar 3 — App Dependencies

Dependencies tell Intune to automatically install another app first before this one. If App B depends on App A (like a plugin depending on its host application), Intune installs App A first, waits for success, then installs App B. No manual sequencing needed.

  • Auto-install dependency: When checked, Intune installs the dependency app automatically before the main app — even if the dependency isn't separately assigned to the user/device.
  • Dependency chain: App A depends on App B which depends on App C — Intune resolves the chain and installs in the correct order automatically.
  • Circular dependency prevention: Intune detects and blocks circular dependencies (A → B → A).
🔗
Real-World Dependency Example

Scenario: You need to deploy a custom .NET 6 application (CompanyApp.exe) to all laptops.
Problem: CompanyApp.exe requires .NET 6 Runtime to be installed first. If .NET isn't there, CompanyApp fails silently.
Solution with Dependencies:
1. Create Win32 app: ".NET 6 Runtime" (using the .NET 6 offline installer)
2. Create Win32 app: "CompanyApp v2.3"
3. In CompanyApp's Dependencies tab → Add .NET 6 Runtime → Check "Auto-install"
Result: When Intune deploys CompanyApp to a device, it first checks if .NET 6 is installed. If not, it installs .NET 6 automatically. Once .NET 6 succeeds, CompanyApp installs. Zero manual pre-requisite management. Works even if .NET 6 isn't separately assigned to the device.

🎤 Interview Questions — Topic 53
What are detection rules in Intune Win32 app deployment and what types are available?
Detection rules tell Intune whether an app is already installed on a device. Without them, Intune doesn't know if deployment succeeded or if it needs to retry. Four types: MSI Product Code (best for MSI installers — checks Windows Installer database), File/Folder (checks for a specific file with optional version check), Registry (checks for a registry key/value), and PowerShell Script (custom logic for complex scenarios — exit 0 means detected, non-zero means not detected).
What is the difference between a dependency and a requirement rule in Intune Win32 apps?
A Requirement Rule is a prerequisite check — the device must meet the condition before Intune even attempts installation (OS version, disk space, RAM, registry check). If the requirement fails, the app is "Not Applicable" for that device. A Dependency is another Intune app that must be installed first — Intune automatically installs the dependency before the main app. Requirements check device state; dependencies install missing software prerequisites.

What happened to Microsoft Store for Business and what replaced it?
Microsoft Store for Business was retired in March 2023. It was replaced by the new Microsoft Store app type in Intune, powered by WinGet (Windows Package Manager). Apps are searched and added directly within the Intune Apps section, making it significantly simpler to deploy common apps like Zoom, Slack, and VLC.
TOPIC 55
iOS App Store & VPP App Deployment
Deploying iOS apps at enterprise scale without personal Apple IDs
The Problem VPP Solves

In a consumer context, downloading an App Store app requires a personal Apple ID and sometimes a payment method. In enterprise, you can't ask 2,000 employees to use personal Apple IDs for work apps, and you can't share a corporate Apple ID across thousands of devices. VPP (Volume Purchase Program, now part of Apple Business Manager) solves this by letting organizations buy app licenses in bulk and assign them to devices or users — without any Apple ID needed on the device.

Device Licensing vs User Licensing
📱
Device-Based Licensing
App license is assigned to the device, not the user. Any user who picks up that device can use the app. No Apple ID required on device. Ideal for shared devices, kiosks, and supervised iPads. License stays on device even if a different user signs in.
👤
User-Based Licensing
App license is assigned to the user's Managed Apple ID. User can install on up to 5 of their devices. Requires Managed Apple IDs from ABM. License follows the user, not the device — good for personal-use corporate iPhones.
Deploying iOS Apps via VPP
  1. 1
    Purchase app licenses in ABM: Go to business.apple.com → Apps and Books → Search app → Buy licenses. Free apps still need to be "purchased" (at $0) to be manageable via VPP.
  2. 2
    Sync ABM token with Intune: Ensure your VPP token is connected (Tenant Administration → Apple VPP tokens). Apps purchased in ABM automatically appear in Intune after token sync (up to 15 minutes).
  3. 3
    Find the app in Intune: Apps → iOS/iPadOS → the app should appear in the list (synced from ABM). Or add it manually: Add → iOS Store App → search → Select VPP license.
  4. 4
    Configure assignment: Assign as Required or Available. Select Device or User licensing. Assign to target group.
  5. 5
    Intune deploys silently: On supervised devices, apps install without any user interaction or App Store prompts. On unsupervised BYOD devices, app appears in Company Portal for user to install.
License Reclamation

One of VPP's most valuable features: when an employee leaves, you can reclaim their app licenses and reassign them to a new employee. Without VPP, paid apps downloaded on personal Apple IDs can't be reclaimed — the license stays with the ex-employee's personal account.

  • Automatic reclamation: When a device is retired/wiped or a user's license assignment is removed, the VPP license is automatically reclaimed.
  • Manual reclamation: Intune → Apps → find the app → Licenses → View device/user assignments → Remove assignment from a specific device/user.
🎤 Interview Questions — Topic 55
What is VPP and why is it needed for iOS enterprise app deployment?
VPP (Volume Purchase Program, now part of Apple Business Manager) lets organizations buy iOS app licenses in bulk and deploy them to devices without requiring personal Apple IDs. Without VPP, each App Store app download requires a personal Apple ID — impractical for enterprise. VPP separates app licenses from personal identity, allows silent app deployment to supervised devices, enables license reclamation when employees leave, and supports both device-based and user-based licensing models.
What is the difference between device-based and user-based VPP licensing?
Device-based licensing assigns the app to the device — any user can access it and no Apple ID is needed. Ideal for supervised, shared, or kiosk devices. User-based licensing assigns to a user's Managed Apple ID — the license follows the user across up to 5 of their Apple devices, requiring Managed Apple IDs from Apple Business Manager. Device-based is simpler for corporate-owned supervised iOS devices; user-based is better when users move between devices.
TOPIC 56
Android Managed Google Play App Deployment
The enterprise app channel for Android — silent deployment without personal Google accounts
What Is Managed Google Play?

Managed Google Play is Google's enterprise version of the Play Store, integrated with Intune. It provides a curated, IT-controlled app store for Android Enterprise devices. Employees on Fully Managed or Work Profile devices see only the apps IT approves — not the public Play Store (which can be blocked on corporate-owned devices).

Connecting Managed Google Play to Intune
  1. 1
    Intune → Devices → Android → Android enrollment → Managed Google Play → Connect. This links your Intune tenant to a Managed Google Play account.
  2. 2
    You're redirected to Google's enterprise signup. Sign in with a corporate Google account (or create one). This becomes your Managed Google Play admin account.
  3. 3
    Approve the integration. The connection is established — Intune can now sync apps from Managed Google Play.
Types of Apps in Managed Google Play
🏪
Public Play Store Apps
Any app from the public Google Play Store — Slack, Zoom, Salesforce, SAP. IT approves them in Managed Google Play portal first, then they appear in the managed store available to users. Employees see only approved apps.
🔒
Private (LOB) Apps
Upload your own .apk directly to Managed Google Play as a private app. Only your organization's devices can see and install it. No Google review required for private apps. Ideal for internal custom Android apps.
🌐
Web Apps
Create web app shortcuts that appear as app icons on Android devices. Points to a URL — runs in a managed Chrome session. No APK needed. Good for web-based enterprise tools.
🏭
OEM / System Apps
Pre-installed device apps specific to a manufacturer (Samsung, Zebra, Honeywell). Enabled or disabled via OEMConfig profiles or specific system app enablement in Intune. No download needed.
Approving and Deploying a Play Store App
  1. 1
    In Intune: Apps → Android → Add → Managed Google Play app
  2. 2
    Search for the app (e.g., "Microsoft Teams"). Click the app → Click Approve — this adds it to your Managed Google Play catalog and accepts the app's permissions on behalf of users.
  3. 3
    After approving, click Sync or wait 15 minutes for it to appear in your Intune app list.
  4. 4
    Find the app in Intune → Assign as Required (silent install) or Available (user-initiated from Managed Play Store).
  5. 5
    On Android Enterprise devices, Required apps install silently without user interaction. Available apps appear in the device's restricted Play Store.
⚠️ Permission Handling for Play Store Apps

When you approve an app in Managed Google Play, you accept its required Android permissions on behalf of your organization. This means users won't see "Allow Camera access?" prompts for work apps — you pre-approve permissions. Be careful: only approve apps that genuinely need the requested permissions. Review permissions before approving. You can also use App Configuration policies to auto-grant specific runtime permissions silently on Android Enterprise devices.

🎤 Interview Questions — Topic 56
What is Managed Google Play and how does it differ from the regular Google Play Store?
Managed Google Play is Google's enterprise app channel — IT controls which apps appear on corporate Android devices. Regular Play Store shows all public apps to users without IT control. With Managed Google Play connected to Intune, only IT-approved apps appear in the device's Play Store. Required apps install silently without user interaction. IT can also publish private LOB apps that are invisible to the public. It also handles permissions pre-approval so users don't see permission prompts for work apps.
TOPIC 57
Microsoft 365 Apps Deployment via Intune
Deploying Word, Excel, Outlook, Teams to Windows devices using the ODT integration
Why M365 Apps Has a Dedicated Intune App Type

Microsoft 365 Apps (formerly Office 365 ProPlus) is complex to deploy — it's a suite of products, supports different update channels, requires click-to-run technology, and has many configuration options. Intune has a dedicated app type for M365 Apps that uses the Office Deployment Tool (ODT) behind the scenes, with a GUI configuration experience instead of XML editing.

Configuring M365 Apps Deployment
  1. 1
    Intune → Apps → Windows → Add → App type: Microsoft 365 Apps for Windows 10 and later
  2. 2
    App Suite Configuration: Select which apps to include — Word, Excel, PowerPoint, Outlook, Teams, OneNote, Publisher, Access, Visio (if licensed). Exclude apps the organization doesn't use.
  3. 3
    App Suite Settings:
    • Update channel: Current Channel (latest features fast), Monthly Enterprise (monthly, tested), Semi-Annual Enterprise (6-month cycle, most stable for enterprises)
    • Version: Latest or pin to a specific version
    • Architecture: 64-bit (recommended) or 32-bit (legacy compatibility)
    • Languages: Add languages to install alongside English
    • Accept EULA: Yes — pre-accept for silent install
    • Shared computer activation: Enable for VDI/RDS environments where multiple users share one machine
  4. 4
    Assign as Required to target groups → Deploy. Devices download M365 from Microsoft's CDN and install silently. ~3–5GB download per device.

Shared Computer Activation for VDI/RDS: In Virtual Desktop Infrastructure (VDI) or Remote Desktop Services environments, multiple users share one Windows session. Without this, Office shows "Not licensed" errors for secondary users on the same VM.
🎤 Interview Questions — Topic 57
Which Microsoft 365 update channel would you recommend for a regulated healthcare organization?
Semi-Annual Enterprise Channel. Healthcare organizations require maximum stability — they can't afford Office crashes or unexpected behavior changes interfering with clinical workflows. Semi-Annual Channel releases major feature updates only twice per year (January and July), giving IT time to thoroughly test updates before they hit clinical workstations. Security updates still arrive monthly on Patch Tuesday. The tradeoff is being 6 months behind on new features, which is acceptable in regulated environments.
What is Shared Computer Activation and when is it needed?
Shared Computer Activation is a Microsoft 365 Apps licensing mode for environments where multiple users share one Windows machine — like VDI (Virtual Desktop Infrastructure) or RDS (Remote Desktop Services). Without it, Office activates to the first user who signs in and other users get "unlicensed" errors. With Shared Computer Activation, each user's license is verified from Microsoft's cloud on sign-in, allowing concurrent activation across multiple users on the same machine.
TOPIC 58
App Configuration Policies
Pre-configuring app settings without user action — for managed devices and managed apps
What Are App Configuration Policies?

App Configuration Policies push settings INTO a managed app — configuring the app's behavior before the user first opens it. Think of it as the app-level equivalent of a Device Configuration Profile. Instead of configuring the device's OS, you're configuring a specific app's settings (email server, default account, feature toggles).

Two Types of App Configuration Policies
💻
Managed Devices
Applies to MDM-enrolled devices. App receives configuration via the MDM channel. Works with any managed app that supports configuration. The device must be enrolled in Intune. Configuration delivered reliably with device check-in.
📱
Managed Apps (MAM)
Applies to apps protected by App Protection Policies — even on non-enrolled (BYOD) devices. App receives config via the Intune MAM SDK. Works even without device enrollment. Used for BYOD scenarios.

TOPIC 63
Required vs Available vs Uninstall — Deployment Intent
Controlling how and whether an app reaches each device
The Three Deployment Intents
🔴 Required
App is FORCED to every device in the assigned group. Installs silently in the background without user action. User cannot prevent installation. If uninstalled by user, Intune reinstalls it on next sync. Use for: security tools, corporate apps everyone must have.
🟢 Available
App appears in the Company Portal for users to install if they want. Not forced. User chooses to install. For enrolled devices, or for BYOD users via Company Portal (web or app). Use for: optional productivity apps, departmental tools not everyone needs.
🟡 Uninstall
Intune actively REMOVES the app from devices in the assigned group. Used to clean up apps that should no longer be on certain devices — e.g., removing a legacy app after migration, or removing a department's app when a user transfers. Works silently.
Intent Assignment Rules
  • Required overrides Available: If the same user is in a "Required" group and an "Available" group for the same app, Required wins — the app installs silently.
  • Required overrides Uninstall: If a user is in both Required and Uninstall groups, Required wins — app stays installed.
  • Uninstall vs Available: If a user is in both Uninstall and Available groups, the app is uninstalled (Uninstall wins over Available).
  • Exclude overrides all: If excluded from a Required assignment, the app is not installed — regardless of other assignments.
Available for Enrolled + BYOD Unenrolled Devices

The "Available" intent has two sub-options:

  • Available for enrolled devices: App appears in Company Portal (app or website) for enrolled MDM devices.
  • Available with or without enrollment: Available for MDM-enrolled AND unenrolled (BYOD MAM-WE) users via the Company Portal website. Great for BYOD where users access the Company Portal web at portal.manage.microsoft.com to install optional apps on their personal devices.
🎤 Interview Questions — Topic 63
What is the difference between Required and Available app assignment in Intune?
Required installs the app silently and automatically on all devices in the assigned group — the user cannot prevent it, and if they uninstall it, Intune reinstalls it on next sync. Available makes the app appear in the Company Portal for users to optionally install at their discretion. Use Required for mandatory corporate apps (security tools, core productivity apps) and Available for optional departmental or productivity tools that not everyone needs.
If a user is in both a "Required" and "Uninstall" assignment group for the same app, what happens?
Required wins. The app is installed (and stays installed). Required intent has the highest priority — it overrides both Available and Uninstall intents. The only exception is Exclude, which removes the device/user from receiving any assignment. If you truly need to uninstall an app for a specific group while keeping it required for others, use Exclude in the Required assignment instead of an Uninstall assignment.

Supersedence — Replacing Old Versions

Supersedence tells Intune that this new app version should replace an older version. When you configure supersedence, Intune automatically uninstalls the old app and installs the new one — without you needing to change assignments or manually uninstall the old version first.

How Supersedence Works
  1. 1
    You have Chrome 108 deployed as a Win32 app to All Devices.
  2. 2

  3. 3
