SCCM + Intune
Better Together

Most large enterprises can't abandon SCCM overnight — they've invested years in it, it manages complex OS deployments, software distribution, and hardware inventory at scale. TOPIC 86
What Is Co-Management?
The concept, value, and why it's the most common enterprise Intune scenario
The Enterprise Reality

Walk into most large enterprise IT departments and you'll find SCCM (now called Microsoft Endpoint Configuration Manager — MECM) running the show. It's been deployed for years, or even decades. It manages software deployment, OS imaging, hardware inventory, patch management, and application lifecycle. The IT team knows it deeply. Ripping it out overnight to replace with Intune isn't realistic.

Co-Management solves this by letting a Windows device be simultaneously managed by both SCCM and Intune. The same device checks in with SCCM for traditional management tasks AND with Intune for cloud management. Over time, you can move ("switch") individual workloads from SCCM to Intune — one at a time, at your own pace, with easy rollback if needed.

Why Co-Management Matters
🌐
Remote Work Enablement
SCCM requires VPN or CMG for remote devices. Intune works everywhere over the internet. Co-management lets remote employees get Intune-managed while office devices still leverage SCCM for bandwidth-intensive software deployment.
🔐
Conditional Access for Legacy Devices
SCCM-only devices can't participate in Azure AD Conditional Access (they have no Azure AD identity). Co-management + Hybrid Azure AD Join gives these devices an Azure AD identity — enabling CA policies and compliance evaluation.
📈
Gradual Cloud Migration
Risk-Free Cloud Migration
Move workloads one at a time. Test Intune compliance policies before SCCM's, move Windows Update management to Intune while keeping software deployment in SCCM. Rollback any workload instantly. No big-bang migration risk.
📊
Unified Reporting
Co-managed device health visible in both SCCM console AND Intune portal. Compliance visible in Azure AD for Conditional Access. Best of both worlds for reporting and monitoring.
The Migration Journey

Co-management isn't the destination — it's the bridge. The typical enterprise journey looks like this:

Stage
1
🏗️ SCCM Only
Starting point for most enterprises. All Windows devices managed exclusively by SCCM on-premises. No Azure AD identity. No Intune. Traditional management model.
Stage
2
🔗 Co-Management Enabled
Devices are Hybrid Azure AD Joined. Both SCCM and Intune manage devices. Workloads initially stay in SCCM. Intune used for Conditional Access compliance and quick wins (Defender policy, compliance policies).
Stage
3
🔄 Gradual Workload Migration
Over months/years, workloads shift to Intune — compliance policies, Windows Update, endpoint security, resource access policies. SCCM still handles OS deployment, software distribution, hardware inventory.
Stage
4
☁️ Intune-Primary (Target)
Most workloads in Intune. SCCM retained only for complex scenarios it does uniquely well (OSD, BIOS management, certain app deployments). New devices provisioned via Autopilot directly into Intune-only management.
💡 SCCM Is Now Called MECM

System Center Configuration Manager (SCCM) was rebranded to Microsoft Endpoint Configuration Manager (MECM) in 2019 as part of the "Microsoft Endpoint Manager" umbrella. In conversations and interviews, both names are used interchangeably. Microsoft documentation uses MECM; most real-world IT shops still say "SCCM." Know both names — they mean the same product.

🎤 Interview Questions — Topic 86
What is Co-Management in Intune and why is it important for enterprises?
Co-Management allows a Windows device to be simultaneously managed by both SCCM (on-premises) and Microsoft Intune (cloud). It's important for enterprises because it enables a risk-free, gradual cloud migration — you can move individual management workloads from SCCM to Intune one at a time without a big-bang migration. It also unlocks cloud capabilities like Conditional Access and remote device management for existing on-premises devices without requiring a complete re-enrollment or infrastructure change.
What is a workload in the context of Co-Management?
A workload is a specific management function — like compliance policies, Windows Update management, or endpoint security configuration. In Co-Management, each workload can be independently assigned to either SCCM or Intune. Moving a workload to Intune means Intune takes over responsibility for that function while SCCM continues managing all other workloads it still owns. This granular control is what makes Co-Management a safe migration path.

Checking Readiness — The SCCM Co-Management Dashboard

SCCM has a built-in Co-management Dashboard (SCCM Console → Monitoring → Co-management) that shows:

  • How many devices are co-management eligible (Hybrid AADJ + correct Windows version)
  • How many are currently co-managed
  • Current workload assignments per workload
  • Client health status for co-managed devices
⚠️ Hybrid Azure AD Join — The Most Common Blocker

The single most common blocker for co-management enablement is Hybrid Azure AD Join not working correctly. Devices may show as domain-joined in on-premises AD but NOT appear in Azure AD. Common causes: Azure AD Connect not syncing device objects (check device sync settings in Azure AD Connect), stale computer accounts in on-premises AD, time sync issues (devices more than 5 minutes out of sync will fail HAADJ), or proxy blocking the device's registration requests to Azure AD.

Verify HAADJ: On any domain-joined PC, run dsregcmd /status — look for AzureAdJoined: YES AND DomainJoined: YES. Both must be YES for co-management to work.

🎤 Interview Questions — Topic 87
What is the minimum SCCM/MECM version required for Co-Management and why?
MECM version 1710 introduced Co-Management support. However, version 2006 or later is strongly recommended as it includes significant improvements to co-management health reporting, the Co-management Dashboard, additional workload options, and better stability. Running an older version risks hitting bugs that are fixed in current branch, and Microsoft only supports current branch releases within a certain time window.
What is Hybrid Azure AD Join and why is it required for Co-Management?
Hybrid Azure AD Join is the state where a device is joined to both on-premises Active Directory AND registered in Azure AD simultaneously. It's required for Co-Management because Intune needs an Azure AD identity to manage the device — without it, Intune has no way to identify or communicate with the device. Azure AD Connect syncs the device's computer account from on-prem AD to Azure AD, and the device runs an auto-registration task that completes the hybrid join. Co-Management uses this Azure AD identity to enroll the device in Intune MDM alongside SCCM.
TOPIC 88
Enabling Co-Management — Step by Step
The exact configuration process from SCCM to co-managed devices
The Two Enrollment Paths
🤖
Path 1: Auto-Enrollment via SCCM
SCCM triggers Intune MDM enrollment on domain-joined devices automatically. No user interaction. Best for bulk enrollment of existing SCCM-managed fleet. SCCM deploys an enrollment task that silently registers devices in Intune.
🔗
Path 2: Azure AD Join (New Devices)
New devices are Azure AD Joined (pure cloud) and enrolled in Intune via Autopilot. SCCM manages them via CMG. Used for new device provisioning strategy — gradually moving away from hybrid join as old devices age out.
Path 1 — Enabling Co-Management from SCCM
  1. 1
    Verify prerequisites: Confirm HAADJ is working (dsregcmd /status on sample devices). Confirm Intune license assigned. Confirm MDM authority = Intune in Azure AD.
  2. 2
    In SCCM Console: Go to Administration → Cloud Services → Co-management → Click Configure co-management.
  3. 3
    Sign into Azure AD: The wizard prompts you to authenticate with your Azure AD Global Admin or Intune Admin credentials. This links SCCM to your Intune tenant.
  4. 4
    Configure automatic enrollment:
    • Pilot: SCCM enrolls only devices in a specific SCCM Collection into Intune. Start here.
    • All: SCCM enrolls ALL eligible devices into Intune. Use after pilot validation.
  5. 5
    Select the Pilot collection: Choose an SCCM Device Collection containing your pilot devices (e.g., "IT Team Laptops" or "Co-Mgmt Pilot"). These devices will be automatically enrolled in Intune.
  6. 6
    Configure workload assignments: For each workload, choose whether it's managed by SCCM, Intune, or a Pilot collection. Keep all workloads in SCCM initially.
  7. 7
    Complete the wizard. SCCM deploys a policy to pilot collection devices. The SCCM client on each device triggers Windows MDM auto-enrollment into Intune via the device's Azure AD identity.
  8. 8
    Verify enrollment: In Intune portal → Devices → All Devices → filter by "Co-managed" enrollment type. Pilot devices should appear within 15–60 minutes. In SCCM dashboard → Co-management → confirm devices show as co-managed.
What SCCM Does Behind the Scenes

When SCCM triggers co-management enrollment, it uses a special MDM enrollment mechanism built into the Windows 10/11 MDM client. SCCM sends a client policy instructing Windows to enroll in Intune using the device's existing Hybrid Azure AD Join token — no user credentials needed, completely silent.

# Registry key SCCM sets to trigger MDM enrollment: HKLM\SOFTWARE\Microsoft\CCM\CoManagement CoManagementFlags = 1 (co-management enabled) AutoEnrollEnabled = 1 (auto-enroll in Intune) # Verify co-management status on a device: # Open SCCM client (Control Panel → Configuration Manager) # Or run PowerShell: (Get-WmiObject -Namespace root\ccm -Class SMS_Client).CoManagementState # Returns: 1 = co-managed, 0 = SCCM only, 97 = transitioning # Verify from Intune portal: # Device → Properties → Enrollment type: Co-managed
The Cloud Management Gateway (CMG)

Without CMG, SCCM can only manage devices connected to the corporate network (or VPN). Cloud Management Gateway is a cloud-based service that extends SCCM management to internet-connected devices — acting as an HTTPS proxy between internet clients and the SCCM on-premises infrastructure.

  • Required for: Co-managing devices that are often off the corporate network (remote workers, laptop users)
  • CMG runs in Azure: It's an Azure VM deployed from the SCCM console. SCCM clients use it to communicate with SCCM over the internet.
  • Cost: Azure VM + outbound data transfer costs. Size the CMG VM appropriately for your client count.
  • Alternative: Without CMG, co-managed remote devices only check in with SCCM when on VPN — Intune handles them the rest of the time.
🎤 Interview Questions — Topic 88
How does SCCM enroll existing domain-joined devices into Intune for Co-Management?
SCCM uses the Windows MDM auto-enrollment mechanism triggered by the SCCM client. When the co-management policy reaches a device, SCCM sets registry keys that instruct Windows to enroll in Intune using the device's Hybrid Azure AD Join token — completely silently, without user interaction. The enrollment uses the device identity in Azure AD (established by HAADJ) to authenticate with Intune. The device then appears in both SCCM (as always) and Intune (new) simultaneously.
What is Cloud Management Gateway (CMG) and why is it needed for Co-Management?
CMG is a cloud-hosted service (Azure VM deployed from the SCCM console) that allows internet-based SCCM clients to communicate with the on-premises SCCM hierarchy without VPN. Without CMG, SCCM can only manage devices on the corporate network or connected via VPN — remote workers are effectively SCCM-dark. For co-management with a remote workforce, CMG ensures SCCM continues managing its workloads for off-network devices while Intune manages its workloads from the cloud. Without CMG, remote device management falls entirely to Intune.

Workload Switching — Which Workloads to Move to Intune
The most important co-management decision — what moves first and why
The Eight Co-Management Workloads

Co-Management has 8 configurable workloads. Each can be independently set to SCCM, Intune, or a Pilot collection (a subset of devices). This granular control is what makes co-management safe — you don't have to move everything at once.

WorkloadWhat It ControlsSCCM StrengthIntune AdvantageWhen to Switch
Compliance Policies Device compliance evaluation (BitLocker, OS version, AV etc.) SCCM compliance settings (DCM) Required for Conditional Access integration — only Intune compliance feeds Azure AD CA Move First — Most impactful. CA requires Intune compliance. Low risk to switch.
Device Configuration Configuration profiles (Wi-Fi, VPN, certificates, device restrictions) SCCM Configuration Items and Baselines Cloud delivery, mobile device support, Settings Catalog Move Second — After compliance. Test thoroughly — many orgs have complex config items.
Endpoint Protection Defender AV, Firewall, BitLocker, ASR, Windows Defender ATP SCCM Endpoint Protection policies MDE integration, Tamper Protection, modern security features, Endpoint Security node Move Early — Intune Endpoint Security is superior. Unlocks MDE integration.
Resource Access Policies Wi-Fi, VPN, Email, Certificate profiles SCCM SCEP certificates, VPN profiles Cloud PKI, modern certificate management, per-app VPN Move When Ready — Requires validating certificates work via Intune before cutting over.
Client Apps Application deployment to Windows devices SCCM is superior — complex deployment types, content delivery, BranchCache, peer cache Win32 apps, Store apps, simpler deployment Move Later — SCCM app deployment is powerful. Only move if migrating fully to cloud.
Office Click-to-Run Apps Microsoft 365 Apps (Office) updates and version management SCCM manages M365 Apps servicing via ODT Intune M365 Apps deployment type, Autopatch for updates Move When M365 Ready — Switch if moving to Autopatch for M365 Apps updates.
Windows Update Policies Windows Update management (quality + feature updates, deferral) SCCM Software Update Management, WSUS integration Windows Update for Business, Windows Autopatch, no WSUS needed Move When Infrastructure Ready — Requires devices to reach Windows Update directly (no WSUS dependency).
Endpoint Analytics Device health, boot performance, reliability scoring SCCM reporting and hardware inventory Intune Endpoint Analytics provides cloud-native insights, proactive recommendations Move Early — Low risk. Intune analytics are more modern. No user impact.
The Recommended Workload Migration Order
🥇
Phase 1 — Quick Wins (Low Risk)
Compliance Policies — Essential for Conditional Access. Move this first — critical business value, low complexity.
Endpoint Protection — Intune's security stack is superior. Unlocks MDE integration and Tamper Protection.
Endpoint Analytics — Zero user impact. Better insights immediately.
🥈
Phase 2 — Configuration (Test Carefully)
Device Configuration — Migrate config items to Intune profiles. Validate each policy in pilot before broad rollout.
Resource Access — Certificate, Wi-Fi, VPN. Test thoroughly — certificate misconfiguration can break Wi-Fi for entire fleet.
🥉
Phase 3 — Updates (Infrastructure Prep)
Windows Update Policies — Remove WSUS dependency first. Test WUfB or Autopatch in pilot.
Office Click-to-Run — Move when ready to align with Autopatch strategy.
🏁
Phase 4 — Apps (Last or Never)
Client Apps — Many organizations keep this in SCCM permanently. SCCM app deployment is mature, handles complex scenarios, has BranchCache for bandwidth management. Only move if going fully cloud-native.
Switching a Workload — The Mechanics
  1. 1
    In SCCM Console: Administration → Cloud Services → Co-management → Properties → Workloads tab
  2. 2
    Find the workload to switch (e.g., Compliance Policies). Slider has three positions: Configuration Manager | Pilot Intune | Intune
  3. 3
    Move the slider to Pilot Intune → Select a small pilot SCCM Collection (e.g., IT team). Only devices in that collection switch to Intune for this workload.
  4. 4
    Validate over 2–4 weeks. Verify Intune policies are applying correctly. Test that no functionality was lost. Confirm compliance status in Azure AD.
  5. 5
    When confident: move slider to Intune — ALL co-managed devices now use Intune for this workload. SCCM stops managing it.
🚨 Never Move the Same Workload to Both

When a workload is set to Intune, SCCM automatically stops managing that workload on co-managed devices. Do NOT configure the same settings in both SCCM and Intune simultaneously — you'll get conflicts and unpredictable behavior. Before switching a workload to Intune, ensure you have equivalent Intune policies ready and validated. The moment you switch the slider, SCCM withdraws its policies for that workload on targeted devices.

🔄
Real-World Workload Migration Example

Company: 3,000-device enterprise bank. SCCM-managed for 8 years. Co-management enabled 6 months ago with all workloads in SCCM. Now ready to start moving workloads.
Month 1: Move Compliance Policies to Intune (Pilot: 50 IT devices first, then all). Immediately unlocks Conditional Access — devices must be Intune-compliant to access Exchange Online. BitLocker requirement added. 200 devices fail compliance (BitLocker off) — automatically blocked from Exchange until fixed. Security team is happy.
Month 2: Move Endpoint Protection to Intune. Deploy MDE via Intune EDR policy. Tamper Protection enabled. SCCM Endpoint Protection policies withdrawn automatically.
Month 3–4: Move Device Configuration. Test Wi-Fi and VPN certificates via Intune SCEP. Pilot validation takes 3 weeks (complex certificate chain). Broad rollout in month 4.
Month 6–12: Windows Update to Intune/Autopatch. Client Apps stay in SCCM — too complex to migrate, SCCM's app model is better for their environment.
Outcome: Hybrid model. Intune manages security and compliance. SCCM manages apps and OS deployment. Best of both worlds.

🎤 Interview Questions — Topic 89
Which Co-Management workload should typically be moved to Intune first and why?
Compliance Policies. This is typically the first workload to move because it has the highest immediate business value (Conditional Access requires Intune compliance to work — SCCM compliance doesn't feed Azure AD), the lowest risk (compliance policies evaluate rather than change device settings, so if something is misconfigured, devices show non-compliant but nothing breaks functionally), and it unlocks security controls that depend on Azure AD knowing device health status.
Why do many organizations keep the Client Apps workload in SCCM even after heavily using Intune?
SCCM's application deployment capabilities are mature and powerful — it handles complex deployment types, bandwidth management via BranchCache and peer cache, complex dependency trees, task sequences, and application supersedence at a scale and complexity that Intune Win32 apps can't always match. SCCM can deploy multi-gigabyte software packages efficiently across slow WAN links. Many organizations find that SCCM app deployment works so well that there's no compelling reason to migrate it, especially since co-management allows keeping this workload in SCCM indefinitely while moving all other workloads to Intune.
TOPIC 90
Pilot Collections & Gradual Migration
How to de-risk co-management rollout using SCCM collections as a safety net
Why Pilot Collections Are Essential

Every workload migration carries risk. Even well-tested Intune policies can behave differently on a specific hardware model, driver version, or line-of-business app combination. Pilot collections are the risk mitigation mechanism — you test each workload on a small, representative set of devices before rolling out to thousands.

Building Your Pilot Collection Strategy
🧪
Phase 0 — Smoke Test (5–10 devices)
IT team's own laptops. The smallest, safest starting point. If something breaks, IT handles it themselves without impacting end users. Validates basic co-management enrollment and Intune communication are working.
🔬
Phase 1 — IT Pilot (50–100 devices)
Wider IT department + tech-savvy volunteers. Diverse hardware and software coverage. People who can self-troubleshoot. Run each new workload at this stage for 2–4 weeks before expanding.
👥
Phase 2 — Business Pilot (5–10% of fleet)
Representative sample of the real user population. Different departments, hardware models, job functions. Covers more edge cases. Run for 2–4 weeks. Confirms no business-critical app or workflow is affected.
🌐
Phase 3 — Broad Rollout (All Devices)
Full fleet switch. Only after Phase 2 validates no issues. Have a rollback plan ready (sliding the workload back to SCCM is a one-click operation).
Creating SCCM Collections for Pilot Phases
  1. 1
    In SCCM Console: Assets and Compliance → Device Collections → Create Device Collection
  2. 2
    Name: "Co-Mgmt Pilot Phase 1 — IT Team" → Limiting Collection: All Windows 10/11 Clients
  3. 3
    Add membership rule: Direct rule → Manually add specific device names (IT team laptops). Or Query rule → e.g., "SELECT * FROM SMS_R_System WHERE SMS_R_System.Name LIKE 'IT-%'"
  4. 4
    For Phase 2: Create a new collection with a percentage-based or department-based query rule, or use an existing SCCM collection (e.g., "Finance Department Laptops")
  5. 5
    In Co-management wizard → Workloads tab → Set each workload to Pilot Intune → Select the Phase 1 collection as the pilot collection
The Pilot Validation Checklist

For each workload switch, validate these before expanding to broad rollout:

WorkloadValidation Points
Compliance PoliciesPilot devices appear in Intune compliance reports. Azure AD shows device as compliant. Conditional Access works — can access Exchange while compliant, blocked when non-compliant.
Endpoint ProtectionDefender AV policies applied (check Windows Security app settings). Tamper Protection active. MDE devices appear in security.microsoft.com portal. No conflict with previous SCCM AV policy.
Device ConfigurationAll migrated config items have Intune equivalent deployed. Test Wi-Fi, VPN, certificates on pilot devices. Verify no settings reverted to defaults after SCCM config items were removed.
Windows Update PoliciesPilot devices getting updates from Windows Update (not WSUS). Check Windows Update history. WUfB deferral settings applied. Devices not stuck on old updates.
Client AppsRequired apps still installed after switch. New app deployments from Intune installing correctly. No apps broken by SCCM management withdrawal. Check IME log on pilot devices.
Rollback — The Safety Net

If a workload switch causes issues in the pilot, rollback is a single slider move:

  • In SCCM Console: Co-management Properties → Workloads → Move slider back to "Configuration Manager"
  • What happens: SCCM immediately re-asserts management of that workload. SCCM policies for that workload are re-deployed to devices within the next client policy cycle (default 60 minutes).
  • Intune policies: The Intune policies for that workload don't automatically delete — you need to manually remove or unassign the Intune policies if you're rolling back permanently.
💡
Communication is Critical: Before each pilot phase expansion, inform the pilot group's users and their managers. Explain what's changing, what they might notice (if anything), and who to contact if issues arise. Even if the change is invisible to end users (compliance policies, AV settings), a heads-up email builds trust and ensures you have a feedback channel if something does go wrong that IT hasn't noticed yet.
🎤 Interview Questions — Topic 90
Why use pilot collections in Co-Management instead of switching all devices at once?
Pilot collections allow testing each workload on a small, controlled device set before rolling out to the entire fleet. Even thoroughly tested Intune policies can interact unexpectedly with specific hardware models, driver versions, or line-of-business applications in the real environment. By validating on 50 IT team devices first, then 5% of the production fleet, you catch issues affecting only a small number of users rather than discovering a critical bug on 5,000 devices simultaneously. Rollback from a pilot issue is simple; rollback from a broken fleet-wide switch is complex.
If a Co-Management workload switch causes problems in the pilot, how do you roll back?
In the SCCM console, go to Administration → Cloud Services → Co-management → Properties → Workloads tab. Move the affected workload's slider back to "Configuration Manager." SCCM re-asserts its policies for that workload and re-deploys them to devices within the next client policy interval (default 60 minutes). Note that the Intune policies for that workload remain in Intune — you need to manually remove their assignments from co-managed device groups to prevent them from conflicting while SCCM has regained management authority.
TOPIC 91
Troubleshooting Co-Management Issues
Diagnosing the most common co-management failures and how to fix them
The Troubleshooting Mindset for Co-Management

Co-management failures are complex because the issue could be in any of four layers: SCCM, Azure AD/HAADJ, Intune enrollment, or workload configuration. Always identify which layer is failing before jumping to solutions. The diagnostic tools span all four layers.

Common Co-Management Issues & Fixes
❌ Device Not Showing as Co-Managed in Intune or SCCM Dashboard
Possible Causes: HAADJ not working (device not in Azure AD), SCCM co-management policy not reaching the device, Intune license not assigned, MDM auto-enrollment blocked by Azure AD MDM scope settings.
Diagnostic steps:
1. Run dsregcmd /status on the device → AzureAdJoined must be YES AND DomainJoined must be YES
2. If AzureAdJoined is NO → HAADJ is broken. Check Azure AD Connect device sync. Check AADTroubleShooter on the device.
3. Check SCCM client logs: C:\Windows\CCM\Logs\CoManagementHandler.log — shows co-management enrollment attempts and errors
4. Verify user has Intune license assigned in Azure AD
5. Verify MDM User Scope in Azure AD Mobility settings includes the device's user
❌ Co-Managed Device Shows Wrong Workload Authority
Problem: Compliance workload was moved to Intune but device still shows SCCM compliance policies applying. Or vice versa — policy from the "wrong" manager applying to a device.
Fix:
1. Verify the co-management workload slider setting in SCCM — check which collection the device belongs to and confirm it's assigned to the expected pilot or full workload setting
2. Trigger a SCCM client policy refresh: Right-click device in SCCM → Run Script → "Trigger Machine Policy Retrieval" or run Invoke-WmiMethod -Namespace root\ccm -Class SMS_Client -Name TriggerSchedule -ArgumentList "{00000000-0000-0000-0000-000000000021}"
3. In Intune, trigger device sync. Check the device's configuration profiles — workload-related profiles should be applying
4. Check CoManagementHandler.log on the device for workload switching events
❌ Hybrid Azure AD Join Failing (HAADJ Broken)
Most common co-management blocker. Device has AzureAdJoined: NO in dsregcmd. Computer accounts in Azure AD are missing or stale.
Fix process:
1. Check Azure AD Connect sync: in Azure AD Connect health or Azure AD portal → Devices — is this device's computer account synced?
2. Enable device sync in Azure AD Connect if not configured: Azure AD Connect → Device options → Configure Hybrid Azure AD join
3. Check C:\Windows\System32\winevt\Logs\Microsoft-Windows-AAD%4Operational.evtx for HAADJ registration errors
4. Run AAD Troubleshooter: dsregcmd /debug /status for verbose output
5. Verify device time sync — time skew >5 minutes from domain controller will fail HAADJ
6. Check proxy — if corporate proxy blocks *.microsoftonline.com, HAADJ registration will fail silently
❌ Intune Policies Not Applying After Workload Switch
Problem: Moved compliance workload to Intune. Device is co-managed. But Intune compliance policy shows "Not applicable" or "Pending" on the device.
Fix:
1. Verify the device appears in Intune → All Devices → Enrolled as "Co-managed"
2. Check if the device is in the Azure AD group that the Intune policy is assigned to
3. Check for dynamic group membership delay (up to 24 hours for dynamic groups)
4. Trigger a manual Intune sync: Settings → Access work or school → Info → Sync
5. Check MDMDiagReport on the device: run MDMDiagnosticsTool.exe -out C:\Temp\MDM and review the HTML report for policy delivery errors
6. Verify no conflicting SCCM policies still assigned to this device for the same workload
❌ Device Shows "Not Ready" in Co-Management Dashboard
Problem: SCCM shows the device as eligible for co-management but it's not enrolled. Common causes: SCCM client not healthy, auto-enrollment blocked, Azure AD token issue.
Fix:
1. Check SCCM client health: Monitoring → Client Status → Verify device is active and healthy
2. Check CoManagementHandler.log for specific error messages
3. Manually trigger enrollment: On the device, run SCCM client repair or force a policy evaluation
4. Check Event Viewer on device: Applications and Services → Microsoft → Windows → DeviceManagement-Enterprise-Diagnostics-Provider → look for enrollment errors
5. Run MDMDiagnosticsTool.exe to capture full enrollment state
6. Verify the device's Azure AD device object is not disabled or stale — check in Azure AD portal

Real Co-Management Troubleshooting Case

Ticket: "We enabled co-management 2 weeks ago. 800 of our 1,200 devices are co-managed. But 400 devices still show as 'Not Ready' in the SCCM co-management dashboard. They're all Windows 10 devices. The 800 that worked are Windows 11."
Investigation:
Step 1: Run dsregcmd /status on one of the failing Windows 10 devices → AzureAdJoined: NO (HAADJ failing)
Step 2: Check Azure AD portal → Devices → filter by OS = Windows 10 → Most Windows 10 computer accounts have "Registered" date = None (never synced to Azure AD)
Step 3: Check Azure AD Connect configuration → Device sync settings → "Hybrid Azure AD join" was configured only for Windows 11 domain suffix, not for the older Windows 10 OU
Step 4: Fix — open Azure AD Connect → Device options → Add the Windows 10 devices' OU to the sync scope → Run delta sync
Step 5: Wait 24 hours for HAADJ auto-registration to run on Windows 10 devices. Run dsregcmd /status again → AzureAdJoined: YES
Result: Within 48 hours, 380 of 400 remaining devices co-enrolled successfully. The last 20 had clock skew issues — fixed by re-syncing time with DC.

🎤 Interview Questions — Topic 91
A device shows "Not Ready" for Co-Management. What is your troubleshooting process?
Start with the foundational check: run dsregcmd /status on the device. If AzureAdJoined is NO, Hybrid Azure AD Join is failing — troubleshoot HAADJ (check Azure AD Connect device sync, time sync, proxy blocking registration endpoints). If HAADJ is working but device still shows Not Ready, open CoManagementHandler.log in C:\Windows\CCM\Logs to see what SCCM is doing and any errors. Verify the device is in the SCCM collection targeted for co-management enrollment. Check that the user has an Intune license. Trigger a SCCM policy refresh and check MDMEnrollment.log for the enrollment attempt result.
What is the CoManagementHandler.log and when do you use it?
CoManagementHandler.log at C:\Windows\CCM\Logs\ is the primary SCCM log for co-management operations. It records every co-management enrollment attempt, the workload switching events, enrollment success or failure with error codes, and the Azure AD token acquisition used during MDM enrollment. I use it whenever a device isn't enrolling into co-management as expected — it shows exactly what step failed and often includes the error code and diagnostic information needed to identify the root cause, whether it's an Azure AD authentication issue, a network connectivity problem, or an SCCM client issue.
After switching the Compliance workload to Intune, devices show compliant in SCCM but non-compliant in Intune. What's happening?
This means the workload switch was successful — Intune is now evaluating compliance, not SCCM. But the Intune compliance policy is stricter or different from the SCCM compliance settings, so devices that met SCCM requirements may not meet Intune requirements. Common causes: Intune compliance policy requires BitLocker (SCCM didn't), or the Intune minimum OS version is set higher. This is expected and correct — the goal is to get devices to meet the Intune compliance requirements. Investigate which specific settings are failing in Intune's compliance report (Device → Device compliance → per-setting results) and remediate accordingly.
