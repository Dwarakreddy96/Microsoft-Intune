TOPIC 66
Endpoint Security Node — Overview of All Blades
Your security command center inside Intune
What Is the Endpoint Security Node?

The Endpoint Security node in Intune is a dedicated security management area — separate from Configuration Profiles — specifically built for hardening devices against cyber threats. It gives security teams a focused workspace with pre-tuned security policies, deep Microsoft Defender for Endpoint integration, and a unified view of the security posture across your managed fleet.

Navigate to it via: Intune portal → Endpoint Security

The Six Security Blades
🦠 Antivirus
Configure Microsoft Defender Antivirus settings — real-time protection, cloud protection, scan schedules, exclusions, PUA protection. The AV engine configuration blade.
🔒 Disk Encryption
Configure BitLocker (Windows) and FileVault (macOS). Silent enablement, TPM settings, startup PIN, recovery key escrow to Azure AD. Your encryption management center.
🔥 Firewall
Windows Defender Firewall configuration per network profile (Domain, Private, Public). Enable/disable, default inbound/outbound behavior, firewall rules for specific apps and ports.
🛡️ Endpoint Detection & Response
Onboard Windows, macOS, iOS, Android devices to Microsoft Defender for Endpoint. Configure EDR telemetry. The gateway to MDE's detection and investigation capabilities.
⚔️ Attack Surface Reduction
ASR rules block common attack techniques used by malware and ransomware. Also contains exploit protection, application control (WDAC), device control (USB block), and web protection.
👤 Account Protection
Windows Hello for Business, LAPS (Local Administrator Password Solution), Credential Guard. Protects credentials and local accounts from theft and lateral movement attacks.
Security Overview Dashboard

At the top of Endpoint Security is the Security overview dashboard — a single-pane view of your organization's security posture. It shows:

  • Unhealthy endpoints count — devices with active security issues
  • Security task status — remediation items from MDE threat & vulnerability management
  • Microsoft Secure Score — your overall security score vs industry benchmark
  • Defender for Endpoint onboarding status — how many devices are reporting to MDE
  • Quick access to all six blades — one click to any security policy area
Security Tasks — MDE to Intune Workflow

One of the most powerful integrations: when Microsoft Defender for Endpoint discovers a vulnerability or misconfiguration on managed devices (via its Threat & Vulnerability Management engine), it can automatically push a Security Task to Intune. The IT admin sees the task in Intune's Endpoint Security node — clicks Accept — and Intune deploys the remediation policy automatically. The vulnerability is patched without ever leaving the Intune portal.

The Endpoint Security node is a dedicated security-focused management area in Intune with six specialized blades: Antivirus, Disk Encryption, Firewall, EDR, ASR, and Account Protection. It differs from Configuration Profiles in depth and focus — Endpoint Security provides more granular, security-specific settings with deep MDE integration, Security Baselines, and a unified security posture dashboard. Microsoft Defender Antivirus Policy via Intune
Configuring AV protection across your Windows fleet from the cloud
Why Configure Defender via Intune?

Microsoft Defender Antivirus is built into every Windows 10/11 device — but its default settings aren't always optimal for enterprise security. Via Intune, you can enforce specific AV configurations across thousands of devices, prevent users from disabling protection, configure cloud-delivered protection strength, manage exclusions centrally, and ensure consistent AV behavior fleet-wide.

Defender AV Exclusions — Handle with Extreme Caution

Exclusions tell Defender not to scan certain files, folders, extensions, or processes. They're necessary for some legitimate business scenarios (antivirus scanning databases can corrupt them; certain backup software conflicts with AV scanning) but are also the #1 way attackers bypass AV protection.

  • Only add exclusions that are verified necessary by the app vendor's documentation.
  • Use process exclusions instead of folder exclusions when possible — scope is narrower.
  • Never exclude entire system paths like C:\Windows\System32 — this disables protection for the most critical area.
  • Audit exclusions quarterly — remove any that are no longer needed.
  • MDE alert: Defender for Endpoint will generate alerts if exclusions are overly broad — use this as a guide.
🚨 Tamper Protection — Critical Setting

Tamper Protection prevents local users (even local admins) and malware from modifying Defender AV settings through local registry edits, PowerShell, or WMI. Without it, ransomware's first step is often disabling AV before launching the attack. With Tamper Protection enabled via Intune + MDE, any attempt to modify Defender settings is blocked and logged. This is one of the highest-value single security settings you can enable.

🦠
Real-World Defender AV Policy Incident

Incident: A manufacturing company had Defender AV deployed but not configured via Intune. Employees disabled real-time protection "because it slowed their PCs." A ransomware infection encrypted 200 workstations overnight — the attacker's first step had been disabling AV.
Remediation: Deploy Intune Endpoint Security AV policy → Enable Tamper Protection → Require real-time protection → Enable cloud protection. Also create a Compliance Policy requiring Defender AV to be running — non-compliant devices blocked from network resources via Conditional Access.
Result: AV can no longer be disabled by users or malware. Any attempt generates an alert in MDE. The company hasn't had an AV-bypass incident since.

🎤 Interview Questions — Topic 67
What is Tamper Protection in Defender AV and why is it important?
Tamper Protection prevents local users, local admins, and malware from modifying Microsoft Defender Antivirus settings locally — through registry edits, PowerShell commands, WMI calls, or Group Policy. Without it, ransomware routinely disables AV as its first step before launching the main attack. With Tamper Protection enabled via Intune and MDE, any modification attempt is blocked at the OS level and generates an alert in Defender for Endpoint. It's one of the most impactful single security settings available.
Why should AV exclusions be kept to a minimum and reviewed regularly?
AV exclusions are a major attack vector. Attackers who gain initial access to a system will often place malware in folders or under processes that are excluded from scanning — knowing AV won't inspect those locations. Broad exclusions like entire system directories effectively disable protection for large swaths of the system. Exclusions should only be added based on verified vendor documentation, should be as narrow as possible (process-level rather than folder-level), and should be audited quarterly to remove any that are no longer needed.

What is the difference between Microsoft Defender Antivirus and Microsoft Defender for Endpoint?
Defender AV is the built-in antivirus that prevents known threats using signatures, heuristics, and cloud lookup — it's free with Windows 10/11. TOPIC 69
Attack Surface Reduction (ASR) Rules
Blocking the techniques attackers use — before malware even runs
What Are ASR Rules?

Attack Surface Reduction rules are security policies that block specific behaviors commonly used in malware and ransomware attacks — regardless of whether the attacking file is recognized as malware. They work at the behavior level: instead of asking "is this file malicious?", ASR asks "is this action something legitimate software does?"

For example: legitimate Word documents don't create child processes. If a Word document tries to spawn cmd.exe or PowerShell, that's a classic macro-based attack technique. The ASR rule "Block Office applications from creating child processes" stops this — even for a brand new zero-day exploit with no signature.

The Three ASR Modes
👁️
Audit Mode
The rule LOGS what would have been blocked without actually blocking anything. Safe to enable immediately — zero production impact. Review logs for 2-4 weeks to identify false positives before switching to Block. Start here ALWAYS.
🚫
Block Mode
The rule actively BLOCKS the behavior and logs it. A notification pops up to the user. The blocked action is recorded in MDE for investigation. Switch to this after validating no false positives in Audit mode.
⚠️
Warn Mode
Shows a warning to the user that the action was blocked, but allows them to bypass it with a justification. A middle ground — blocks by default but allows exceptions. Good for rules that occasionally have legitimate uses.
The Essential ASR Rules — Know These Cold
Block Office apps from creating child processes
Prevents Word, Excel, PowerPoint from launching other processes (cmd.exe, PowerShell, wscript.exe). Blocks macro-based malware delivery — one of the most common attack vectors.
Blocks: Macro malware, phishing attachments with embedded code
Block credential stealing from LSASS
Prevents tools like Mimikatz from reading credential hashes from the LSASS process memory. Without this, any local admin can dump all credentials from memory instantly.
Blocks: Mimikatz, credential dumping, lateral movement
Block untrusted/unsigned processes from USB
Prevents unsigned executables from running directly off USB drives. Blocks USB-delivered malware — a common insider threat and physical access attack vector.
Blocks: USB-borne malware, BadUSB attacks, physical access attacks
Block executable content in email/webmail
Prevents .exe, .dll, .js, .vbs files delivered via email from executing. Kills the initial access phase of phishing attacks delivered as email attachments.
Blocks: Phishing attachments, malicious email links, drive-by downloads
Block Office apps from injecting code into other processes
Prevents Office apps from injecting malicious code into other running processes (process injection). Blocks a common technique for escalating privileges and evading detection.
Blocks: Process injection attacks, DLL injection via Office
Block JavaScript/VBScript from launching executables
Prevents script interpreters (wscript.exe, cscript.exe) from launching downloaded executables. Blocks script-based malware delivery — extremely common in enterprise attacks.
Blocks: Script-based malware loaders, ransomware delivery scripts
Block persistence through WMI event subscription
Prevents malware from using WMI (Windows Management Instrumentation) to create persistent event subscriptions that survive reboots. A stealthy persistence mechanism used by APTs.
Blocks: APT persistence, fileless malware, stealthy backdoors
Block Win32 API calls from Office macros
Prevents VBA macros in Office from making Windows API calls (CreateProcess, WriteProcessMemory). Cripples advanced macro-based malware that goes beyond simple child process creation.
Blocks: Advanced macro malware, VBA-based shellcode execution
Configuring ASR Rules via Intune
  1. 1
    Intune → Endpoint Security → Attack Surface Reduction → Create Policy → Platform: Windows → Profile: Attack Surface Reduction Rules
  2. 2
    Phase 1: Set ALL rules to Audit mode. Assign to All Devices. Deploy.
  3. 3
    Review Audit logs for 2-4 weeks: In MDE portal → Events → Filter by ASR. Check which rules triggered and whether the triggers were legitimate activities or actual attack behaviors. Note any false positives (legitimate apps triggering rules).
  4. 4
    Configure exclusions: For any legitimate apps that trigger ASR rules, add them as exclusions in the ASR policy (specific file path or process exclusion).
  5. 5
    Phase 2: Switch high-confidence rules to Block mode. Start with the highest-impact rules (credential stealing, Office child processes). Keep complex rules in Audit longer.
💡
Never Jump Straight to Block: ASR rules can break legitimate applications if not tested in Audit mode first. For example, the "Block Office from creating child processes" rule can break some email add-ins and PDF printers that use Word to create files. Always Audit first for at least 2 weeks in your real production environment before switching to Block. False positives found in Audit mode can be excluded before enforcement.
🎤 Interview Questions — Topic 69
What are ASR rules and how do they differ from traditional antivirus?
Attack Surface Reduction rules block specific behaviors commonly used in cyberattacks — regardless of whether the file is recognized as malware. Traditional AV matches files against known malware signatures. ASR works at the behavior level — blocking actions like Office spawning child processes or credential dumping from LSASS even for zero-day unknown threats. ASR prevents the techniques attackers use; AV identifies known malicious files. They complement each other — ASR reduces the attack surface, AV catches known threats.
Why must ASR rules be tested in Audit mode before enabling Block mode?
Audit mode logs what would be blocked without actually blocking — allowing you to see false positives before they impact production. Some ASR rules can break legitimate applications — for example, "Block Office from creating child processes" can interfere with email add-ins or PDF printers that use Office APIs. Switching to Block without Audit testing first can break business-critical workflows, causing urgent rollback scenarios. Always Audit for 2-4 weeks in the real production environment, configure exclusions for legitimate triggers, then switch to Block confidently.
Which ASR rule would block Mimikatz credential dumping?
The "Block credential stealing from LSASS process" ASR rule (GUID: 9e6c4e1f-7d60-472f-ba1a-a39ef669e4b0). Mimikatz reads credential hashes from the LSASS process memory by opening it with special access rights. This ASR rule blocks any process from opening LSASS with the access flags that credential dumping tools require — effectively preventing Mimikatz and similar tools from extracting credentials, even when run by a local administrator.
TOPIC 70
Firewall Policies via Intune
Managing Windows Defender Firewall centrally across the enterprise
Why Manage Firewall via Intune?

Windows Defender Firewall is enabled by default — but its default configuration may not match your organization's security requirements. Via Intune, you can ensure the firewall is enabled on all network profiles, configure inbound/outbound defaults, push specific firewall rules for corporate apps, and prevent users from disabling the firewall — all without Group Policy or on-prem infrastructure.

Three Firewall Profiles
🏢
Domain Profile
Active when the device is connected to a corporate domain network. Usually the most permissive — domain controllers and internal resources need to communicate. For Hybrid AADJ devices connected to on-prem network.
🏠
Private Profile
Active when the user manually marks a network as Private (home network). More permissive than Public but less than Domain. Most employees on home Wi-Fi will use this profile.
☕
Public Profile
Active on any network not marked as Private or Domain — coffee shops, airports, hotels. Should be the MOST restrictive profile. Block all inbound connections that aren't explicitly allowed.
Key Firewall Settings to Configure
SettingRecommendedWhy
Firewall enabled (all profiles)✅ EnabledNever let the firewall be disabled — fundamental host-based protection.
Block all inbound connections (Public profile)✅ EnabledOn public Wi-Fi, there should be no unsolicited inbound connections to a corporate laptop.
Default inbound action (Private profile)BlockBlock all inbound unless explicitly allowed by a rule.
Default outbound action (all profiles)AllowMost organizations allow all outbound by default. If restricting outbound, use explicit allow rules.
Allow unicast responses to multicast/broadcastYesRequired for some network discovery and print functionality.
Notification to user when blockedYesHelps users report legitimate apps being blocked.
Deploying Firewall Rules for Specific Apps

Beyond the base firewall profile settings, you can deploy specific rules — "Allow TCP port 8443 inbound for Corporate VPN app." Create these under: Endpoint Security → Firewall → Create Policy → Profile: Windows Firewall Rules

# Example: Allow a specific corporate app through firewall # Rule Name: Allow Corporate App TCP 8080 # Direction: Inbound # Action: Allow # Protocol: TCP # Local Port: 8080 # App: C:\Program Files\CorporateApp\app.exe # Profiles: Domain, Private (not Public) # This deploys the rule silently to all targeted devices # No need for local admin rights to add firewall rules
💡 Firewall Policy vs Group Policy Firewall

On-prem environments historically used Group Policy to manage Windows Firewall. Via Intune, cloud-native and remote devices get the same firewall configuration without needing domain connectivity. For Hybrid AADJ devices, both Group Policy and Intune firewall policies can apply — ensure they don't conflict. Intune Endpoint Security Firewall policies generally win when there's a conflict with Group Policy firewall settings on Intune-enrolled devices.

🎤 Interview Questions — Topic 70
What are the three Windows Firewall profiles and which is the most restrictive?
Domain profile — active on corporate domain networks (most permissive, needed for internal communication). Private profile — active on home or trusted networks (moderately restrictive). Public profile — active on untrusted public networks like coffee shops and airports (most restrictive). The Public profile should block all inbound connections by default — a corporate laptop on a public Wi-Fi network should accept no unsolicited inbound traffic whatsoever.
How do you deploy a specific firewall rule to all managed Windows devices via Intune?
In Intune, Endpoint Security → Firewall → Create Policy → Platform: Windows → Profile: Windows Firewall Rules. Create a rule specifying: name, direction (inbound/outbound), action (allow/block), protocol (TCP/UDP/Any), local port or remote port, and optionally a specific application path. Assign to the target device group. Devices receive the firewall rule silently on next check-in — no local admin rights required, no Group Policy needed.
TOPIC 71
Disk Encryption — BitLocker Policy, Silent Enablement & Recovery Keys
Protecting data at rest on every Windows and macOS device in your fleet
Why BitLocker Is Non-Negotiable

Laptops get stolen. Hard drives get lost. Without disk encryption, anyone who physically obtains a drive can read every file on it — including HR records, source code, client data, financial documents — by simply booting from a USB drive and mounting the unencrypted drive. BitLocker makes stolen data inaccessible without the encryption key. For any organization with a laptop fleet, mandatory BitLocker is table stakes.

BitLocker States — Know These for Troubleshooting
✅ Protection On
BitLocker is fully enabled and protecting the drive. The drive is encrypted and Intune has the recovery key escrowed. This is the target state.
❌ Protection Off
BitLocker is not enabled. Drive is unencrypted. Compliance policy should flag this as Non-Compliant immediately.
⚠️ Suspended
BitLocker is enabled but temporarily suspended — usually during OS updates or BIOS changes. Should auto-resume. If stuck suspended, Intune reports device as non-compliant depending on policy.
🔵 Waiting for Activation
BitLocker is being configured but waiting for user to set a PIN or waiting for device restart. Policy received but activation not yet complete.
Configuring BitLocker — Silent Enablement

Silent enablement is the enterprise gold standard — BitLocker activates automatically in the background without any user prompts, PINs, or interaction. It requires a TPM 2.0 chip and Azure AD Joined or Hybrid Azure AD Joined devices.

  1. 1
    Intune → Endpoint Security → Disk Encryption → Create Policy → Platform: Windows → Profile: BitLocker
  2. 2
    BitLocker base settings:
    Enable full disk encryption: ✅
    Hide BitLocker prompt during Azure AD/Intune join: ✅ (key for silent enablement)
    Allow standard users to enable during Autopilot: ✅
  3. 3
    OS drive settings:
    OS drive encryption: Require
    Compatible TPM startup: Allow TPM (no additional PIN for silent)
    Recovery key: Enable — store in Azure AD automatically
  4. 4
    Recovery options:
    Save BitLocker recovery info to Azure AD: ✅ Required
    Recovery key rotation: Enable — after each use, a new key is generated
  5. 5
    Assign to All Devices → Deploy. On next check-in, TPM-equipped devices silently enable BitLocker and escrow the recovery key to Azure AD. User sees nothing.
Recovery Key Escrow — Critical Configuration

When BitLocker is enabled via Intune, the recovery key is automatically stored in Azure AD. This is critical — when a user forgets their BitLocker PIN or a device prompts for a recovery key (after a BIOS change or failed TPM boot), IT can retrieve the key from:

  • Intune portal: Devices → find device → Recovery keys tab → View BitLocker recovery key
  • Azure AD portal: Azure AD → Devices → find device → BitLocker keys
  • Helpdesk delegation: You can give helpdesk staff permission to view recovery keys without giving them full Intune admin — via Intune RBAC scope and custom roles.
BitLocker Requirements — Checklist
  • TPM 2.0 chip: Required for silent enablement. Verify in Device Manager → Security Devices → TPM 2.0.
  • UEFI boot mode: Required (not Legacy BIOS). Check in msinfo32 → BIOS Mode: UEFI.
  • Secure Boot enabled: Strongly recommended — BitLocker + Secure Boot prevents bootkit attacks.
  • Azure AD or Hybrid AADJ: Required for recovery key escrow to Azure AD.
  • Windows 10/11 Pro or Enterprise: BitLocker not available on Windows Home editions.
🚨 BitLocker Compliance vs Config Profile — Common Confusion

A Compliance Policy with "Require BitLocker" checks IF BitLocker is on — and marks the device Non-Compliant if it's off. But it does NOT turn BitLocker on.
An Endpoint Security BitLocker policy actually ENABLES BitLocker on the device.
You need BOTH: the Endpoint Security policy to turn BitLocker on, and the Compliance policy to verify it stays on and enforce Conditional Access consequences if it's disabled. Common mistake: deploying only the Compliance Policy and wondering why BitLocker isn't enabling — the Compliance Policy can't enable it, only evaluate it.

💻
Real-World BitLocker Deployment

Scenario: 300-device laptop fleet. Company needs 100% BitLocker coverage for GDPR compliance and cyber insurance requirements. Currently at 40% — some users dismissed the BitLocker setup wizard.
Solution:
1. Deploy Endpoint Security BitLocker policy → Silent enablement → Assign to All Devices
2. Deploy Compliance Policy → Require BitLocker → Actions: Day 1 mark non-compliant → Conditional Access blocks SharePoint for non-compliant devices
3. Monitor: Within 3 days, Intune silently enables BitLocker on all remaining 180 devices during next check-in. Recovery keys escrowed to Azure AD automatically.
4. Compliance dashboard shows 98% compliant (2 devices had hardware issues — legacy without TPM).
Result: GDPR BitLocker requirement met. Cyber insurance audit passed. Zero user calls about BitLocker setup.

🎤 Interview Questions — Topic 71
What is BitLocker silent enablement and what are its requirements?
Silent enablement means BitLocker activates automatically in the background without any user prompts or PIN setup. Requirements: TPM 2.0 chip (verifies the boot environment), UEFI boot mode (not Legacy BIOS), the device must be Azure AD Joined or Hybrid Azure AD Joined (for recovery key escrow), and the Intune BitLocker policy must have "Hide BitLocker prompt" set. Without TPM 2.0, Intune must prompt the user to set a password or startup PIN — silent enablement is not possible.
Where does Intune store BitLocker recovery keys and how does a helpdesk admin retrieve one?
Intune automatically stores BitLocker recovery keys in Azure AD when the BitLocker policy is configured to save recovery info to Azure AD. To retrieve: in Intune portal, find the device, click the Recovery keys tab, and view the BitLocker recovery key. Alternatively in the Azure AD portal, find the device and view BitLocker keys. RBAC roles can be scoped to allow helpdesk staff to view recovery keys without full Intune admin access — principle of least privilege applied to recovery key access.
A device shows BitLocker status as "Suspended" — should you be concerned?
A temporary suspension is normal and expected during Windows OS updates, BIOS/UEFI firmware updates, or hardware configuration changes — these require suspending BitLocker to allow the boot environment to change without triggering a recovery key prompt. BitLocker should automatically resume after the next reboot. If a device shows Suspended for more than 1-2 days with no pending updates, investigate — run manage-bde -status to see the suspension reason and manually resume with manage-bde -protectors -enable C: if needed.
TOPIC 72
Account Protection — LAPS & Credential Guard
Protecting local accounts and credential theft — stopping lateral movement
The Account Protection Threat

One of the most common lateral movement techniques in enterprise attacks: an attacker gains access to one workstation, extracts the local administrator password, and finds that the same password is set on every other workstation in the company (because IT set them all to the same password for convenience years ago). They now own the entire fleet. LAPS and Credential Guard are the two primary defenses against this.

LAPS — Local Administrator Password Solution

Windows LAPS (built into Windows since April 2023, replacing the old standalone LAPS tool) automatically generates a unique, random password for the local Administrator account on each device and stores it in Azure AD (or on-prem AD). The password rotates on a schedule. IT can retrieve it per-device when needed.

How LAPS Works
  1. 1
    LAPS is configured via Intune Endpoint Security → Account Protection → Create Policy → Windows LAPS
  2. 2
    Key settings:

    • Password age (days): 30 days recommended — rotating monthly
    • Password complexity: Large letters + small letters + numbers + specials
    • Password length: 14–20 characters recommended
    • Post-auth action: Reset password after local admin account is used (prevents reuse after IT access)

Why LAPS Is Critical for Security
  • Stops Pass-the-Hash attacks: Even if an attacker extracts one device's local admin hash, it only works on that one device — not the entire fleet.
  • Eliminates shared credentials: No more "the IT password is Password1!" across all workstations.
  • Audit trail: Every LAPS password retrieval is logged — who viewed which device's password and when.
  • Emergency access: If a device is locked out (user forgot password, BitLocker boot issues), LAPS provides a tested recovery path without reimage.
Credential Guard

Windows Defender Credential Guard uses hardware virtualization (Hyper-V) to isolate credential hashes in a protected virtual container (VSM — Virtual Secure Mode). Even if an attacker has SYSTEM-level access or runs Mimikatz on the machine, they cannot access the credential hashes because they're isolated in a hardware-protected environment unreachable from the normal OS.

Windows Hello for Business

Also under Account Protection: Windows Hello for Business replaces passwords for Windows sign-in with biometrics (face, fingerprint) or a device-bound PIN that is cryptographically linked to the TPM — making it phishing-resistant (unlike passwords which can be stolen via phishing).

  • PIN ≠ Password: WHfB PIN is tied to the specific device's TPM. Even if someone knows your PIN, they can't use it on another device.
  • Phishing-resistant: No password to steal via phishing — the credential is hardware-bound.
  • Configure via Intune: Endpoint Security → Account Protection → Windows Hello for Business profile
  • Key settings: Require Hello (not optional), minimum PIN length (8+), allow biometrics, require anti-spoofing for face recognition
🎤 Interview Questions — Topic 72
What is LAPS and what security problem does it solve?
LAPS (Local Administrator Password Solution) automatically generates a unique random password for the local Administrator account on each device and stores it in Azure AD (or on-prem AD), rotating it on a schedule. It solves the lateral movement problem caused by shared local admin passwords — in many environments, all workstations have the same local admin password, meaning one compromised device gives an attacker admin access to the entire fleet. With LAPS, each device has a unique password — compromising one device's local admin credential doesn't help the attacker on any other device.
How does Credential Guard protect against Mimikatz?
Credential Guard uses hardware virtualization to move credential hashes (NTLM hashes, Kerberos tickets) from the normal LSASS process into an isolated Virtual Secure Mode container protected by the CPU's virtualization extensions. Mimikatz works by reading from the LSASS process memory, but with Credential Guard, the actual credential material is in the VSM container which is completely inaccessible from the normal OS — even from SYSTEM context or kernel level. Mimikatz can still run but gets empty or unusable data instead of real credential hashes.

⚠️ Security Baseline Conflicts with Other Policies

Security Baselines and other Configuration Profiles/Endpoint Security policies can set the same settings to different values — causing conflicts. When a conflict occurs on a specific setting, the baseline value is NOT applied for that setting (the conflict resolution rules apply — see Block 4). Best practice: If you use Security Baselines, avoid configuring the same settings in separate Endpoint Security profiles or Config Profiles. The Baseline should own its settings exclusively to avoid conflicts.

What Is EDR Onboarding?

EDR onboarding is the process of installing the MDE sensor on devices so they report security events, alerts, and telemetry to the Microsoft Defender for Endpoint portal. Without onboarding, a device has Defender AV running but MDE's detection and response capabilities are blind to that device. Onboarding is the bridge from basic AV to full EDR visibility.

🎤 Interview Questions — Topic 75
What is EDR onboarding and why is it necessary if Defender AV is already running?
EDR onboarding installs the Microsoft Defender for Endpoint sensor on a device, enabling it to send detailed security telemetry (process activity, network connections, file events, registry changes) to the MDE cloud portal. Defender AV running locally prevents known threats but provides no visibility into the MDE portal — the security team can't investigate attacks, see device timelines, or run threat hunting queries. Onboarding is what transforms a device from "AV-protected" to "EDR-monitored" — giving the security team full investigation and response capabilities.
What is the recommended method for onboarding Intune-enrolled Windows devices to MDE at scale?
The recommended method is using the Intune EDR policy (Endpoint Security → Endpoint detection and response → Create Policy). With the MDE-Intune connector configured, select "Auto from connector" as the configuration package type — Intune automatically obtains the onboarding blob from MDE and deploys it to all targeted devices silently. This is fully automated and scales to thousands of devices with zero manual effort. Devices appear in the MDE portal within 30-60 minutes of the policy applying.
How do you verify that a Windows device has been successfully onboarded to MDE?
Three verification methods: (1) MDE portal — device inventory at security.microsoft.com should show the device within 30-60 minutes of onboarding. (2) Intune portal — the EDR policy's device status should show "Succeeded" for that device. (3) On the device itself — check Services for "Microsoft Defender Advanced Threat Protection Service" (should be Running), or check Event Viewer → SENSE logs for onboarding success events, or run the MsSense.exe health check command which reports the sensor's operational status.
