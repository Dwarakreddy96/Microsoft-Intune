Enrollment Methods Overview
Every way to get a device into Intune — by platform and scenario
Why So Many Methods?

There isn't one single way to enroll devices in Intune — there are at least 15+ different methods across platforms. The right method depends on the platform (Windows, iOS, Android, Mac), who owns the device (company or employee), the volume (single device or 10,000), and how much user involvement you want. This topic gives you the master map.

The Enrollment Decision Tree
🧭 How to Pick the Right Enrollment Method

Step 1 — What platform? Windows / iOS / Android / macOS
Step 2 — Company-owned or personal (BYOD)?
→ Company-owned: Use corporate enrollment methods (Autopilot, ADE, Fully Managed Android)
→ Personal (BYOD): Use user-initiated methods (Company Portal, Work Profile, MAM-WE)
Step 3 — How many devices?
→ Scale (100+): Use zero-touch methods (Autopilot, ADE, Zero Touch, PPKG)
→ Small batch: Manual or semi-automated methods
Step 4 — What level of control?
→ Full control (kiosk, shared): Self-deploying Autopilot, Dedicated Android, Apple Configurator
→ User-owned, partial control: Work Profile, MAM-WE

🎤 Interview Questions — Topic 16
What enrollment methods are available for company-owned Windows devices?
Windows Autopilot (user-driven or self-deploying), Automatic enrollment via Azure AD Join at OOBE, Bulk enrollment using Windows Configuration Designer (PPKG), and Group Policy-based auto-enrollment for Hybrid Azure AD Joined devices.
What's the difference between supervised and unsupervised iOS devices in Intune context?
Supervised iOS devices (enrolled via ADE/ABM) give Intune much deeper control — you can prevent device removal from MDM, disable the App Store, enforce restrictions that unsupervised devices can ignore, enable Single App Mode, and more. Unsupervised devices (BYOD via Company Portal) have limited management capabilities — users can always remove the MDM profile.
TOPIC 17
Windows Autopilot — Concept, Setup & Deployment
Zero-touch Windows provisioning — the most powerful Intune enrollment feature
What Is Windows Autopilot?

Windows Autopilot is a zero-touch device provisioning service. It transforms the Out-of-Box Experience (OOBE) so that when an employee receives a brand-new laptop and turns it on, the machine automatically configures itself — joining Azure AD, enrolling in Intune, installing all required apps, applying all policies — without IT ever touching the machine physically.

The old way: IT receives the laptop → images it → configures it → ships it to the employee. 2–4 hours of IT time per device.
The Autopilot way: Device ships directly from manufacturer to employee's home. Employee turns it on. Done. 0 hours of IT time per device.

How Autopilot Works — The Technical Flow
  1. 1
    Hardware Hash Registration: The device's unique hardware identifier (hardware hash — contains CPU, motherboard, NIC details) is extracted and uploaded to the Autopilot service. This can be done by the OEM (Dell, HP, Lenovo) at the factory, or by IT using PowerShell.
  2. 2
    Autopilot Profile Assignment: In Intune portal, you create an Autopilot Deployment Profile and assign it to the device or a device group. This profile defines the OOBE experience — skip privacy settings, skip license agreement, set the account type, etc.
  3. 3
    Device Powers On: Employee receives the device, connects to internet (Wi-Fi or ethernet), powers it on.
  4. 4
    Windows OOBE contacts Autopilot service: Windows detects it's an Autopilot device (by matching hardware hash), downloads the Autopilot profile, and applies the customized setup experience.
  5. 5
    User signs in with Azure AD credentials: Instead of creating a local account, user enters name@company.com. Azure AD authenticates them (with MFA if required).
  6. 6
    Azure AD Join + Intune Auto-enrollment: Device joins Azure AD. MDM scope is configured, so Intune enrollment triggers automatically. No enrollment URL needed.
  7. 7
    Enrollment Status Page (ESP): Shows the employee progress — "Setting up your device... Installing apps... Applying policies..." This keeps the user from using the device before everything is done.
  8. 8
    Device Ready: All apps installed, all policies applied, device renamed (if configured), user can start working. IT never touched it.
Registering Devices into Autopilot
🏭
OEM Registration (Best)
Dell, HP, Lenovo, Surface can register devices directly into your Autopilot tenant at the factory. You give them your tenant ID when ordering. Devices arrive ready. Zero effort.
🤝
Reseller/Partner Registration
Microsoft CSP partners can register devices on behalf of customers. Common when buying through a Microsoft partner or VAR (Value Added Reseller).
⚙️
Manual via PowerShell
Run the Get-WindowsAutoPilotInfo script on an existing device to extract hardware hash, export as CSV, then import into Intune Autopilot blade. For existing devices or small batches.
📤
CSV Import in Intune
Upload a CSV file of hardware hashes directly in Intune → Devices → Enrollment → Windows → Autopilot Devices → Import. Works for any quantity.
PowerShell Script to Extract Hardware Hash
# Step 1: Install the script module (run as admin) Install-Script -Name Get-WindowsAutoPilotInfo # Step 2: Extract hardware hash and save to CSV Get-WindowsAutoPilotInfo -OutputFile C:\AutopilotHash.csv # Step 3: For immediate Autopilot registration (needs internet) Get-WindowsAutoPilotInfo -Online # The CSV contains: Device Serial Number, Windows Product ID, Hardware Hash # Import this CSV into Intune → Devices → Enrollment → Autopilot → Import

Enrollment Status Page (ESP)

The Enrollment Status Page is a critical Autopilot component. It shows the user a progress screen while the device is being set up. More importantly, it blocks the user from using the device until all required apps and policies are installed. Without ESP, a user might start working before security policies are applied.

  • Configure in: Intune → Devices → Enrollment → Windows → Enrollment Status Page
  • Block device until apps install: Set this to Yes for required apps
  • Allow users to reset device if error: Useful for Autopilot failures during initial setup
  • Show error when installation takes longer than X minutes: Default 60 minutes, can increase for large app deployments

Company: IT services firm, hiring 200 new employees across India in 3 months.

Autopilot process: Company orders Dell laptops and tells Dell to register hardware hashes directly to their Autopilot tenant. Dell ships laptops directly to each employee's address. Each employee turns on laptop, connects to WiFi, signs in with company account. 45 minutes later, the machine is fully configured with all apps and policies. IT team spent 2 hours total setting up the Autopilot profile — not 400 hours.
Saving: 398 hours of IT time. Zero physical device handling. Laptops shipped from Dell warehouse to employees' homes directly.

🎤 Interview Questions — Topic 17
What is Windows Autopilot and what problem does it solve?
Autopilot is a zero-touch device provisioning service. It eliminates the need for IT to manually image or configure new Windows devices. Devices can be shipped directly from manufacturer to employee, and they self-configure on first boot by connecting to the Autopilot service and Intune — joining Azure AD, installing apps, applying policies automatically.
How do you register a device into Autopilot?
Four methods: OEM registration (manufacturer registers at factory), CSP/reseller registration, manual registration using the Get-WindowsAutoPilotInfo PowerShell script to extract hardware hash and import as CSV, or using Windows Configuration Designer. For large deployments, OEM or CSP registration is preferred.
What is the Enrollment Status Page and why is it important?
ESP displays a setup progress screen during Autopilot enrollment. Critically, it blocks the user from using the device until all required apps and policies are installed. Without ESP, a user could start working on an unsecured device before security baselines, antivirus, or encryption policies have been applied.
What is a device naming template in Autopilot and give an example?
A naming template automatically sets the computer name during Autopilot. Example: "IND-%RAND:4%" would name devices IND-A3F2, IND-B7K9 etc. Or "CORP-%SERIAL%" uses the device serial number. Consistent naming helps with asset management, support tickets, and log searching.
TOPIC 18
Autopilot — User-Driven vs Self-Deploying Mode
Choosing the right Autopilot mode for your scenario
The Two Primary Autopilot Modes
👤
User-Driven Mode
The employee interacts with the device during setup. They enter their Azure AD credentials. The device gets associated with that specific user. All user-specific policies and apps are applied based on who they are. Most common mode for regular employee laptops.
🤖
Self-Deploying Mode
Zero human interaction. Device provisions itself completely automatically — no username, no password entry needed. Uses the device's TPM 2.0 chip to authenticate with Azure AD. Used for kiosks, shared workstations, meeting room devices, digital signage.

Third Mode — Pre-Provisioning (White Glove)

Autopilot Pre-Provisioning (formerly called White Glove) is a hybrid approach where IT or a reseller pre-stages the device before handing it to the user. In this mode:

  1. 1
    IT/Reseller stage: Device boots, technician presses Windows key 5 times at OOBE, selects "Pre-provision." Device installs device-level apps and policies. A "success" provisioning key is stored.
  2. 2
    Device is shipped/handed to user: The device is now partially configured with device-side content.
  3. 3
    User completes setup: User turns on device, enters their credentials. Only user-specific apps and policies need to install — much faster than a full enrollment.
💡
When to use Pre-Provisioning: When your devices have large apps (like Autodesk, Adobe suite) that take 2+ hours to install. Pre-install device-side apps in the warehouse, so the user only waits 10-15 minutes for their user-specific apps. Improves the first-day experience significantly.
Self-Deploying Mode Requirements
⚠️ Critical Requirements for Self-Deploying Mode

1. TPM 2.0 chip is MANDATORY. The device uses TPM attestation to authenticate itself to Azure AD without user credentials. Virtual machines and older hardware without TPM 2.0 will fail self-deploying enrollment with error 0x800705B4.
2. Network connectivity at boot: The device must reach the internet immediately — before user login. Configure DHCP or ensure ethernet is available. Wi-Fi profile can't be applied before enrollment, so ethernet is recommended for first boot.
3. The device MUST be in Autopilot first. Hardware hash must be registered before the device ships.

🏪
Self-Deploying Kiosk Scenario

Company: Retail chain deploying 500 self-checkout terminals across 100 stores.
Requirements: Each terminal must run exactly one app (checkout software) in kiosk mode. No user login. Device must auto-recover if something goes wrong. IT can't visit each store.
Solution: Self-Deploying Autopilot + Kiosk Mode profile. Each terminal arrives, boots, connects to store WiFi (pre-configured via PPKG on first boot), self-deploys via Autopilot, Intune pushes Kiosk profile restricting the device to only the checkout app. If terminal needs a reset, it re-provisions itself automatically. IT manages 500 terminals from one Intune portal in Mumbai.

🎤 Interview Questions — Topic 18
When would you use Self-Deploying Autopilot over User-Driven?
Self-Deploying is for devices with no dedicated user — kiosks, shared workstations, meeting room PCs, digital signage. No user credentials are entered during setup; the device provisions itself using TPM attestation. User-Driven is for personal-use corporate laptops where the device gets associated with the employee who signs in.
A self-deploying Autopilot device fails with error 0x800705B4. What is the cause?
This error means the device does not have a TPM 2.0 chip, or TPM 2.0 is not enabled in BIOS/UEFI. Self-deploying mode requires TPM 2.0 for device attestation since no user credentials are entered. The fix is to either enable TPM in BIOS or replace with hardware that has TPM 2.0.
What is Autopilot Pre-Provisioning and when is it useful?
Pre-provisioning (White Glove) allows IT or a reseller to pre-stage device-side apps and policies before handing the device to the user. The user then only needs to complete the user-side setup, which is much faster. It's useful when devices have large applications that take hours to install — IT installs those upfront, improving the employee's first-day experience.
TOPIC 19
Windows Enrollment — Automatic, Manual & Bulk (PPKG)
Non-Autopilot Windows enrollment methods for various scenarios
1. Automatic Enrollment via Azure AD Join

When a device is Azure AD Joined (user signs in with work account during Windows setup), Intune auto-enrollment is triggered automatically — provided the MDM User Scope is configured in Azure AD. This is the simplest enrollment path for new devices.

How it works: User sets up Windows → selects "Set up for an organization" → enters work email → authenticates → Windows joins Azure AD → Azure AD detects MDM scope → automatically enrolls in Intune. No separate enrollment step needed.

MDM User Scope — The Trigger for Auto-Enrollment
  1. 1
    Go to entra.microsoft.com → Identity → Mobility (MDM and MAM) → Microsoft Intune
  2. 2
    Set MDM User Scope to: All (everyone auto-enrolls) or Some (only specific Azure AD groups)
  3. 3
    Optionally set MAM User Scope for BYOD app management without full enrollment
  4. 4
    Save. Now any Azure AD Joined device for users in scope will auto-enroll in Intune.
2. Manual Enrollment (Settings App)

For existing Windows devices that are already set up and need to be enrolled. User does this themselves or IT can do it remotely.

  1. 1
    Go to Settings → Accounts → Access work or school
  2. 2
    Click Connect
  3. 3
    Enter work email address
  4. 4
    Sign in with Azure AD credentials
  5. 5
    Device joins Azure AD and auto-enrolls in Intune (if MDM scope is configured)
💡 Difference: Connect vs Join

In Settings → Access work or school: "Connect" = Azure AD Register (BYOD — just registers, doesn't fully join). "Join this device to Azure AD" (found by clicking "Join this device to Azure Active Directory" option) = full Azure AD Join. For full MDM management, you want the device to be Azure AD Joined, not just registered.

3. Bulk Enrollment via PPKG (Provisioning Package)

A Provisioning Package (.ppkg) is a file created with Windows Configuration Designer (WCD) — a free tool from the Microsoft Store. The PPKG contains an enrollment token and configuration. You put it on a USB drive and run it on any compatible Windows device — it enrolls the device in Intune automatically, no internet during setup required.

✅
When to Use PPKG
Devices that don't have internet at first boot (factory floors, ships, remote sites). Legacy hardware not supported by Autopilot. Bulk provisioning of devices bought from non-OEM-registered vendors. Quick enrollment of 10-50 devices without Autopilot setup.
⚠️
PPKG Limitations
The enrollment token inside the PPKG expires (180 days by default). PPKG creates a bulk enrollment service account — devices won't be associated with individual users initially. Needs re-generation when token expires.
Creating a PPKG — Step by Step
  1. 1
    Download and install Windows Configuration Designer from Microsoft Store on a Windows PC.
  2. 2
    Open WCD → Click "Provision desktop devices"
  3. 3
    Go to Set up device → Enter device name template
  4. 4
    Go to Set up network → Optionally pre-configure Wi-Fi SSID and password
  5. 5
    Go to Enroll in device management → Enter your organization's UPN domain (e.g., contoso.com) → Authenticate with Intune admin account → A bulk enrollment token is fetched and embedded in the PPKG.
  6. 6
    Click Create → Export as .ppkg file
  7. 7
    Copy .ppkg to USB drive. On target device, run the .ppkg file at any point during or after OOBE. Device enrolls automatically.
4. Group Policy Auto-Enrollment (Hybrid AADJ)

For Hybrid Azure AD Joined devices (in both on-prem AD and Azure AD), you can trigger Intune enrollment automatically via Group Policy — no user action needed. IT sets a GPO and all domain-joined machines enroll in Intune silently in the background.

# GPO Path: Computer Configuration → Policies → Administrative Templates → Windows Components → MDM → Enable automatic MDM enrollment using default Azure AD credentials # Set to: Enabled # User Credential: Select "Device Credentials" (recommended) # After GPO applies, device enrolls in Intune silently # Check enrollment: Settings → Access work or school → Info button
🎤 Interview Questions — Topic 19
What is a PPKG and when would you use it over Autopilot?
A Provisioning Package is a file created with Windows Configuration Designer containing an Intune enrollment token. It's used when Autopilot isn't available — devices without internet at setup, hardware not registered in Autopilot, or legacy devices. You put the PPKG on USB and run it on any device. Use Autopilot for modern deployments at scale; use PPKG for edge cases or legacy hardware.
What setting must be configured in Azure AD for automatic Intune enrollment to work?
The MDM User Scope in Azure AD → Mobility (MDM and MAM) → Microsoft Intune must be set to "All" or "Some" (targeting specific groups). Without this setting enabled, Azure AD Join won't trigger Intune enrollment automatically — the device joins Azure AD but doesn't enroll in Intune.

iOS/iPadOS Enrollment — ADE, Apple Configurator & BYOD
Every iOS enrollment method — supervised vs unsupervised, corporate vs personal
The Critical iOS Concept: Supervised vs Unsupervised

iOS has a concept that Android and Windows don't have in the same way: Supervised mode. A supervised iPhone/iPad gives Intune significantly more management capabilities. Understanding this is fundamental to iOS management.

Method 1 — ADE (Automated Device Enrollment) via Apple Business Manager

ADE (formerly called DEP — Device Enrollment Program) is the gold standard for corporate iOS enrollment. Devices purchased through Apple or an authorized reseller are automatically enrolled in your Intune tenant when activated — the user can't bypass it.

  1. 1
    Purchase iPhones/iPads through Apple or an authorized ABM reseller. Ask them to link devices to your Apple Business Manager (ABM) account.
  2. 2
    In Apple Business Manager (business.apple.com): Go to MDM Servers → Add MDM Server → Download the public key certificate.
  3. 3
    In Intune Portal → Devices → iOS/iPadOS → Enrollment → Enrollment program tokens → Add. Upload the ABM server token. Download the Intune public key and upload to ABM. This creates the trusted connection.
  4. 4
    In Intune, create an Enrollment Profile for ADE: Set supervised mode = Yes. Configure OOBE setup screens to skip. Set MDM profile removal = Prohibited.
  5. 5
    Assign the enrollment profile to devices in Intune (they appear automatically since they're synced from ABM).
  6. 6
    Employee activates the iPhone. The device automatically pulls the enrollment profile from Apple's servers and enrolls in Intune. Supervised. IT-managed. User cannot remove MDM.
Method 2 — Apple Configurator 2

For organizations that don't have ABM or need to enroll existing devices not purchased through an authorized channel. Requires a Mac with Apple Configurator 2 installed.

  • Use case: Batch-enrolling 50 existing iPhones without ABM. School with donated iPads. Healthcare with older device inventory.
  • How: Connect iPhones via USB to Mac running Configurator 2. Apply an enrollment profile that points to your Intune tenant. Devices get supervised and enrolled.
  • Limitation: The enrollment profile CAN be removed by the user (unlike ADE where it's locked). Better than BYOD but not as locked as ADE.
  • New method: Apple Configurator 2 can now add devices to ABM via QR code or Wi-Fi pairing — eliminating the need for USB cables.
Method 3 — BYOD via Company Portal App

Employee uses their personal iPhone to access company resources. They install the Company Portal app from the App Store and follow the enrollment steps.

  1. 1
    Employee installs Intune Company Portal from App Store
  2. 2
    Opens app → Signs in with company Azure AD account
  3. 3
    Company Portal guides them through MDM profile installation
  4. 4
    A management profile is installed — device is now MDM enrolled
  5. 5
    Compliance policies apply. If non-compliant, Conditional Access blocks company resources until fixed.
⚠️ Important BYOD iOS Limitation

On personal iOS devices enrolled via Company Portal, the user CAN remove the MDM profile at any time from Settings → General → VPN & Device Management. When they do, all company apps and data installed by Intune are removed. IT gets notified but cannot prevent it. This is by design for privacy reasons. For devices where you need permanent management, use ADE.

🏥
Healthcare iOS Deployment Scenario

Company: Hospital deploying 300 iPhones to nurses for patient care app access.
Requirements: Devices must always be managed (nurses can't remove MDM). Single purpose — only the patient care app. Devices are shared between shifts. Must be supervised for full control.
Solution: Purchase iPhones through Apple Business Manager. Link to Intune via ADE token. Create enrollment profile: Supervised = Yes, MDM removal = Prohibited, Skip all OOBE screens. Create Kiosk mode configuration profile allowing only the patient care app. Result: Phones boot directly to the app. Nurses can't access personal apps. IT can remotely lock or wipe lost devices. MDM cannot be removed.

🎤 Interview Questions — Topic 20
What is ADE and how does it differ from enrolling via Company Portal?
ADE (Automated Device Enrollment) enrolls iOS devices automatically at activation using Apple Business Manager. Devices are supervised, and the MDM profile cannot be removed by the user. Company Portal enrollment is user-initiated for personal devices, gives less management control, and the user can remove the MDM profile at any time. ADE is for corporate-owned devices; Company Portal is for BYOD.
What does iOS Supervised mode enable that unsupervised does not?
Supervised mode enables: preventing MDM removal, disabling the App Store, Single App Mode / Kiosk mode, forced software updates, Activation Lock bypass, IT-managed Lost Mode, per-app VPN, web content filtering, and AirDrop/AirPlay restrictions. Unsupervised devices have very limited management capabilities.
What is an APNs certificate and why is it required for iOS management?
Apple Push Notification service (APNs) certificate is required for Intune to communicate with iOS/iPadOS devices. Apple requires all MDM solutions to have an APNs certificate to push management commands to iOS devices. It must be renewed annually — if it expires, Intune loses the ability to manage all iOS devices. It's created in Apple's Push Certificates Portal and uploaded to Intune under Tenant Administration → Apple MDM Push Certificate.
TOPIC 21
Android Enrollment — Android Enterprise, BYOD & Knox
The most complex enrollment landscape — multiple modes for multiple scenarios
Android Enrollment is More Complex — Here's Why

Android has the most diverse enrollment landscape of any platform because Android is open — any manufacturer can customize it, and there are vastly different use cases (personal phones, kiosks, rugged scanners, tablets). Microsoft has aligned with Android Enterprise — Google's enterprise management framework — as the modern approach. Legacy Device Administrator (DA) mode is deprecated. Always use Android Enterprise.

Android Enterprise Enrollment Modes
📱
Work Profile (BYOD)
Creates a separate encrypted work container on the employee's personal Android phone. Company manages only the work profile — personal apps and data are completely isolated. User privacy is maintained. Employee can remove the work profile at any time.
🔒
Fully Managed (COBO)
Company-owned, dedicated to one user. Company has full control of the entire device. No personal use. Device is enrolled during first boot. All apps come from managed Google Play. Used for sales reps, field workers with company phones.
🏪
Dedicated Device (Kiosk)
Company-owned, no user identity, single-purpose. Think warehouse scanner, retail checkout tablet, delivery driver device. Runs one or a small set of apps. No Google account. IT manages remotely.
🏢
Corporate-Owned Work Profile (COPE)
Company owns the device but allows limited personal use. Creates a work profile like BYOD, but IT also has full device visibility. Best of both worlds — employees can have personal apps while IT manages work data and the full device.

QR Code Enrollment for Fully Managed — Step by Step
  1. 1
    In Intune: Devices → Android → Android enrollment → Corporate-owned, fully managed user devices → Create profile. Configure enrollment token and download the QR code.
  2. 2
    Factory reset the Android device (or use new out-of-box device).
  3. 3

  4. 4

  5. 5

Samsung Knox Mobile Enrollment (KME)

Samsung has their own enterprise enrollment framework called Knox Mobile Enrollment (KME). Like Apple Business Manager, KME lets you register Samsung devices and have them auto-enroll in Intune when activated — without user interaction. Works only on Samsung devices with Knox hardware (most modern Samsung business phones).

  • How: Register devices in Knox Mobile Enrollment portal (knox.samsung.com). Create an MDM profile pointing to Intune. Assign to devices.
  • Benefit: Zero-touch Samsung enrollment — similar to Autopilot for Windows or ADE for iOS.
  • Best for: Companies standardized on Samsung Android devices needing bulk zero-touch enrollment.

Work Profile — Technical Deep Dive

When Work Profile is set up on an Android device, the operating system creates a separate encrypted partition called the work profile. Apps installed in the work profile (shown with a briefcase badge) are completely isolated from personal apps.

  • Data isolation: Work apps cannot access personal photos, contacts, or files. Personal apps cannot access work data. Copy-paste can be blocked between profiles.
  • Separate app instances: The user has TWO instances of some apps — e.g., personal Gmail AND work Gmail. They appear separately on the home screen.
  • Profile toggle: Users can pause the work profile (e.g., on weekends) — work notifications stop, work apps are inaccessible. IT can disable this feature.
  • IT visibility: IT can only see work profile apps and data. Personal side is completely invisible to IT. Android Enterprise guarantees this privacy by design.
  • Removal: User can remove the work profile from Settings → Accounts → Work Profile → Remove. This is a trade-off for BYOD scenarios.
Fully Managed — What IT Controls
  • All apps: Only apps deployed via managed Google Play are allowed. User cannot install apps from Play Store directly.
  • Factory reset protection: IT-managed — only IT can perform factory reset, not the user.
  • System settings: Block access to device settings, prevent changing Wi-Fi, prevent USB file transfer.
  • Screen capture: Can be blocked for sensitive apps.
  • Location: Can enforce location reporting.
  • Compliance: Full Intune compliance evaluation — blocks access if device compromised.

Linking ABM to Intune — Step by Step
  1. 1
    Create an APNs Certificate: In Intune → Tenant Administration → Apple MDM Push certificate. Download the CSR file. Upload to https://identity.apple.com/pushcert. Download the .pem certificate. Upload back to Intune. This enables Intune to communicate with Apple's push service for all iOS/macOS devices.
  2. 2
    Create MDM Server in ABM: Go to business.apple.com → MDM Servers → Add MDM Server → Name it "Microsoft Intune" → Download the public key token (.p7m file).
  3. 3
    Upload token to Intune: In Intune → Devices → iOS/iPadOS → Enrollment program tokens → Add → Upload the .p7m token file. Download Intune's public key and upload to ABM. This establishes secure trust between ABM and Intune.
  4. 4
    Add VPP token: In ABM → Apps and Books → Your organization → Get token. Upload to Intune → Tenant Administration → Apple VPP Tokens. Now you can purchase and deploy apps via Intune.
  5. 5
    Sync devices: Devices assigned to MDM in ABM will appear in Intune's Autopilot/ADE device list within 15 minutes (or force-sync in Intune).
APNs Certificate — Critical Maintenance
🚨 APNs Certificate Expiry — A Real Incident Risk

The APNs certificate is valid for only 1 year. If it expires, Intune immediately loses the ability to manage ALL iOS and macOS devices. Commands stop being delivered. Apps stop deploying. Policies stop applying. Compliance evaluation stops.

Critical rules:
1. Set a calendar reminder 60 days before expiry.
2. The renewal MUST be done with the SAME Apple ID that created the original certificate. Using a different Apple ID creates a new APNs certificate — all currently enrolled devices immediately lose management and must re-enroll.
3. Never use a personal Apple ID for APNs — use a corporate Apple ID that IT controls.
4. Check expiry in Intune → Tenant Administration → Apple MDM Push Certificate.

What Are Enrollment Restrictions?

Enrollment Restrictions are Intune policies that control which devices are allowed to enroll. Before a device even gets managed, these restrictions act as a gatekeeper. You can block personal devices, require a minimum OS version, limit the number of devices per user, and more.

Location in Intune: Devices → Enrollment → Enrollment device limit restrictions / Enrollment device platform restrictions

Type 1 — Device Platform Restrictions
RestrictionWhat It ControlsExample Use

Type 2 — Device Limit Restrictions

Controls how many devices a single user can enroll. Default is 15 devices per user. You can lower this.

  • Standard employees: Set limit to 2 (1 laptop + 1 phone)
  • IT admins: Higher limit (10+) for testing and management
  • Frontline workers: May need limit of 1 (shared devices, not user-specific)
Priority and Assignment

Enrollment restrictions have a priority system. When multiple restrictions exist, the highest-priority restriction assigned to the user's group is applied. The default restriction (lowest priority) applies to everyone not covered by a specific restriction.

🔐
Real-World Enrollment Restriction Scenario

Company policy: Only company-owned devices can enroll for most employees. Developers can enroll personal devices for testing. No iOS device below version 16 allowed.
Setup:
Restriction 1 (Priority 100 — assigned to "Developers" group): Allow personal devices on all platforms. Max 5 devices.
Restriction 2 (Priority 200 — assigned to "All Employees"): Block personally-owned devices on all platforms. Minimum iOS 16.0. Minimum Android 11. Max 2 devices per user.
Default restriction: Block everything except IT Admins group.
Result: Developers can enroll personal test devices. Regular employees can only enroll company devices. Personal devices from employees get blocked with a "Device type not allowed" error.

What Are Device Categories?

Device Categories allow you to classify devices into types when they enroll. During enrollment, the user is asked to select a category (e.g., "Corporate," "Warehouse," "Shared," "Developer"). The device is tagged with this category. You then create dynamic Azure AD groups that automatically include devices with that category tag.

Setting Up Device Categories
  1. 1
    Go to Intune → Devices → Device categories → Create device category
  2. 2
    Create categories like: "Corporate Laptop," "Warehouse Scanner," "Shared Kiosk," "Developer Device"
  3. 3
    During Company Portal enrollment, the user sees a prompt: "What type of device is this?" They select a category.
  4. 4
    Create a dynamic Azure AD group with rule: device.deviceCategory -eq "Corporate Laptop"
  5. 5
    Assign Intune policies (configuration profiles, apps, compliance policies) to that dynamic group.
  6. 6
    Any newly enrolled device that selects "Corporate Laptop" category automatically joins the group and gets the right policies. Zero manual work.

The Troubleshooting Mindset

Enrollment failures are the most common tickets you'll handle as an L1/L2 Intune admin. The key is a systematic approach — check each layer in order: License → MDM Scope → Enrollment Restriction → Platform-specific issue → Network/Certificate. Don't jump to complex causes when simple ones haven't been ruled out.

Enrollment Troubleshooting Checklist (Universal)
  1. 1
    Does the user have an Intune license? → Check Azure AD → User → Licenses. Without Intune license, enrollment fails immediately.
  2. 2
    Is the user in the MDM User Scope? → Check Azure AD → Mobility (MDM and MAM) → Microsoft Intune → MDM User Scope. If set to "Some," verify the user is in the included group.
  3. 3
    Is there an enrollment restriction blocking this device? → Check Intune → Devices → Enrollment → Device Platform Restrictions. Is the platform blocked? Is it a personal device and personal devices are blocked?
  4. 4
    Has the device limit been reached? → In Intune, find the user, check their enrolled device count vs their device limit restriction.
  5. 5
    Is the enrollment error code documented? → Note the exact error code and look it up in Microsoft Docs or Intune Troubleshooting blade.
  6. 6
    Use the Intune Troubleshooting Blade: → Intune → Troubleshoot & Support → Search user → Review Enrollment failures, Device compliance, Policy issues all in one view.
