TOPIC 81
What Is Windows Autopatch?
The "managed patching service" — what it is, what it patches, and the problem it solves
The Patching Problem Autopatch Solves

Patch Tuesday happens every month. Microsoft releases security updates, cumulative updates, feature updates, Edge updates, and Microsoft 365 App updates — all on the same day. For a 1,000-device fleet, an IT team must test these updates, plan rollout schedules, create Update Rings in Intune, monitor deployment, investigate failures, and handle exceptions. It takes significant IT time every single month — and if patches are delayed, devices become vulnerable.

Windows Autopatch is Microsoft's answer: hand the entire update management lifecycle to Microsoft. You enroll your devices, and Microsoft handles scheduling, deployment rings, monitoring, failure detection, and rollback — automatically. Your IT team goes from spending days per month on patch management to spending close to zero.

What Autopatch Manages
🪟
Windows Quality Updates
Monthly cumulative security and quality patches (Patch Tuesday). Autopatch targets 95% of devices updated within 21 days of release. Devices are deployed in rings to catch issues before broad rollout.
⬆️
Windows Feature Updates
Annual Windows version upgrades (e.g., Windows 11 22H2 → 23H2). Autopatch manages the gradual rollout across rings with automatic compatibility checking before deployment.
📊
Microsoft 365 Apps Updates
Office app updates (Word, Excel, Outlook, Teams). Autopatch aligns M365 Apps updates with the corresponding Windows update rings for coordinated, tested deployments.
🌐
Microsoft Edge Updates
Chromium-based Edge browser updates. Autopatch keeps Edge current across the fleet automatically, aligned with the same ring structure.
🔑
Microsoft Teams Updates
Teams desktop app updates managed alongside other Microsoft product updates through Autopatch.
🚫
Does NOT Manage
Third-party apps (Chrome, Adobe, Zoom), Windows Server, non-Microsoft software, BIOS/firmware updates. Autopatch is Microsoft-product-focused only.

How Autopatch Fits in the Microsoft SLA

Microsoft makes a formal Service Level Objective (SLO) with Autopatch: 95% of devices will have Windows quality updates installed within 21 days of release. This is not a guarantee you can hold Microsoft contractually liable for, but it represents Microsoft's commitment and target. For many organizations, this SLO is better than what their internal IT team achieves with manual processes.

✅ The Core Value Proposition

Autopatch shifts the update management SLA from "we'll get to it when we can" to a Microsoft-backed commitment. Security teams can point to Autopatch as their patch management control — satisfying auditors, cyber insurance requirements, and compliance frameworks that demand timely patching. The IT team is freed from Patch Tuesday firefighting to focus on higher-value work.

Licensing Requirements
  • Required: Windows 10/11 Enterprise or Professional with one of:
    — Microsoft 365 E3 or higher
    — Windows 10/11 Enterprise E3 or E5
    — Microsoft 365 Business Premium
  • Also required: Azure AD Premium P1 (for dynamic groups), Microsoft Intune (for device management)
  • NOT available for: Windows Home edition, Windows Server, non-Windows platforms

TOPIC 82
Autopatch Groups & Deployment Rings
How Autopatch intelligently stages updates across your fleet
The Ring Architecture Concept

The core of Autopatch is a deployment ring architecture — devices are divided into groups that receive updates at different times. The earliest rings get updates first and act as canaries; if updates cause problems in early rings, Autopatch detects this and pauses rollout before it reaches the majority of devices. This prevents a bad update from impacting your entire fleet simultaneously.

Device Assignment to Rings

Autopatch automatically assigns devices to rings when they enroll. The default assignment uses a random distribution based on the ring percentages (1/9/20/70). You can manually move specific devices between rings:

  • Test Ring candidates: IT team's own devices, dedicated test machines, lab devices — people who can handle early-adoption issues.
  • First Ring candidates: Tech-savvy early adopters, IT department members, developers who benefit from staying current.
  • Fast Ring candidates: Flexible knowledge workers, departments with quick issue-reporting capability.
  • Broad Ring candidates: Executives, critical business systems, operational staff, users who need maximum stability.
Autopatch Groups (The New Model)

Microsoft introduced Autopatch Groups as an enhancement to the basic ring structure. Autopatch Groups let you create custom deployment groups with their own ring configurations — useful for organizations with multiple business units, subsidiaries, or compliance domains that need different update timing.

🏢
Default Autopatch Group
The standard group created automatically during enrollment. Uses the 4-ring model (Test/First/Fast/Broad) with Microsoft's default timing. Covers the majority of enrolled devices. Suitable for most organizations as-is.
⚙️
Custom Autopatch Groups
Create separate deployment groups with custom ring timing. Example: a "Critical Systems" Autopatch Group with a 30-day deferral before Broad ring, separate from the standard Default Group. Each group has its own Test → Broad progression.
🏥
Compliance Use Case
Healthcare organization with clinical devices that must be updated within 7 days (compliance requirement) AND back-office devices where stability matters more. Two Autopatch Groups with different SLA timings, both monitored in one dashboard.
Automatic Pause Logic — How Autopatch Protects Your Fleet

One of Autopatch's most valuable capabilities: it automatically monitors deployment quality across rings and pauses rollout when issues are detected. The thresholds that trigger a pause:

Manual Pause and Rollback

IT administrators can also manually intervene:

  • Pause an update: Intune → Reports → Windows Updates → Autopatch → Find the specific update → Pause. Stops deployment to remaining rings until you resume.
  • Resume deployment: After investigating and confirming the issue is resolved, resume the update rollout from where it stopped.
  • Release a device from a ring: Move a specific troubled device to the Test ring for investigation without affecting other ring members.
🛑
Autopatch Auto-Pause Real Scenario

Scenario: Microsoft releases October Patch Tuesday. Autopatch deploys to Test ring (5 devices) on Day 0 — no issues. Deploys to First ring (45 devices) on Day 1 — no issues. Deploys to Fast ring (100 devices) on Day 6.
Day 7: 8 of the 100 Fast ring devices report BSOD after update. Failure rate = 8% — exceeds the 5% threshold.
Autopatch action: Automatically pauses the Broad ring (350 devices) — they don't receive the update. Raises a Service Health advisory in the Intune portal: "Quality update deployment paused due to elevated device failure rate. Investigating."
Microsoft investigation: The update had a conflict with a specific driver version on HP EliteBook models.
Resolution: Microsoft releases an updated build (KB revision) that fixes the driver conflict. Autopatch automatically resumes the Broad ring deployment with the fixed build.
Protected: 350 devices never received the bad update. Zero IT intervention required — Autopatch handled everything.

🎤 Interview Questions — Topic 82
Describe the four default Autopatch deployment rings and their timing.
Test ring (1% of devices): receives updates immediately on release day — zero deferral. First ring (9%): receives updates 1 day after Test, for early adopters. Fast ring (20%): receives updates 6 days after Patch Tuesday release, after Test and First have validated. Broad ring (70%): receives updates 9 days after release, after the majority of the fleet validation period. This staged approach catches issues in smaller populations before they affect the bulk of devices.
What happens automatically when Autopatch detects an elevated failure rate in a deployment ring?
When the device failure rate in a ring exceeds 5% within 2 days of an update deploying, Autopatch automatically pauses the rollout to subsequent rings. For example, if 8% of Fast ring devices fail, Broad ring deployment is paused immediately — protecting 70% of devices from receiving the problematic update. A Service Health advisory appears in the Intune portal. Autopatch then investigates, and if Microsoft releases a fixed update, Autopatch automatically resumes deployment with the corrected build — all without IT intervention.
What are Autopatch Groups and when would you create a custom one?
Autopatch Groups are deployment units with their own ring configuration and timing, beyond the single default four-ring model. You'd create a custom Autopatch Group when different parts of your organization need different update policies — for example, a "Critical Infrastructure" group with a 30-day deferral before Broad ring deployment (for servers or OT-adjacent PCs that need maximum stability testing), while the Default Group follows the standard 9-day timeline. Each group has independent monitoring and pause/resume controls.

Autopatch Enrollment Requirements
What your environment needs to be ready for Autopatch — the full checklist
The Prerequisites Checklist

Autopatch doesn't work in every environment. Before enrolling, your environment must meet specific requirements. Microsoft provides a Readiness Assessment tool built into the enrollment flow that automatically checks most of these — but knowing them beforehand saves troubleshooting time.

🪪
Azure AD Join or Hybrid Azure AD Join — Autopatch requires devices to have an Azure AD identity. Pure on-premises AD joined devices (no Azure AD) are not supported.
Check: dsregcmd /status → AzureAdJoined: YES
📱
Microsoft Intune enrollment — Devices must be enrolled in Intune (MDM). Autopatch uses Intune to deploy update policies and collect device status telemetry.
Check: Settings → Accounts → Access work or school → Info (MDM enrolled)
🪟
Windows 10 1809+ or Windows 11 (any version) — Older Windows 10 versions are not supported. All editions must be Enterprise or Professional.
Windows Home edition is NOT supported by Autopatch
🔑
Supported license — Microsoft 365 E3, E5, F3, Windows 10/11 Enterprise E3/E5, or Microsoft 365 Business Premium. The specific Intune + Azure AD P1 entitlements within these licenses are what enables Autopatch.
M365 Business Basic does NOT include Autopatch
🌐
Windows Update accessible — Devices must be able to reach Windows Update (windowsupdate.microsoft.com) and related endpoints. If using a proxy or firewall, these URLs must be allowlisted.
If using WSUS, it must NOT be configured to replace Windows Update for Autopatch-managed devices
🔄
No conflicting Windows Update policies — Existing Intune Update Rings or Configuration Profile settings that configure Windows Update settings will conflict with Autopatch. These must be removed from Autopatch-enrolled devices before or during enrollment.
Autopatch creates its own Update Ring policies — duplicates cause conflicts
🔒
Windows Update for Business (WUfB) not conflicting — Existing WUfB policies (deferral settings, deployment rings) must not be applied to devices that will be enrolled in Autopatch.
Autopatch replaces WUfB management for enrolled devices
📊
Windows Diagnostic Data — Devices must send Windows Diagnostic Data at Enhanced level or higher to Microsoft. Autopatch uses telemetry to determine update health and failure detection.
Required: Full telemetry (DiagTrack service running). Basic telemetry is insufficient for failure detection.
👥
Azure AD groups available — Autopatch creates four Azure AD device groups during enrollment (Test, First, Fast, Broad rings). These groups must be manageable by Intune (not locked by policy).
Group names: Modern Workplace Devices - Windows Autopatch - Test/First/Fast/All
The Readiness Assessment Tool

Before fully enrolling, run the Readiness Assessment which automatically checks your environment against all prerequisites:

  1. 1
    Go to Intune → Devices → Windows Updates → Windows Autopatch
  2. 2
    Click "Enroll" or "Run Readiness Assessment"
  3. 3
    The tool checks: license validity, Azure AD configuration, Intune connectivity, existing policy conflicts, network requirements, and tenant health.
  4. 4
    Review results: each check is marked ✅ Ready, ⚠️ Advisory, or ❌ Not Ready
  5. 5
    Fix all "Not Ready" items. Advisory items are warnings — consider fixing but not blockers. When all critical checks pass, proceed with enrollment.
Enrollment Steps
  1. 1
    Fix existing Update Ring conflicts: Before enrolling, exclude the devices you want to enroll in Autopatch from any existing Intune Update Ring assignments. Autopatch will create its own rings.
  2. 2
    Run Readiness Assessment and resolve all blockers.
  3. 3
    Enroll the tenant: Click "Enroll" in the Autopatch blade. This creates the four Azure AD groups and four Intune Update Ring policies.
  4. 4
    Assign devices to rings: Add devices to the Autopatch device groups. Autopatch randomly distributes them per ring percentages, or you can manually assign to specific rings.
  5. 5
    Verify enrollment: Devices should appear in the Autopatch device registration report within 24 hours. Check for "Not ready" devices that need remediation.
⚠️ WSUS Conflict — Common Blocking Issue

Many enterprise environments use WSUS (Windows Server Update Services) to approve and distribute Windows updates. If your Intune-enrolled devices have WSUS configured (via Group Policy or Intune Settings Catalog), Autopatch cannot manage their updates — devices will only see WSUS-approved updates, not Windows Update directly.

Fix: For Autopatch-enrolled devices, ensure the Windows Update source is set to Windows Update (the internet) — not WSUS. In Group Policy, remove any "Specify intranet Microsoft Update service location" setting for these devices. In Intune Settings Catalog, ensure no Update Source or WSUS server is configured for the Autopatch device groups.

🎤 Interview Questions — Topic 83
What are the key prerequisites for enrolling devices in Windows Autopatch?
Devices must be Azure AD Joined or Hybrid Azure AD Joined, enrolled in Microsoft Intune MDM, running Windows 10 1809 or later (Enterprise or Professional edition — Home not supported), and licensed with M365 E3/E5, Windows Enterprise E3/E5, or M365 Business Premium. The Readiness Assessment automatically evaluates the environment against all Autopatch prerequisites — license validity, Azure AD configuration, Intune enrollment health, existing Update Ring conflicts, WSUS conflicts, network connectivity to Windows Update, and tenant configuration. Running it before enrollment surfaces blockers that would cause Autopatch to malfunction silently after enrollment — like conflicting Update Rings that would fight with Autopatch's own rings. Fixing blockers first ensures a clean, functional Autopatch deployment.
TOPIC 84
Monitoring Autopatch — Reports, Alerts & Service Health
Staying informed without micromanaging — the Autopatch monitoring toolkit
The Monitoring Philosophy

The promise of Autopatch is that you don't need to actively monitor patch deployment — Autopatch monitors itself and alerts you when action is needed. Your role shifts from active manager (checking every patch cycle) to exception handler (reviewing alerts and approving decisions). Understanding the monitoring tools lets you trust Autopatch while staying informed.

Key Reports in Autopatch
📊
Windows Quality Update Report
The primary report — shows deployment status for the current month's quality update across all four rings. Per-ring breakdown: how many devices are Scheduled, In Progress, Up to date, or Failed. Track progress toward the 21-day SLO.
📅
Feature Update Report
Tracks Windows feature update (version upgrade) deployment progress. Shows which ring is being deployed, current success rate, devices pending, and any compatibility blocks preventing upgrade.
🩺
Device Registration Report
Health status of enrolled devices — Registered (healthy), Not Ready (prerequisites not met), Inactive (not checking in), or Paused (manually or automatically paused). Your daily enrollment health dashboard.
📱
Microsoft 365 Apps Update Report
Deployment status of M365 Apps (Office) updates across rings. Version breakdown showing which devices are on which Office build and how many are current.
🌐
Edge Update Report
Microsoft Edge version distribution across enrolled devices. Shows current version compliance and how quickly the fleet moved to the latest Edge build after Autopatch deployed it.
⚠️
Windows Quality Update Alerts
Active alerts about problematic updates — paused deployments, devices failing to update, policy conflicts detected, devices not meeting the 21-day SLO threshold. Action items that need admin attention.

Service Health — The Alert Feed

Service Health (also called Windows Autopatch Service Health in the Intune portal) is where Autopatch communicates issues that need your attention. Think of it as Autopatch's communication channel to IT:

  • Advisory: Low-priority informational items — minor deployment delays, devices at risk of missing SLO, optimization suggestions.
  • Incident: Active issues requiring IT attention — automatic pause engaged, high failure rate detected, Microsoft asking IT to approve a remediation action.
  • Post-Incident Review: After a resolved incident, a summary of what happened, why, and what changed.

Access via: Intune → Reports → Windows Updates → Windows Autopatch → Service incidents and advisories

Setting Up Autopatch Notifications

Don't miss critical Autopatch events — configure notifications so the right people are alerted:

  1. 1
    In the Autopatch Admin Center: Tenant Administration → Windows Autopatch → Tenant management
  2. 2
    Add Admin contact emails — these receive Autopatch service health notifications, incident reports, and advisory emails directly.
  3. 3
    Optionally set up Azure Monitor Alerts → route Intune Autopatch events to a Microsoft Teams channel or email distribution list via Logic App or Action Group.
The SLO Dashboard — 21-Day Target

Autopatch's quality update SLO targets 95% of devices updated within 21 days of release. You can see your progress toward this target in the Quality Update report:

  • Green zone: Current patch cycle coverage above 95% — SLO met.
  • Yellow zone: 85–95% coverage — SLO at risk. Review devices in "Not Ready" or "Failed" state.
  • Red zone: Below 85% — SLO breach. Autopatch will flag this and may create a Service Incident asking IT to investigate the gap.
📊
Weekly Autopatch Review — What to Check

Recommended weekly review (15 minutes):
1. Device Registration Report: Any new "Not Ready" devices? Investigate and remediate.
2. Quality Update Report: What's the current month's patch coverage percentage? Are we trending toward the 21-day SLO?
3. Alerts/Service Health: Any active incidents or advisories? Any paused deployments needing admin approval to resume?
4. Failed devices count: More than 2–3% failure rate? Investigate the common error — may indicate a driver conflict or environmental issue specific to your fleet.
5. Inactive devices: Devices that haven't checked in for 7+ days. Reach out to users or escalate to asset management.
This 15-minute review replaces the 8-20 hours of traditional patch management work.

🎤 Interview Questions — Topic 84
What does the "Not Ready" device status mean in Autopatch and how do you resolve it?
Not Ready means a device has a pre-enrollment or post-enrollment readiness issue preventing Autopatch from managing its updates. Common causes: WSUS configured on the device (overrides Windows Update), conflicting Intune Update Ring policy still assigned, device doesn't meet OS version requirements, Windows Diagnostic Data not set to Enhanced/Full, or a licensing issue. Resolution: click into the specific "Not Ready" device to see which readiness check is failing, then remediate that specific condition — remove the WSUS setting, remove the conflicting Update Ring policy, or fix the telemetry setting.
What is the Autopatch quality update SLO and where do you monitor progress toward it?
The SLO targets 95% of enrolled devices having the latest quality update installed within 21 days of its release. Monitor progress in Intune → Reports → Windows Updates → Windows Autopatch → Quality updates report. This shows per-ring deployment progress, the percentage of devices currently "Up to Date," and highlights if the fleet is on track. Service Health alerts fire if progress falls below the threshold, requiring admin investigation into failed or inactive devices.
TOPIC 85
Autopatch vs Windows Update for Business (WUfB)
Two approaches to patch management — when to use each
Understanding Both Tools

Windows Update for Business (WUfB) is the traditional Intune-based approach to managing Windows Updates — you configure Update Rings (deferral periods, active hours, restart behavior) via Intune policies, and devices follow those configurations. IT controls and monitors everything manually.

Windows Autopatch builds on top of WUfB and takes that management away from IT — Microsoft handles the ring configuration, timing, monitoring, and remediation automatically.

🤖 Windows Autopatch
  • Microsoft manages ring timing and deferral
  • Automatic pause when failure threshold exceeded
  • Microsoft-backed SLO (95% in 21 days)
  • Manages Windows + M365 Apps + Edge + Teams
  • Minimal IT configuration after enrollment
  • Less control over exact timing
  • Requires M365 E3+ license
  • Requires Windows Enterprise/Pro (not Home)
  • Readiness requirements must be met
  • IT monitors alerts + handles exceptions
⚙️ Windows Update for Business
  • IT fully controls ring timing and deferral
  • Manual pause — IT monitors and decides
  • No SLO — IT-defined timelines
  • Windows updates only (M365/Edge separate)
  • Requires creating and maintaining Update Rings
  • Full control over exact scheduling
  • Any Intune license supports it
  • Works on all Windows editions with Intune
  • No additional readiness requirements
  • IT monitors and manages actively

When to Choose Autopatch
  • IT team is small and patch management overhead is significant
  • Security/compliance requires a documented, time-bound patching SLA
  • Cyber insurance or audit requirements demand evidence of systematic patching
  • You have M365 E3+ licensing (Autopatch included at no additional cost)
  • You want unified management of Windows + M365 Apps + Edge + Teams updates together
  • You trust Microsoft to make good ring timing decisions for your fleet
When to Keep WUfB
  • You need precise control over exactly when updates deploy (regulated environments, trading floors, manufacturing lines)
  • You use WSUS and cannot move to direct Windows Update internet access
  • You're on M365 Business Basic or licensing that doesn't include Autopatch
  • You have specific Windows Home devices in your fleet that Autopatch can't manage
  • You need custom deferral timing — e.g., 60-day deferral for critical infrastructure that Autopatch's fixed rings don't accommodate
  • Your compliance framework requires specific patch approval processes (change management, CAB approval before deployment)
Can You Use Both?

Yes — for different device populations. Some organizations use Autopatch for their standard employee laptop fleet while using WUfB manually for specialized devices (kiosks, OT-adjacent PCs, executive devices needing special care). The key is ensuring each device only has ONE update management approach — a device enrolled in Autopatch must not also have conflicting WUfB Update Ring policies assigned.

💡
Interview Positioning: When asked about patch management in an Intune interview, demonstrate awareness of BOTH approaches. Show you understand WUfB (which you've likely used if working with Intune for any time) AND Autopatch (which is the modern direction). Be able to articulate when you'd choose one over the other. This shows depth — not just knowing the tools exist, but understanding the decision criteria for each.

Real-World Decision — Autopatch or WUfB?

Company A — 500-person SaaS company, M365 E5 licenses, 3-person IT team:
Decision: Autopatch. Small IT team can't afford 15 hours/month on patching. M365 E5 includes Autopatch at no extra cost. Cyber insurance requires documented patching SLA — Autopatch provides it. No WSUS. All devices Azure AD Joined. Clear choice.

Company B — Manufacturing company, 2,000 devices, mix of production floor PCs and office PCs, WSUS infrastructure in place:
Decision: WUfB for production floor, evaluate Autopatch for office PCs. Production floor PCs cannot be updated during production hours — need precise scheduling control that Autopatch's fixed rings can't provide. WSUS is used for offline manufacturing environment. Office PCs could move to Autopatch. Hybrid approach.

Company C — Bank, 1,500 devices, change management process requiring CAB approval before patches deploy:
Decision: WUfB with manual rings. Banking CAB process requires human approval before patches deploy. Autopatch's automated deployment violates the change management policy. IT uses WUfB with extended deferrals to give CAB time to approve before each ring deployment.

🎤 Interview Questions — Topic 85
What is the key difference between Windows Update for Business and Windows Autopatch?
Windows Update for Business is IT-managed — you configure Update Ring deferral periods, active hours, and restart policies via Intune, and IT monitors and manages the entire patch cycle manually. Windows Autopatch takes that management off IT — Microsoft manages the ring structure, timing, monitors for failures, automatically pauses on elevated failure rates, and alerts IT only when exceptions need attention. WUfB gives IT full control; Autopatch gives IT freedom from patch management overhead in exchange for less control over exact timing.
Can you use both Autopatch and WUfB in the same Intune tenant?
Yes, for different device populations. Some devices can be enrolled in Autopatch (standard employee laptops) while others remain on manual WUfB Update Rings (specialized devices, production floor PCs). The critical rule is that each device can only have one approach — a device enrolled in Autopatch must not have conflicting WUfB Update Ring policies applied to it. Separate the device populations via Azure AD groups and ensure Autopatch-enrolled devices are excluded from WUfB Update Ring assignments.
A security auditor asks for evidence that your organization patches within 30 days of release. Would you choose Autopatch or WUfB, and why?
Autopatch, if licensing allows. Autopatch provides a Microsoft-backed SLO of 95% of devices updated within 21 days — better than the 30-day requirement — with built-in reporting you can export as evidence. The Quality Update report shows percentage of devices updated with timestamps you can share with auditors. With WUfB, you'd need to configure rings to achieve similar timing and manually pull compliance reports from Windows Update Compliance. Autopatch provides both the enforcement mechanism and the documentation in one managed service.
What is the main reason a company might NOT be able to use Windows Autopatch?
Several blockers can prevent Autopatch adoption: using WSUS as the update source (Autopatch requires direct Windows Update access), having a strict change management process requiring human CAB approval before patches deploy (Autopatch automates this), licensing below M365 E3 (Autopatch requires E3, E5, or Windows E3/E5), needing custom deferral periods beyond Autopatch's 9-day maximum for Broad ring, or devices running Windows Home edition (not supported). For any of these scenarios, Windows Update for Business remains the appropriate approach.
