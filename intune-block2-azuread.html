TOPIC 09
Azure AD vs Active Directory
Two different technologies — often confused, always critical to distinguish
The Confusion — Why People Mix Them Up

Both are called "Active Directory." Both store user accounts. Both are made by Microsoft. But they are fundamentally different technologies built for different eras. Mixing them up in an interview is a red flag for hiring managers. Let's make sure you never confuse them again.

🏢
Active Directory (AD DS)
On-premises directory service. Lives on your Windows Server inside your office/data center. Created in 1999. Uses LDAP and Kerberos protocols. Manages domain-joined Windows computers and users inside a corporate network.
☁️
Azure Active Directory (Azure AD / Entra ID)
Cloud-based identity service. Lives in Microsoft's Azure cloud. Created in 2010. Uses OAuth 2.0, OpenID Connect, SAML protocols. Manages users and devices accessing cloud services like Microsoft 365, Intune, Salesforce, etc.

The Analogy That Makes It Click
💡 Think of It This Way

Active Directory is like your office building's security system — it works great when you're inside the building (on the corporate network). Swipe your badge (domain join), you get in. But if you're working from home, the badge doesn't work — you need a VPN tunnel back to the building first.

Azure AD is like a cloud security service — it works from anywhere in the world. Your identity is verified in the cloud, and you can access cloud resources from home, a café, or another country — no VPN needed.

The Rebranding — Entra ID

In 2023, Microsoft renamed Azure Active Directory to Microsoft Entra ID. The functionality is identical — it's just a new name. In the real world, both terms are used interchangeably. The portal is now called Entra admin center (entra.microsoft.com). Intune documentation uses both names. Know both.

Why Does This Matter for Intune?
  • Intune uses Azure AD users and groups to target policies. It does NOT read from on-prem AD directly.
  • Devices must have an Azure AD identity to be enrolled in Intune — even if they're also in on-prem AD.
  • Conditional Access is an Azure AD feature — not an Intune feature. Intune feeds compliance data TO Azure AD.
  • There are no OUs in Azure AD — you cannot use GPO-style "apply this policy to this OU" targeting. Instead you target Azure AD groups.
  • Dynamic Groups (Azure AD P1) replace OUs in many scenarios — auto-add devices to groups based on attributes.
🏭
Real-World Scenario

Problem: An L1 admin is trying to apply an Intune policy to a specific OU in Active Directory (like they used to do with GPO). They can't find OUs in Intune.
Explanation: Intune doesn't know about OUs. OUs are an on-prem AD concept. In Intune, targeting is done through Azure AD Groups. The solution is to create an Azure AD Group (static or dynamic), add the relevant users/devices, and assign the Intune policy to that group. For hybrid environments, Azure AD Connect can sync users from specific OUs, and you can then create Azure AD groups mirroring those OU members.
Key Takeaway: OUs → Replace with Azure AD Groups when working with Intune.

🎤 Interview Questions — Topic 9
What is the difference between Active Directory and Azure Active Directory?
Active Directory is an on-premises directory service using Kerberos/LDAP, designed for managing domain-joined Windows machines in a corporate network. Azure AD is a cloud-based identity service using OAuth 2.0 and SAML, designed for managing access to cloud applications like Microsoft 365 and Intune. They serve different purposes and are NOT just "cloud version of AD."
Does Intune use Active Directory or Azure AD?
Intune uses Azure AD. All users, groups, and device identities that Intune references are in Azure AD. In hybrid environments, users are synced FROM on-prem AD TO Azure AD via Azure AD Connect — but Intune itself only reads from Azure AD.
Why can't I use OUs to target Intune policies?
OUs are an Active Directory concept and don't exist in Azure AD. Intune policy targeting works through Azure AD Groups. To replicate OU-based targeting, create Azure AD groups and assign policies to those groups. Dynamic groups can automate membership based on device or user attributes, similar to how OUs organize objects in AD.
TOPIC 10
Azure AD Join vs Hybrid Azure AD Join vs Workplace Join
The Three Device Registration States — Critical for Intune Enrollment
Why Device States Matter

Before a device can be managed by Intune, it needs an identity in Azure AD. How that identity is established determines what the device can do, how it authenticates, and what management options are available. There are three device states — and knowing when each applies is a critical skill for any Intune admin.

🔷 Azure AD Registered
Personal/BYOD devices. User registers their personal phone or laptop to access company resources. Company has minimal control. Used for MAM-WE scenarios. Device is NOT managed — only the user account is known.
🟣 Azure AD Joined
Company-owned device, cloud-only. No on-prem domain. Device joins Azure AD directly. Fully managed by Intune. Best for modern, cloud-native environments. No domain controller required.
🟢 Hybrid Azure AD Joined
Company-owned device, joined to BOTH on-prem AD AND Azure AD. Best for hybrid orgs still using SCCM + GPO + Intune together. More complex setup. Required for Co-management scenarios.

How Azure AD Joined Works (Step by Step)
  1. 1
    New company laptop is powered on for the first time (or after Autopilot reset).
  2. 2
    During OOBE (Out-of-Box Experience), user selects "Set up for an organization."
  3. 3
    User enters their Azure AD work account (name@contoso.com) and password + MFA.
  4. 4
    Windows contacts Azure AD, authenticates, and joins the device to Azure AD. A device object is created in Azure AD.
  5. 5
    Azure AD detects that this user/device should be MDM-managed (MDM scope is configured), and automatically triggers Intune enrollment.
  6. 6
    Intune pushes configuration profiles, apps, compliance policies to the device. No IT admin needed to touch the machine.
How Hybrid Azure AD Join Works
  1. 1
    Device is joined to on-prem Active Directory in the traditional way (IT manually joins it or it's imaged by SCCM).
  2. 2
    Azure AD Connect syncs the device computer account from on-prem AD to Azure AD (Device Writeback or device sync).
  3. 3
    Device runs an auto-registration task that registers itself with Azure AD, creating a device object in the cloud.
  4. 4
    Device is now joined to BOTH systems — on-prem AD (for GPO, Kerberos, SCCM) AND Azure AD (for Intune, Conditional Access, cloud SSO).
  5. 5
    Co-management can now be enabled — SCCM and Intune manage the device simultaneously.
⚠️ Common Admin Mistake

Many admins try to enable Intune for devices that are only on-prem AD joined (no Azure AD registration at all). This will NOT work. Intune requires the device to have an Azure AD identity. The fix is to either Hybrid Azure AD Join the device (register it with Azure AD while keeping the on-prem domain join) or re-provision it as Azure AD Joined.

🏦
Real-World Decision — Which Join Type to Choose?

Scenario A — New startup, no on-prem infrastructure: Azure AD Joined. Cloud-only. Fast to set up. Autopilot handles provisioning. Intune manages everything. Users sign in with Azure AD credentials.

Scenario B — Bank with 10,000 existing domain-joined PCs, can't migrate overnight: Hybrid Azure AD Join. Keep existing on-prem AD, GPO, SCCM. Add Azure AD identity so Intune can co-manage. Gradual migration to cloud over 2–3 years.

Scenario C — Employee's personal iPhone accessing company email: Azure AD Registered (BYOD). Employee registers their iPhone in Azure AD. MAM policies applied to Outlook/Teams apps. Device not domain joined or MDM enrolled. Company controls only company data in apps.

🎤 Interview Questions — Topic 10
What is the difference between Azure AD Joined and Hybrid Azure AD Joined?
Azure AD Joined devices only exist in Azure AD — no on-prem domain join. Best for cloud-native environments. Hybrid Azure AD Joined devices are joined to both on-prem Active Directory AND Azure AD. This is for organizations that still have on-premises infrastructure and need the device to receive GPO from on-prem DC while also being manageable by Intune via Azure AD.
What is Azure AD Registered and when is it used?
Azure AD Registered is for personal (BYOD) devices. The user registers their personal device with Azure AD so they can access company resources, but the company doesn't manage the full device. It's typically paired with MAM-WE — only company app data is controlled, not the entire device.
A user's device shows "on-prem AD joined only" and isn't enrolling in Intune. What's the issue?
The device has no Azure AD identity. For Intune enrollment, the device must be registered in Azure AD — either as Azure AD Joined, Hybrid Azure AD Joined, or Azure AD Registered. The fix is to configure Hybrid Azure AD Join via Azure AD Connect, or re-provision the device as Azure AD Joined.
TOPIC 11
Azure AD Users & Groups — Static vs Dynamic
How Intune Policy Targeting Works — Groups Are Everything
Why Groups Are the Core of Intune

In Intune, almost everything is assigned to a group. Configuration profiles, compliance policies, apps, scripts — all targeted at Azure AD groups. Understanding groups is not optional — it's the primary mechanism for making Intune work in any environment.

Types of Azure AD Groups
👥
Security Groups
Used to assign access to resources, Intune policies, apps, and licenses. This is the primary group type you'll use in Intune. Can contain users, devices, or both.
📧
Microsoft 365 Groups
Used for collaboration — Teams, SharePoint, shared mailbox. Can be used for Intune targeting too, but Security Groups are preferred for policy assignment. Cannot contain devices.

✅ Best Practice — User Groups vs Device Groups

When assigning a policy to user groups, the policy follows the USER regardless of which device they log into. If the policy is sensitive to the device's hardware (e.g., BitLocker, which needs a TPM chip), use device groups instead. For app deployment to all company devices regardless of who logs in — device groups. For per-user settings like email profile — user groups.

All Users and All Devices — Built-in Groups

Intune has two special virtual groups you'll use constantly:

  • All Users: Every user in your Azure AD tenant. Useful for company-wide policies. Be careful — applies to every single user including admins.
  • All Devices: Every device enrolled in Intune. Useful for baseline policies that every managed device should get.
💡
Pro Tip: Use "All Devices" for your baseline security configuration (e.g., require password, enable firewall). Then use specific groups for role-based or department-based policies. The layered approach gives you a solid security floor while allowing customization on top. Always test new policies on a pilot group before rolling to "All Devices."
Include vs Exclude in Policy Assignment

When assigning a policy in Intune, you can both include and exclude groups. This is powerful for handling exceptions:

⚙️
Include/Exclude Real Example

Scenario: You want to push a strict password policy to all employees — EXCEPT the IT Admins group (because they use different authentication methods like FIDO2 security keys).
Assignment: Include: "All Users" / Exclude: "IT Admins"
The policy applies to everyone EXCEPT the IT Admins group. Clean, simple, maintainable.

Important rule: Exclude always wins over Include. If a user is in both the included group and excluded group, they are excluded from the policy.

🎤 Interview Questions — Topic 11
What license is required for Dynamic Groups in Azure AD?
Azure AD Premium P1 or higher. Dynamic groups are not available in the free Azure AD tier. They're included in Microsoft 365 Business Premium, EMS E3, M365 E3, and above.
When would you use a device group vs a user group for Intune policy assignment?
Use device groups when the policy is specific to hardware capabilities (like BitLocker needing a TPM) or when you want the policy to apply regardless of who is logged in (like a shared device kiosk). Use user groups when the policy follows the user across devices — like email profiles or VPN configuration tied to a user's role.
Write a dynamic group rule that targets all iOS devices in Intune.
The rule would be: (device.deviceOSType -eq "iPhone") or (device.deviceOSType -eq "iPad"). This creates a device group that automatically includes any enrolled Apple mobile device. You could also use device.operatingSystem -eq "iOS" depending on the attribute populated during enrollment.
If a user is in both an included and excluded group for an Intune policy, which wins?
Exclude always wins. If a user or device is in both an included group and an excluded group for a policy, the exclusion takes precedence and the policy does not apply to them.

Azure AD Registered vs Azure AD Joined Devices
Device Identity Deep Dive — How Intune Sees Your Devices
How to Check a Device's State

You can verify a device's Azure AD state in multiple ways:

  • On the device itself (Windows): Run dsregcmd /status in Command Prompt. Look at AzureAdJoined, WorkplaceJoined, and DomainJoined fields.
  • In Azure AD portal: Go to entra.microsoft.com → Devices → All devices. See each device's "Join type" column.
  • In Intune portal: Devices → All devices → Click a device → See "Azure AD device ID" and join type in properties.

🔑 The Chain

A device must be registered in Azure AD → then enrolled in Intune → then evaluated against a compliance policy → then the compliance status is passed to Conditional Access → which decides if the device can access company resources. Each step depends on the previous one.

What Happens When a Device is Enrolled
  1. 1
    Device registers in Azure AD — a device object is created with a unique Azure AD Device ID.
  2. 2
    An MDM enrollment certificate is installed on the device. This is how Intune authenticates communication with the device.
  3. 3
    The Intune Management Extension (IME) is installed on Windows devices — this enables advanced features like PowerShell scripts, Win32 apps, and Proactive Remediation.
  4. 4
    Device checks in with Intune. Assigned policies, apps, and compliance rules are received and applied.
  5. 5
    Device reports back its state — hardware info, installed apps, OS version, compliance status — all visible in Intune portal.
🎤 Interview Questions — Topic 12
How do you check if a Windows device is Azure AD Joined?
Run dsregcmd /status in Command Prompt. Look for AzureAdJoined: YES. If DomainJoined is also YES, it's Hybrid Azure AD Joined. If AzureAdJoined is NO but WorkplaceJoined is YES, it's Azure AD Registered (BYOD).
What is the Intune Management Extension and why is it important?
The Intune Management Extension (IME) is an agent installed on Windows devices enrolled in Intune. It enables features that the standard MDM channel can't handle — Win32 app deployment, PowerShell script execution, and Proactive Remediation. Many Intune features require IME to be installed and running. You can check its log at C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\.
TOPIC 13
Conditional Access — What It Is and Why It Matters
The Zero Trust Gatekeeper — Intune's Most Powerful Integration
The Problem Conditional Access Solves

In the old world, being on the corporate network meant you were trusted. VPN in → access everything. But modern work has broken this model. Users work from home, from cafés, on personal devices, on unmanaged phones. You can no longer use network location as a signal of trust.

Conditional Access replaces network-based trust with identity + device + context based trust. Every access request is evaluated: WHO is asking? From WHERE? On WHAT device? Is it healthy? Only then is access granted — or denied, or step-up authentication required.

Grant Controls — What Access Controls You Can Apply
  • Block access: Completely deny. User gets an error page saying access is blocked.
  • Require MFA: User must complete MFA before access is granted.
  • Require device to be marked compliant: Device must pass Intune compliance policy. Most important for Intune integration.
  • Require Hybrid Azure AD Joined device: Only devices in both on-prem AD and Azure AD are allowed.
  • Require approved client app: Only Microsoft apps (Outlook, Teams) — not generic browsers.
  • Require app protection policy: MAM policy must be applied to the app. For BYOD scenarios.
  • Require password change: Force the user to reset their compromised password.
Real Policy Examples — What You'll Build at Work
🔐
Policy 1 — Require Compliant Device for Exchange Online

IF: User is in "All Employees" group AND accessing Exchange Online
THEN: Require device to be marked compliant (by Intune)
Result: Only Intune-enrolled, compliant devices can check company email. An unmanaged personal PC or non-enrolled phone gets blocked — even with valid credentials.

🌏
Policy 2 — Block Access from Outside India

IF: User is in "All Employees" group AND accessing All Cloud Apps AND location is NOT "India Named Location" (defined IP range)
THEN: Block access
Result: No one can sign in from outside India. Prevents credential stuffing from foreign IPs. Exception: Create a separate policy for executives who travel.

🔑
Policy 3 — Require MFA for Admin Accounts Always

IF: User is in "Admin Accounts" group AND accessing Any Cloud App AND Any Location
THEN: Require MFA
Result: Every admin login — even from the office — requires MFA. Critical because admin accounts are prime targets. No exceptions, no bypasses.

Report-Only Mode — Your Best Friend

Never deploy a new Conditional Access policy in enforcement mode on day one. Use Report-Only mode first. In Report-Only, the policy evaluates against real users and logs what WOULD have happened — without actually blocking anyone. Review the logs in Azure AD Sign-in logs to see who would be affected. Only switch to enforce after you're confident in the impact.

🚨 Real Incident — CA Policy Gone Wrong

An admin created a Conditional Access policy requiring MFA for All Users accessing All Apps — and forgot to exclude the Emergency Access (Break Glass) accounts. Then MFA infrastructure had an outage. Result: ALL users including all admins were locked out. The break-glass account ALSO required MFA — which was down. Complete tenant lockout for 4 hours.

Always: Exclude at least two Break Glass accounts from ALL Conditional Access policies. These accounts are your last resort if something goes wrong.

🎤 Interview Questions — Topic 13
What is Conditional Access and how does it relate to Intune?
Conditional Access is an Azure AD feature that acts as a policy engine — evaluating every access request based on signals like user identity, device compliance, location, and app being accessed. Intune provides the device compliance signal. When a device is marked compliant by Intune, Conditional Access can use that to grant or block access to resources like Exchange, SharePoint, and Teams.
What is Report-Only mode in Conditional Access?
Report-Only mode evaluates a Conditional Access policy against real sign-ins and logs what would have happened — without actually enforcing the policy. It's used to test new policies safely before enabling them, allowing you to see the potential impact (who would be blocked, who would need MFA) without disrupting users.
What are Break Glass accounts and why must they be excluded from CA policies?
Break Glass accounts are emergency administrator accounts that bypass normal Conditional Access policies. They must be excluded because if MFA infrastructure fails, or if a CA policy accidentally locks out all admins, Break Glass accounts are the last resort to recover the tenant. Without them, a misconfigured CA policy could cause a complete tenant lockout.
What happens when a non-compliant device tries to access Exchange Online with a CA policy requiring compliance?
The user is blocked and shown an error page saying they need to enroll/fix their device. Azure AD Sign-in logs record the blocked attempt showing the failure reason as "Device not compliant." The user must enroll in Intune and resolve compliance issues before access is granted.
TOPIC 14
MFA Setup and Enforcement
Multi-Factor Authentication — Setup, Methods, and Enforcement via Intune & Azure AD
Why MFA Is Non-Negotiable

According to Microsoft, 99.9% of compromised accounts didn't have MFA enabled. Password-only authentication is a solved problem for attackers — credential stuffing, phishing, and password spray attacks make single-factor auth effectively useless as a security measure. MFA is the single highest-impact security control you can implement.

Three Ways to Enforce MFA in Microsoft 365
🔐
1. Security Defaults
Microsoft's one-click baseline security. Requires MFA for all users, blocks legacy auth, requires MFA for admins. Free. No customization. Good for very small orgs with no IT team. Disable this if using Conditional Access.
📋
2. Conditional Access (Recommended)
Create CA policies that require MFA under specific conditions. Full control — require MFA from outside office, for specific apps, for all admins. Requires Azure AD P1. This is what enterprises use.
👤
3. Per-User MFA (Legacy)
Enable MFA on a per-user basis in the old Microsoft 365 admin portal. Legacy method. Not recommended — no conditional logic, all-or-nothing per user. Only for backward compat scenarios.

  2. 2

  3. 3

MFA Fatigue Attack — What It Is and How to Stop It
🚨 MFA Fatigue Attack

Attacker gets the user's password via phishing. Then bombards the user's phone with push notification MFA requests — sometimes 50-100 in a row, at 2am. The exhausted/confused user eventually taps "Approve" just to make it stop.

Prevention: Enable Number Matching in Microsoft Authenticator — the user must enter the number shown on the sign-in screen into the app. An attacker's push won't show the right number. Also enable Additional Context (shows app name and location in the push). Both are now on by default in new tenants.

📱
Real-World MFA Registration Campaign

Scenario: A company of 500 users needs to enable MFA. Just turning on a CA policy requiring MFA immediately would block everyone who hasn't registered yet.
Proper approach:
Week 1: Send communication to all users explaining that MFA is coming. Provide instructions to install Microsoft Authenticator.
Week 2: Create a CA policy in Report-Only mode — see who hasn't registered yet. Follow up with those users.
Week 3: Use "Microsoft Managed" MFA registration campaign in Azure AD to nudge users during sign-in to register.
Week 4: Switch CA policy to Enforce. Set grace period of 14 days for users who haven't registered (they can skip but get reminded).
Week 5+: Monitor for blocked sign-ins. Help stragglers register. MFA now active for all 500 users with minimal disruption.

🎤 Interview Questions — Topic 14
What is the difference between Security Defaults and Conditional Access for MFA enforcement?
Security Defaults is a one-click setting that applies Microsoft's baseline MFA requirements with no customization — all users, all apps, always. Conditional Access gives granular control — you can require MFA only for specific apps, from specific locations, for specific groups. If an organization needs any customization (which most do), Conditional Access is the right choice and Security Defaults should be disabled.
What is an MFA fatigue attack and how does Microsoft Authenticator's Number Matching prevent it?
An MFA fatigue attack is when an attacker who has stolen a password sends repeated MFA push notifications hoping the user approves one by mistake. Number Matching requires the user to type the specific number shown on the sign-in screen into the Authenticator app. An attacker's push doesn't display the correct number, so even if sent, the user won't be able to approve it without seeing the legitimate sign-in screen.
Which MFA method is the most secure?
FIDO2 security keys and Windows Hello for Business are the most secure because they are phishing-resistant — the private key never leaves the device and cannot be intercepted by a phishing site. TOTP and push notifications can be intercepted by real-time phishing (adversary-in-the-middle) attacks.

Common License-Related Troubleshooting Scenarios
⚠️ Scenario 1: Dynamic Group Won't Work

Symptom: Admin creates a dynamic group but membership shows 0 even though devices exist.
Root Cause: The admin's Azure AD tenant only has Free tier — no P1 license for Dynamic Groups.
Fix: Assign at least one Azure AD P1 license to the tenant. Dynamic group evaluation requires P1.

⚠️ Scenario 2: Conditional Access Policy Not Evaluating

Symptom: CA policy exists but users aren't being prompted for MFA.
Root Cause: Users don't have Azure AD P1 license assigned. CA requires P1 per user.
Fix: Assign Azure AD P1 (or M365 Business Premium/E3) licenses to affected users.

⚠️ Scenario 3: Risk-Based CA Policy Not Available

Symptom: Admin tries to add Sign-in Risk as a CA condition but the option is grayed out.
Root Cause: Risk-based conditions require Azure AD Identity Protection — only in P2 / M365 E5.
Fix: Upgrade to Azure AD P2, or use standard Conditional Access without risk conditions (P1 sufficient).

  1. 1
    Go to entra.microsoft.com → Users → All Users

    Search for the user → Click their name
  3. 3
    Click Licenses in the left menu of the user profile
  4. 4
    Review what is assigned. Expand each license to see individual service plans (e.g., expand M365 E3 to see if Intune is enabled within that license).
💡
Pro Tip: Sometimes a license is assigned but a specific service WITHIN the license is disabled. For example, an M365 E3 license is assigned, but the Intune service plan within it is turned off for that user. Always expand the license in Azure AD to check individual service plan states — not just the top-level license.
License Assignment Best Practices
  • Use Group-Based Licensing (P1 feature): Assign licenses to an Azure AD group. Every member automatically gets the license. Leavers who are removed from the group lose the license automatically. Far better than manual assignment.
  • Audit license usage monthly: In Microsoft 365 admin center, check how many licenses are assigned vs purchased. Unused licenses cost money.
  • Don't assign P2 to everyone by default: P2 is significantly more expensive. Only users who need PIM, Access Reviews, or Risk-based CA need P2. Everyone else needs only P1.
  • Frontline workers (Deskless workers): Consider Microsoft 365 F1 or F3 licenses — cheaper, includes Intune, suitable for factory floor / retail scenarios.
🎤 Interview Questions — Topic 15
What is the minimum license required to use Conditional Access in Azure AD?
Azure AD Premium P1 is required for Conditional Access. This is included in Microsoft 365 Business Premium, M365 E3, EMS E3, and above. Basic Security Defaults (which applies some MFA rules without customization) are free, but proper Conditional Access policies require P1.
What features require Azure AD P2 specifically?
Azure AD P2 adds Identity Protection (user risk and sign-in risk signals), Privileged Identity Management (PIM for just-in-time admin access), Access Reviews, and Entitlement Management. Any risk-based Conditional Access condition also requires P2 because it uses Identity Protection data.
A user has an M365 E3 license but can't enroll their device in Intune. What do you check?
I'd check the user's license details in Azure AD — expand their M365 E3 license and look at individual service plans. The Intune service plan within E3 may be disabled for this specific user. I'd also check if they're within the MDM user scope configuration in Azure AD (Mobility settings). And verify they're not hitting an enrollment restriction policy in Intune.
