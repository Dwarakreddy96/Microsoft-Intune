TOPIC 43
What Is Compliance in Intune?
Purpose, workflow, and how compliance fits into the Zero Trust model
The Core Purpose

A Compliance Policy in Intune is an evaluation engine — it checks whether an enrolled device meets a set of security requirements you define, then marks the device as Compliant or Non-Compliant. It doesn't configure anything. It evaluates and reports. The power comes from what happens based on that result.

Combined with Conditional Access, compliance becomes a gatekeeper: only compliant devices can access Exchange Online, SharePoint, Teams, and other corporate resources. Non-compliant devices are blocked at the door — regardless of whether the user has valid credentials.

Compliance Statuses — What Each Means
✅ Compliant
Device meets ALL requirements in the compliance policy. Conditional Access grants access. Device is healthy and policy-adherent.
❌ Non-Compliant
Device fails one or more requirements. CA can block access. Actions for non-compliance kick in (notifications, remote wipe after X days).
⏳ In Grace Period
Device is non-compliant but within the admin-configured grace period. Access may still be allowed temporarily while user fixes the issue.
🔵 Not Evaluated
No compliance policy is assigned to this device. Intune treats it as non-compliant by default. Important for Conditional Access design.
🚨 Critical: Devices With No Compliance Policy Are Non-Compliant

If a device has no compliance policy assigned, Azure AD treats it as non-compliant by default. This means if your Conditional Access policy requires device compliance, devices without a compliance policy assigned will be BLOCKED. Always assign a baseline compliance policy to All Devices to avoid unintended lockouts when enabling Conditional Access.

Compliance Within Zero Trust

Compliance is the device health signal in Microsoft's Zero Trust framework. Zero Trust says "never trust, always verify" — even if a user has valid credentials and MFA, access should still be conditional on the device being healthy. Compliance policies provide that device health signal to Conditional Access, completing the Zero Trust verification loop.

  • Identity verified: Azure AD + MFA confirms WHO is signing in.
  • Device verified: Intune compliance confirms the device is HEALTHY.
  • Access granted: Only when BOTH identity and device are verified.
🎤 Interview Questions — Topic 43
What is the difference between a Compliance Policy and a Configuration Profile?
A Configuration Profile actively pushes and applies settings to a device — it configures the device. A Compliance Policy passively evaluates whether the device already meets security requirements — it checks and reports. Config profiles change device state; compliance policies read device state. Non-compliant devices can then be blocked from corporate resources via Conditional Access, which config profiles cannot do on their own.
What happens to a device in Intune that has no compliance policy assigned?
Azure AD marks it as non-compliant by default. If a Conditional Access policy requires device compliance, that device will be blocked from accessing Exchange, SharePoint, Teams, etc. — even if the device is actually secure. Best practice is to always assign a baseline compliance policy to All Devices to ensure every enrolled device has a compliance evaluation.
How does Intune compliance fit into the Zero Trust security model?
Zero Trust requires verifying both identity and device health for every access request. Intune compliance provides the device health signal — is the device encrypted? Is the OS patched? Is it jailbroken? Azure AD Conditional Access combines this device health signal with the identity verification (MFA, risk level) to make an access decision. Together they implement the "verify explicitly, use least privilege, assume breach" principles of Zero Trust.

🎤 Interview Questions — Topic 44
What Windows compliance settings would you consider essential for a corporate environment?
The non-negotiable baseline: Require BitLocker encryption, require Windows Defender Antimalware with real-time protection and up-to-date signatures, require Secure Boot, set a minimum OS version (current supported build), require a screen lock password, and set max inactivity lock time. For higher security environments, add Microsoft Defender for Endpoint risk score requirements — blocking devices with Medium or High threat levels from accessing corporate resources.
Why should rooted/jailbroken devices be flagged as non-compliant?
Jailbreaking iOS or rooting Android removes the operating system's security sandbox — the fundamental layer that isolates apps and protects data. A jailbroken/rooted device can run unsigned code, bypass app permissions, access any file on the device, and disable security features. Any malicious app can gain full device control. Such a device cannot be trusted to handle corporate data — it should be immediately flagged non-compliant and blocked from accessing company resources.
TOPIC 45
Compliance Policy Assignment & Evaluation
How compliance policies are assigned, when they evaluate, and what triggers re-evaluation
How Assignment Works for Compliance

Compliance policies are assigned to Azure AD groups — the same mechanism as Configuration Profiles. You can assign to user groups (policy evaluates on devices the user signs into) or device groups (policy evaluates on specific devices regardless of user). The same include/exclude and filter mechanics apply.

💡 Key Difference: User vs Device for Compliance

If you assign a compliance policy to a user group and the user signs into multiple devices, the policy evaluates on ALL those devices. Each device gets individually evaluated and marked compliant or non-compliant independently. The user's access via Conditional Access depends on whether the SPECIFIC DEVICE they're currently using is compliant — not a combined status of all their devices.

When Does Compliance Evaluate?
TriggerWhat Happens
Initial enrollmentCompliance evaluation begins immediately after enrollment. Device gets 3 days by default to become compliant (configurable grace period).
Periodic check-in (every 8 hours)Device reports current state to Intune. Compliance re-evaluated against current policy settings.
Manual sync triggeredImmediate re-evaluation — useful after fixing a compliance issue to quickly restore access.
Policy changeWhen a compliance policy is modified or a new one assigned, Intune pushes a sync request — devices re-evaluate within minutes.
OS update installedWindows reports new OS version to Intune → compliance re-evaluated (minimum OS version check updates).
User manually syncsTriggers immediate re-evaluation. Useful after user installs an OS update or enables encryption.
Multiple Compliance Policies — Precedence Rules

A device can receive multiple compliance policies (e.g., one from a user group assignment and another from a device group assignment). The rules for multiple policies are different from Configuration Profile conflicts:

  • Most restrictive wins: If Policy A requires password length 6 and Policy B requires 8, the device must have length 8 to be compliant. The stricter requirement takes precedence.
  • All policies must pass: The device is only marked Compliant if it meets ALL requirements across ALL assigned compliance policies. Failing any single policy from any assigned compliance policy = Non-Compliant.
  • No "conflict" status: Unlike Configuration Profiles, compliance policies don't create conflicts. They stack — more policies = more requirements to meet.
Compliance Evaluation Scope
  1. 1
    Device checks in → reports hardware specs, OS version, encryption state, Defender status, etc.
  2. 2
    Intune evaluates each compliance policy assigned to the device/user against the reported device state.
  3. 3
    Per-setting result: Each individual setting is marked Compliant or Non-Compliant (visible in Intune device compliance details).
  4. 4
    Overall status = Compliant only if EVERY setting in EVERY assigned policy passes. One failure = Non-Compliant overall.
  5. 5
    Status written to Azure AD → Conditional Access reads it → Access decisions made in real-time.
🏦
Real-World Multiple Policy Scenario

Setup: Finance department has two compliance policies assigned:
Policy 1 (All Devices group): Require BitLocker, Minimum OS Windows 11 22H2, Require Defender AV.
Policy 2 (Finance Devices group): Require MDE risk score ≤ Low (stricter than the default Medium), Require Secure Boot, Password length ≥ 12.
A Finance laptop must meet ALL six requirements across both policies to be marked Compliant. If it's missing Secure Boot (even though Policy 1 doesn't require it), it's Non-Compliant because Policy 2 requires it. This layered approach lets the security team add Finance-specific hardening on top of the baseline — without modifying the baseline policy.

🎤 Interview Questions — Topic 45
If a device has two compliance policies assigned and fails a requirement in only one of them, what is the overall compliance status?
Non-Compliant. A device must meet ALL requirements across ALL assigned compliance policies to be marked Compliant. Failing any single requirement in any single assigned policy results in an overall Non-Compliant status. Unlike Configuration Profile conflicts, compliance policies don't create conflict states — they stack, and the most restrictive requirement wins.
How quickly does a device's compliance status update after a user fixes a non-compliance issue?
The device won't automatically reflect the fix until the next check-in (up to 8 hours). To get immediate re-evaluation: the user can manually sync from Company Portal or from Settings → Access work or school → Sync. The admin can also trigger a sync from Intune portal → Device → Sync. After sync, Intune re-evaluates and updates Azure AD compliance status within minutes — restoring Conditional Access if now compliant.
TOPIC 46
Grace Period & Actions for Non-Compliance
What happens when a device fails compliance — automated escalating responses
The Grace Period

When a device first becomes non-compliant (either at enrollment or after a policy change), Intune provides a configurable grace period before consequences kick in. During this window, the device is marked non-compliant in reports but Conditional Access may still allow access.

  • Default grace period: 3 days after enrollment. Configurable from 0 to 365 days.
  • Purpose: Give users time to fix compliance issues (install OS update, enable BitLocker) without being immediately locked out.
  • Zero grace period: Device is immediately blocked on enrollment if non-compliant. Harsh — use only for high-security scenarios where new devices must be 100% compliant before any access is granted.
  • Long grace period (30+ days): Gives generous time to fix issues. Risk: non-compliant devices with security issues still access resources for weeks.
Actions for Non-Compliance — Escalating Responses

Intune allows you to configure automated actions that trigger at specific intervals after a device becomes non-compliant. These actions escalate over time — start with a gentle notification, escalate to blocking, and ultimately to remote wipe if the device stays non-compliant.

Designing a Non-Compliance Action Schedule
📅
Example Non-Compliance Escalation Schedule

Day 30: Retire device (selective wipe of corporate data, unenroll from Intune).

Always customize it with: the company's IT support phone number and link, a clear explanation of what the user needs to do in plain English (not IT jargon), a link to a self-service help article, and a deadline. A well-written email dramatically reduces helpdesk calls about "why can't I access email?"
🎤 Interview Questions — Topic 46
What is a grace period in Intune compliance and when would you set it to zero?
A grace period is the time after a device becomes non-compliant before consequences like Conditional Access blocking take effect. The default is 3 days. You'd set it to zero for high-security environments — like classified government systems or financial trading platforms — where any non-compliant device must be immediately blocked, even if it means the user is instantly locked out. For most enterprise environments, a 1-3 day grace period is appropriate to allow users time to self-remediate.
Walk me through the non-compliance actions you would configure for BYOD mobile devices.
For BYOD mobiles: Day 0 — send email and push notification explaining what's non-compliant and how to fix it. Day 1 — mark as non-compliant in Intune (triggers Conditional Access blocking of corporate apps). Day 7 — second reminder email CC'd to manager. Day 14 — remote lock the device. Day 30 — Retire the device (selective wipe of corporate data, leave personal data intact, unenroll from Intune). For BYOD, never use full device wipe — you don't own the device and wiping personal data creates legal liability.

TOPIC 48
Compliance + Conditional Access Integration
How device compliance becomes access control — the most powerful Intune integration
The Integration Architecture

Compliance + Conditional Access is the most important integration in the Microsoft security stack for endpoint admins. Here's how they connect:

  1. 1
    Intune evaluates device against compliance policy → marks Compliant or Non-Compliant → writes status to Azure AD.
  2. 2
    User tries to access Exchange Online, SharePoint, Teams, or any CA-protected app.
  3. 3
    Azure AD Conditional Access intercepts the request → evaluates the CA policy conditions.
  4. 4
    CA policy condition: "Require device to be marked compliant" → checks Azure AD for the device's compliance status.
  5. 5
    If Compliant: Access Granted. If Non-Compliant or Not Evaluated: Access Blocked (or redirect to a remediation page).
Building the Compliance + CA Policy Combination
ComponentWhere ConfiguredWhat It Does
Compliance PolicyIntune → Devices → ComplianceDefines what "secure device" means. Evaluates devices. Writes status to Azure AD.
Conditional Access PolicyAzure AD → Protection → Conditional AccessReads compliance status from Azure AD. Blocks or allows access based on compliance.
Combined effectBoth must be configuredNon-compliant devices are automatically blocked from corporate resources.
The Three Access Scenarios
✅
Scenario 1: Full Access
User signs in with MFA ✅ + Device is compliant ✅ → Conditional Access grants full access to Exchange, SharePoint, Teams. No friction for the user — they don't even notice CA is running.
❌
Scenario 2: Blocked
User signs in ✅ + Device is Non-Compliant ❌ → Access blocked. User sees an error page: "You can't get there from here. Your device doesn't meet security requirements." Link to fix the issue.
⚠️
Scenario 3: Personal Device (MAM-WE)
User on personal BYOD device (not enrolled) → CA checks "Require app protection policy" instead of device compliance → grants access only via managed apps (Outlook, Teams) with MAM DLP applied.
The "Require compliant device OR app protection" Pattern

A sophisticated CA design allows two paths to access — useful when you support both fully enrolled devices AND BYOD MAM-WE:

  • Path 1: Device is enrolled and compliant → full access via any app (Outlook, browser, etc.)
  • Path 2: Device is NOT enrolled (personal) → requires App Protection Policy (MAM) → access only via managed apps with DLP enforced

In the CA policy Grant controls: select "Require device to be marked compliant" OR "Require app protection policy" (select "Require one of the selected controls"). This is the most common enterprise configuration for supporting both corporate and BYOD devices.

What Users See When Blocked

When CA blocks a non-compliant device, the user is redirected to a remediation page — not just an error. The page explains:

  • What requirement their device is failing
  • Steps to fix it (install OS update, enable encryption, etc.)
  • A link to the Company Portal app to check compliance status after fixing
  • IT helpdesk contact information

You can customize this page via the Azure AD Conditional Access portal → Customize continuous access evaluation message.

🚨 Common Mistake — Enable CA Before Compliance Policies Exist

A new admin turns on a CA policy requiring device compliance — BEFORE assigning compliance policies to devices. Result: ALL enrolled devices are treated as Non-Compliant (no policy assigned = non-compliant). ALL users are immediately blocked from Exchange, SharePoint, Teams. The organization grinds to a halt. Always: Create and assign compliance policies first → verify devices are showing Compliant → THEN enable CA policy in Report-Only mode → review impact → THEN enforce.

🔐
Enterprise CA + Compliance Rollout Plan

Week 1: Create compliance policies. Assign to All Devices. Monitor — see what % of devices pass without taking action. Identify the most common failures (usually BitLocker or OS version).
Week 2–3: Fix the top failure causes. Push OS updates via Update Rings. Enable BitLocker via Endpoint Security policy. Aim for 95%+ compliance before enabling CA.
Week 4: Create CA policy requiring device compliance. Set to Report-Only mode. Review Sign-in logs to see who WOULD be blocked.
Week 5: Communicate to all employees: "Starting [date], your device must meet security requirements to access company resources." Provide self-service fix guide.
Week 6: Switch CA policy from Report-Only to Enforce. Monitor helpdesk tickets. Support stragglers. Target 99%+ compliance within 2 weeks of enforcement.

🎤 Interview Questions — Topic 48
How does Intune compliance integrate with Conditional Access?
Intune evaluates devices against compliance policies and writes the result (Compliant or Non-Compliant) to Azure AD as a device attribute. When a user accesses a CA-protected resource like Exchange Online, Azure AD evaluates the CA policy conditions — including "Require device to be marked compliant." If the device's compliance status in Azure AD is Non-Compliant, CA blocks the access request. If Compliant, access is granted. Intune provides the signal; CA acts on it.
What is the "Require compliant device OR app protection policy" CA grant control pattern and why is it used?
This CA grant pattern allows two paths to corporate resources: fully enrolled compliant devices get unrestricted app access, while unenrolled personal (BYOD) devices can still access corporate resources but only via apps protected by MAM App Protection Policies (like Outlook with DLP controls). It prevents the situation where personal device users are completely locked out — instead they get controlled access via managed apps. This is the standard pattern for organizations supporting both corporate-enrolled and BYOD scenarios.
Why should you never enable a CA policy requiring compliance before assigning compliance policies to devices?
Because devices with no compliance policy assigned are treated as Non-Compliant by Azure AD by default. Enabling CA requiring compliance before assigning policies means every enrolled device immediately appears Non-Compliant — Conditional Access blocks all users from Exchange, SharePoint, and Teams simultaneously. The correct order is: create and assign compliance policies → verify devices are marking Compliant → enable CA in Report-Only mode → review impact → then enforce.
TOPIC 49
Per-Platform Compliance — Windows, iOS, Android, macOS
Platform-specific compliance considerations and configuration differences
Why Platform Compliance Differs

Each platform has different capabilities, different security models, and different threat vectors. A compliance policy must be tailored to each platform — you can't apply Windows compliance settings to iOS. Intune enforces this: compliance policies are created per-platform.

Windows Compliance — Deep Notes
  • Windows Health Attestation: Uses the device's TPM chip to verify the boot chain integrity at a hardware level. Requires Defender for Endpoint P1/P2. Very high assurance — can't be spoofed by a compromised OS.
  • MDE Integration for Windows: Compliance can require a maximum Defender for Endpoint risk score (None/Low/Medium/High). Devices with active threats automatically become non-compliant → blocked by CA → forces remediation before access is restored.
  • BitLocker compliance vs BitLocker config profile: Compliance CHECKS if BitLocker is on. Config Profile ENABLES BitLocker. Both are needed — config profile to turn it on, compliance to verify it stayed on.
  • OS version compliance on Windows: Intune can check for specific KB (Knowledge Base) patch levels, not just feature versions. Example: require KB5031356 or later, ensuring a specific security patch is installed.
iOS Compliance — Deep Notes
  • Jailbreak detection: Intune checks for jailbreak indicators on iOS devices. If detected, marks as Non-Compliant immediately. Note: sophisticated jailbreaks can evade detection — defense in depth is needed.
  • iOS compliance only evaluates at check-in: Unlike Windows (which can report health attestation in near real-time), iOS compliance is evaluated at each check-in. A device jailbroken between check-ins stays Compliant until the next evaluation.
  • Jamf integration: If using JAMF Pro for Mac management, you can configure Jamf → Intune integration where Jamf sends compliance signals to Intune, and Intune passes them to Azure AD for CA evaluation. Best of both worlds.
  • Managed device vs MAM-WE: An MDM-enrolled iPhone can have full compliance evaluation. A MAM-WE (non-enrolled) personal iPhone has limited compliance checking — mostly via App Protection Policy conditions in CA instead.
Android Compliance — Deep Notes
  • Android SafetyNet Attestation (now Play Integrity): Google's hardware-backed device integrity check. Compliance can require passing Google's Play Integrity API — which validates that the device hasn't been tampered with and is running genuine Android. Much stronger than software-only root detection.
  • Android threat level via Defender: If Defender for Endpoint for Android is deployed, compliance can check the mobile threat defense risk score — blocking devices with banking trojans, network attacks, or phishing apps detected.
  • Work Profile vs Fully Managed compliance: Compliance policies work the same way for both enrollment modes — but available settings may differ slightly. Fully Managed has more device-level settings (like factory reset protection verification) than Work Profile.
  • Samsung Knox compliance: Knox Platform for Enterprise provides additional hardware attestation for Samsung devices — compliance can specifically require Knox attestation, providing a Samsung-level hardware security guarantee beyond standard Android compliance.
macOS Compliance — Deep Notes
  • FileVault + Intune escrow: Compliance can require FileVault to be enabled. The Configuration Profile can enable it silently. Intune stores the personal recovery key — admin can view it if the user is locked out. Compliance checks that it's still enabled and the key is escrowed.
  • Gatekeeper enforcement: Compliance can require Gatekeeper (macOS's app trust system) to be set to "App Store and identified developers" minimum, blocking unsigned and untrusted apps.
  • Jamf + Intune co-management: Many enterprises use Jamf Pro for Mac management + Intune for compliance signal and CA. Jamf sends device health data to Intune → Intune evaluates compliance → Azure AD CA enforces access. The admin manages Macs primarily in Jamf but leverages Intune/CA for access control.

🎤 Interview Questions — Topic 49
How does Intune detect jailbroken iOS devices and what are the limitations?
Intune checks for jailbreak indicators at each device check-in — looking for files, processes, and behaviors characteristic of jailbroken devices. The limitation is timing: evaluation only happens at check-in (every 6-8 hours), so a device jailbroken between check-ins remains Compliant until the next evaluation. Also, sophisticated jailbreaks can evade detection. Defense in depth is essential: App Protection Policies should also be configured to prevent corporate data from being accessed on jailbroken devices, as MAM can detect jailbreak via the SDK independently of Intune MDM evaluation.
What is Google Play Integrity and how does it enhance Android compliance?
Play Integrity (formerly SafetyNet Attestation) is Google's hardware-backed API that verifies whether an Android device is genuine, unmodified, and running certified Android. Compliance can require devices to pass Play Integrity checks — which validates the device hasn't been rooted, the OS isn't modified, and the device is a legitimate Android device. This is significantly stronger than software-only root detection because it uses hardware-level attestation that sophisticated root kits can't easily bypass.

Troubleshooting Compliance Issues
Diagnosing why a device shows non-compliant — and how to fix it systematically
The Compliance Troubleshooting Approach
  1. 1
    Find exactly which setting is failing: Devices → All Devices → Device → Device compliance tab → Click non-compliant policy → read per-setting results. Note the exact failed setting and the current vs required value.
  2. 2
    Verify the device actually has the issue: Is BitLocker really off? Check on the device itself (Control Panel → BitLocker). Is the OS really below the required version? Check winver.exe. Don't assume Intune is wrong — verify the device state directly.
  3. 3
    Fix the underlying issue: Enable BitLocker. Install the OS update. Set a screen lock PIN. The compliance check will re-evaluate on next sync.
  4. 4
    Force a sync to re-evaluate immediately: On device: Settings → Accounts → Access work or school → Info → Sync. Or in Intune: Devices → Device → Sync. Don't wait 8 hours.
  5. 5
    Verify compliance status updated: Check the device compliance tab again — should now show Compliant. CA access should restore within a few minutes of status update.

Why Is User Blocked Despite Device Showing Compliant?

This is one of the most confusing compliance scenarios — Intune says Compliant, but the user is still blocked by CA. Here's the systematic check:

  • Check Azure AD Sign-in logs: Go to Azure AD → Sign-in logs → filter by the user → find the blocked attempt → see the failure reason. Is it compliance? Or is it MFA? Or location? CA block reason is always logged here.
  • Compliance status propagation delay: Intune updates the device compliance status in Azure AD — but there can be a few minutes of delay. Have the user wait 5 minutes then try again.
  • Is it the right device? The user might be signing in from a different device (personal laptop) that is NOT compliant. CA checks the device the user is CURRENTLY on, not all their devices.
  • CA policy target: Verify the CA policy applies to this user's group and this app. The user might be blocked by a different CA policy than the compliance one.
  • Token caching: Azure AD access tokens can be cached — the user's session might still have an old (pre-compliance) token. Have them sign out completely and sign back in to force fresh token evaluation.

Real Troubleshooting Case

Ticket: "I enabled BitLocker on my laptop as IT asked. Company Portal says I'm compliant. But I still can't access Outlook on the web — it says my device doesn't meet requirements."
Investigation:
Step 1: Intune → Device → Device compliance: Shows Compliant. BitLocker check: Passed. ✅
Step 2: Azure AD → Sign-in logs → filter user → find blocked attempt → Failure reason: "Device not compliant."
Step 3: Timestamp of sign-in attempt (2:47 PM) vs compliance update timestamp (2:51 PM). The user tried to access Outlook 4 minutes BEFORE Intune finished propagating the new compliance status to Azure AD.
Step 4: Ask user to sign out of all browser sessions (clear cached tokens), then sign back in.
Result: Access restored immediately after fresh sign-in. Root cause: compliance status propagation delay + old cached token. Not a real compliance failure — just a timing issue.

🎤 Interview Questions — Topic 50
A user is blocked by Conditional Access but Intune shows their device as compliant. What do you check?
First, check Azure AD Sign-in logs for the blocked attempt — the failure reason tells you exactly which CA condition failed. Compliance? MFA? Location? Then verify the device the user is signing in FROM — they might be on a different device than the one showing compliant in Intune. Check for compliance status propagation delay (Intune → Azure AD can take minutes). If delay, have user sign out completely and sign back in to get a fresh token. Also verify the CA policy actually applies to this user's group and app combination.
A Windows device shows non-compliant for BitLocker but BitLocker appears to be enabled on the device. What could cause this?
Several possibilities: (1) BitLocker is enabled but not protecting the OS drive (maybe only a data drive), (2) BitLocker is suspended — a suspended state is not the same as fully enabled (check Manage-bde -status in Command Prompt), (3) Compliance data is stale — the device's last reported state to Intune was before BitLocker was enabled — force a manual sync, (4) The specific BitLocker check in the compliance policy requires additional settings (like startup PIN or TPM-only protection) that aren't met even though encryption is on. Run Manage-bde -status C: to see the full BitLocker state on the OS drive.
Where is the best first place to look when a user is blocked by Conditional Access and you need to know why?
Azure AD Sign-in Logs — always the first stop. Filter by the user's UPN, find the failed sign-in attempt, and look at the Conditional Access tab of that event. It shows every CA policy that evaluated, whether it passed or failed, and the specific condition that failed (compliance, MFA, location, risk level). This tells you in 30 seconds exactly what's blocking the user — saving the time of checking multiple Intune reports, compliance tabs, and guessing.
