TOPIC 106
Intune Troubleshooting Methodology
The Step-by-Step Approach That Senior Engineers Use
Why Methodology Matters

Random clicking and hoping is not troubleshooting. Senior Intune engineers follow a structured approach that finds root causes quickly — even for issues they've never seen before. The method below works for 95% of Intune issues.

The 7-Step Intune Troubleshooting Framework
  1. 1
    Define the problem precisely. "Intune is broken" is not useful. "Policy X is not applying to device Y, assigned via group Z, and compliance shows Non-Compliant since date W" is actionable. Collect: device name, user UPN, OS version, enrollment date, which policy/app/profile is failing.
  2. 2
    Check the Intune portal first. Go to Devices → the specific device → Device configuration / Compliance / Apps. Look at assignment status. Is it showing "Not applicable", "Error", "Pending", or "Succeeded"? The portal gives you the first clue before touching any log.
  3. 3
    Verify assignment and targeting. Is the policy actually assigned to the correct group? Is the user or device in that group? Dynamic group filters — are they evaluating correctly? Check group membership in Azure AD.
  4. 4
    Check the device sync status. When did the device last check in? If it was 7 days ago, the policy may have been assigned after the last sync. Trigger a manual sync: Intune portal → Device → Sync, or on the device via Settings → Accounts → Access work or school → Info → Sync.
  5. 5
    Read the logs. IME log for app issues, MDMDiagReport for policy issues, Event Viewer for enrollment issues. Logs tell you the exact error code and what the device tried to do.
  6. 6
    Look up the error code. Every Intune error has a code (e.g., 0x87D1FDE8, 0x80070005). Search Microsoft Docs or the Intune error code reference. Never guess what an error means — look it up.
  7. 7
    Test in isolation. Create a test group with a single test device. Apply only the failing policy. This eliminates conflicts and confirms the issue is reproducible. If it works in isolation, the problem is likely a conflict with another policy.

✅ Senior Engineer Habit

Before doing anything, open the Intune Troubleshooting blade (Troubleshooting + Support → Troubleshoot) and search for the user. This single screen shows device list, enrollment status, app install status, compliance state, and policy assignments — all in one place. It saves 10 minutes of clicking around.

🎤 Interview Questions — Topic 106
Walk me through how you'd troubleshoot a policy that's not applying to a device.
I follow a structured approach: first confirm the device is enrolled and synced recently; check the portal for assignment status on that device; verify group membership; then check logs — MDMDiagReport or Event Viewer for Windows. If I see an error code, I look it up in Microsoft Docs. Finally, I test in isolation with a clean test group to rule out conflicts.

This gives me a starting point in under 2 minutes.
TOPIC 107
Key Log Files
IME Log, MDMDiagReport, Event Viewer — Know Where to Look
Why Logs Are Your Best Friend

Every Intune action leaves a trace. When a policy fails, a log records why. When an app fails to install, a log shows the exact exit code. Knowing which log to read — and how to read it — is what separates a junior admin from a senior engineer.

Log #1 — IntuneManagementExtension.log (IME Log)

Location: C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log

Use it for: Win32 app deployments, PowerShell script execution, proactive remediations, shell script issues. This is the most-used log file for day-to-day Intune troubleshooting on Windows.

Sample IME Log Entry — App Install Failure [Win32App] Processing app: Chrome Enterprise
[Win32App] Detection rule result: Not Detected
Downloading content for app Chrome Enterprise from CDN...
Install command: GoogleChromeEnterpriseBundle64.msi /quiet
[Win32App] Install exit code: 1603
[Win32App] App install failed. Retry attempt 1 of 3.
💡
Exit code 1603 = Fatal error during installation. Usually means a prerequisite is missing, another installer is running, or the system needs a reboot. Exit code 0 = success. Exit code 3010 = success but reboot required.
Log #2 — MDMDiagReport.xml / HTML

Generated by: MDMDiagnosticsTool.exe (covered in Topic 108)

Use it for: MDM enrollment status, policy CSP application status, certificate issues, MDM channel communication. It shows every MDM policy that was sent down, whether it applied, and any errors. Essential for compliance and configuration profile troubleshooting.

📋 What MDMDiagReport Shows

Device info (OS version, join type, enrollment date) · MDM enrollment state · Certificate store · All applied MDM policies and their status · Defender status · Wi-Fi/VPN profiles · Windows Update policies · Event log snapshots. The HTML version is easiest to read — open in a browser.

Log #3 — Windows Event Viewer

Event Viewer contains several Intune/MDM-relevant channels. Know these paths:

Event Log PathWhat It CoversKey Event IDs
Applications and Services Logs → Microsoft → Windows → DeviceManagement-Enterprise-Diagnostics-Provider → Admin MDM policy application, enrollment events, CSP errors 72, 404, 405, 814
Microsoft → Windows → AAD → Operational Azure AD join events, token acquisition, PRT issues 1098, 1104
Microsoft → Windows → User Device Registration → Admin Device registration, Hybrid AADJ events, Workplace Join 300, 304, 360
Application Log MSI installer events, Win32 app install errors 1033, 1034, 11707
Microsoft → Windows → Autopilot → Operational Autopilot provisioning events, profile download, ESP phases 100, 101, 109
Log #4 — AgentExecutor.log

Location: C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\AgentExecutor.log

Specifically tracks PowerShell script execution via Intune. If a script assigned via Intune isn't running or is failing, this log shows the exact PowerShell output and error.

Log #5 — Microsoft Endpoint Manager Portal Audit Logs

Not a file on the device — these are in the Intune portal under Tenant administration → Audit logs. They answer "who changed what and when?" Essential for change-related troubleshooting and compliance audits.

🎤 Interview Questions — Topic 107
Where is the IME log and what is it used for?
The IntuneManagementExtension log is at C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\. It's the primary log for troubleshooting Win32 app deployments, PowerShell scripts, and proactive remediations on Windows. It shows the full install flow including download, detection, install command, exit code, and retry attempts.
An app deploys fine to 99 devices but fails on one. Where do you look first?
The IME log on the failing device. I'd look for the app name in the log, find the install attempt, and check the exit code. Then cross-reference the exit code — 1603 means fatal install error, often a dependency or reboot issue. I'd also check the Application Event Log for MSI installer events that give more detail.
TOPIC 108
MDMDiagnosticsTool
Collecting and Reading the Full Diagnostic Package
What Is MDMDiagnosticsTool?

MDMDiagnosticsTool.exe is a built-in Windows command-line utility that collects a comprehensive snapshot of the device's MDM state into a single report. It's the first thing Microsoft Support asks for when you raise an Intune support ticket. Learning to run it and read it yourself saves hours.

How to Run MDMDiagnosticsTool

Method 1 — Command Prompt (Admin):

Collect full MDM diagnostics to a folder MDMDiagnosticsTool.exe -area Autopilot;DeviceEnrollment;DeviceProvisioning;TPM -zip C:\Temp\MDMLogs.zip

# For all areas (most complete):
MDMDiagnosticsTool.exe -area all -zip C:\Temp\MDMDiag_Full.zip

Method 2 — Settings App (No Admin Required): Settings → Accounts → Access work or school → select the enrolled account → Info → Create Report. This saves the report to C:\Users\Public\Documents\MDMDiagnostics

What's Inside the Diagnostic Package
📄
MDMDiagReport.html
The main readable report. Open in a browser. Shows enrollment info, applied policies, CSP status, and errors in organized sections.
📋
MDMDiagReport.xml
Machine-readable version. Useful for scripting or comparing two device states programmatically. Same data as HTML but in XML format.
🪵
Event Log Exports
Zipped .evtx files from the DeviceManagement, AAD, UserDeviceRegistration, and Autopilot event channels.
🔐
Registry Exports
Key MDM-related registry paths exported as .txt. Includes enrollment keys, policy CSP values, and certificate thumbprints.
How to Read MDMDiagReport.html
  1. 1
    Enrollment Information section: Confirm device is enrolled, MDM authority is Intune, enrollment date, UPN of enrolled user, and device AAD join type. If enrollment info is missing or wrong, stop here — enrollment itself is the issue.
  2. 2
    Policies section: Lists every MDM policy CSP that was delivered to the device. Each entry shows: policy name, area, setting, value applied, and result (success/error code). Look for anything with an error code.
  3. 3
    Certificates section: Shows all certificates in the device's certificate stores. Critical for SCEP/PKCS troubleshooting — verify the expected cert is present, not expired, and in the right store.
  4. 4
    Windows Update / Autopatch section: Shows WUfB policies applied, deferral settings, and last scan/install times. Use for patching-related issues.
  5. 5
    Autopilot section (if applicable): Deployment profile details, ESP configuration, phase completion status. Essential for Autopilot troubleshooting.
🌏
Real-World Scenario

Problem: A device shows compliant in Intune but BitLocker reports in Defender for Endpoint show it as unencrypted. No error in the portal.

Action: Run MDMDiagnosticsTool on the device. Open MDMDiagReport.html → Policies section → search for "BitLocker". Find: BitLocker policy was delivered successfully, but CSP result shows error 0x80310059 — "BitLocker is already enabled but for the wrong drive."

Root cause: Device had a pre-existing partial BitLocker encryption from before Intune enrollment. The Intune policy couldn't complete because of the conflict.

Fix: Decrypt the drive, let Intune re-apply the policy, encryption completes silently.

🎤 Interview Questions — Topic 108
How do you collect MDM diagnostics from a Windows device?
Two ways: run MDMDiagnosticsTool.exe from an elevated command prompt with the -area and -zip flags to get a full diagnostic package, or go to Settings → Accounts → Access work or school → Info → Create Report for a no-admin option. The output is an HTML and XML report plus event log exports. I open the HTML in a browser and look at the Enrollment, Policies, and Certificates sections.
What's in the MDMDiagReport and why is it useful?
It's a complete snapshot of the device's MDM state — enrollment info, every policy that was applied with its result code, certificate store contents, registry exports, and event log captures. It's useful because it shows you in one file what Intune sent to the device and whether the device accepted it successfully or threw an error, which narrows down where the break is.
TOPIC 109
Troubleshooting Device Enrollment Failures
Why Devices Won't Enroll — and How to Fix It
The Enrollment Pipeline

Before fixing enrollment errors, understand the pipeline: User authenticates with Azure AD → MDM Terms of Use accepted → MDM auto-enrollment triggered → Intune agent installed → Device syncs policies. A break at any stage shows a different error.

Common Enrollment Errors — Windows
0x80180026
MDM enrollment not permitted
MDM enrollment scope is set to "None" or user is not in the MDM enrollment scope group. This is the most common enrollment error for new Intune setups.
Go to Azure AD → Mobility (MDM and MAM) → Microsoft Intune → MDM user scope. Set to "All" or add the user's group to "Some". Then have user retry enrollment.
0x80180014
Enrollment limit reached
The user or device has hit the enrollment limit (default: 15 devices per user, but org may set lower). Intune refuses new enrollments once the limit is hit.
Check Intune → Devices → Enrollment restrictions → Device limit restrictions. Either increase the limit, or delete old/stale device records for this user in Azure AD and Intune.
0x80180003
This device is already enrolled
A stale MDM enrollment exists on the device from a previous tenant or partial enrollment. The registry still has old MDM keys.
Run: dsregcmd /status to check join state. If stale, use dsregcmd /leave to remove old enrollment. Then re-enroll. On older devices, manually clean the registry under HKLM\SOFTWARE\Microsoft\Enrollments.
0x80090016
TPM keyset not available
TPM-related error during enrollment certificate creation. Common on devices with a dirty/locked TPM or after a system image deployment where TPM wasn't cleared.
Clear the TPM: Settings → Windows Security → Device Security → Security processor details → Clear TPM. Reboot and retry enrollment. If persistent, check BIOS TPM settings.
Common Enrollment Errors — iOS/iPadOS
Error / SymptomLikely CauseFix
Profile installation fails — "Cannot Install Profile" MDM profile conflicts with an existing MDM profile (e.g., from Apple Configurator or another MDM) Remove existing MDM profile from Settings → General → VPN & Device Management. Retry enrollment.
ADE device doesn't show MDM enrollment prompt Device not assigned to MDM server in Apple Business Manager, or sync hasn't completed In ABM, assign device to Intune MDM server. In Intune, sync ABM tokens. Factory reset device and re-run Setup Assistant.
User enrollment fails — "Your organization has not configured enrollment" iOS/iPadOS enrollment not configured in Intune, or Apple MDM Push Certificate missing or expired Check Intune → Devices → iOS/iPadOS enrollment → Apple MDM Push certificate. Renew if expired. Must use same Apple ID to renew — using a different Apple ID requires re-enrollment of all iOS devices.
Useful Commands for Enrollment Troubleshooting

dsregcmd /status

# Key fields to look for:
# AzureAdJoined : YES
# MDMUrl : https://enrollment.manage.microsoft.com (Intune)
# MdmEnrollmentURL: should match your tenant

🎤 Interview Questions — Topic 109
A user gets error 0x80180026 when trying to enroll their Windows device. What's the problem?
This means MDM enrollment is not permitted for that user. The MDM user scope in Azure AD Mobility settings is either set to None or the user isn't in the allowed group. Fix: go to Azure AD → Mobility → Microsoft Intune → MDM user scope and either set to All or add the user's group to the scope.
What command do you run on a Windows device to verify MDM enrollment status?
dsregcmd /status — this shows the device's Azure AD join state, whether MDM is enrolled, the MDM URL (which should point to Intune), and the Primary Refresh Token (PRT) status. It's the first command I run for any enrollment or Conditional Access issue on Windows.
TOPIC 110
Troubleshooting App Deployment Failures
Win32, Store, and LOB App — Diagnosing Install Failures
App Deployment Flow (Win32)

Understanding the flow helps you know where to look when it breaks: Intune portal assigns app → IME agent downloads content from Azure CDN → IME extracts and runs install command → Detection rule runs → Result reported back to Intune portal.

Most Common Win32 App Failures
0x87D1FDE8
Remediation failed
One of the most common Win32 app error codes. Usually means the install command ran but returned a non-zero exit code. The installer itself failed. Also triggered if the detection rule doesn't find the app after a successful install.
Check IME log for the actual installer exit code. Fix the installer command or detection rule. Most common cause: detection rule path is wrong (wrong registry key or file path after install).
0x80070005
Access denied
IME runs in SYSTEM context by default. If install command requires user context or tries to write to user-specific paths, it gets access denied. Also triggered if antivirus blocks the installer.
In the Win32 app config, change "Install behavior" to "User" context if the app is per-user. Or adjust the install script to not require user paths. Check Defender exclusions if AV is blocking.
-2016281112
Intune app not applicable
The app requirement rules aren't met on the device. OS version, disk space, or a custom requirement script returned "not applicable."
Review the app's Requirement rules in Intune. Test requirement scripts manually on the device. Most common: minimum OS version set too high, or disk space requirement not met.
Troubleshooting Steps — Win32 App
  1. 1
    Check portal status: Devices → the device → Apps tab. Note the status — "Failed", "Not Applicable", "Pending Install". The error code is shown here.
  2. 2
    Open IME log: C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log. Search for the app name. Find the install attempt, note exit code.
  3. 3
    Test install manually: Run the exact same install command that Intune uses, from an elevated command prompt as SYSTEM (use PsExec: psexec -s cmd.exe). This confirms if the installer itself is the problem.
  4. 4
    Test detection rule manually: After a manual install, check if the detection rule condition is met — does the registry key exist? Does the file exist at the path specified? Detection rule mismatch is the #1 cause of "failed" status even when the app installed successfully.
  5. 5
    Force sync and monitor: After fixing, trigger a device sync and monitor IME log in real time using: Get-Content -Path "...\IntuneManagementExtension.log" -Wait -Tail 50

🎤 Interview Questions — Topic 110
An app shows "Failed" in Intune portal but the user says it's actually installed. What happened?
Almost certainly a detection rule mismatch. The app installed successfully (exit code 0 or 3010) but Intune's detection rule didn't find the expected registry key, file, or product code after install. I'd open the IME log to confirm the exit code, then manually verify if the detection rule condition exists on the device. Fix the detection rule to match the actual installed state.
How do you run an Intune install command manually as SYSTEM for testing?
Use PsExec from Sysinternals: psexec -s cmd.exe — this opens a command prompt running as the SYSTEM account, which is the same context IME uses. From there, run the exact install command string from the Intune app configuration. This lets you see the raw installer output and exit code without waiting for Intune to retry.
TOPIC 111
Troubleshooting Policy Not Applying
Configuration Profiles, Compliance Policies, and Security Baselines
The Policy Delivery Chain

For a policy to apply: it must be assigned → device must be in the target group → device must sync → MDM channel must accept the CSP setting → no conflict with another policy overriding it. Any break in this chain = policy not applying.

Diagnostic Steps
  1. 1
    Check per-device policy status in portal: Intune → Devices → [Device] → Device configuration. Each profile shows: Succeeded / Error / Conflict / Not Applicable / Pending. "Not Applicable" often means platform mismatch or assignment filtering excluded this device.
  2. 2
    Check group membership: Open the group the policy is assigned to in Azure AD. Confirm the device or user is a member. For dynamic groups, check if the dynamic rule evaluates correctly for this device's attributes.
  3. 3
    Check for Conflict: If status shows "Conflict", two policies are targeting the same setting with different values. Intune marks the setting as conflicted and applies neither. Go to the specific setting in the profile to see which other profile is conflicting.
  4. 4
    Check MDMDiagReport: Open the Policies section. Find your policy's CSP. See if it was delivered and what the result code was. Error code 65007 = compliance policy not met as prerequisite. Error 0x87D1FDE8 = setting couldn't be applied by the OS.
  5. 5
    Check Registry (for Settings Catalog policies): Most Settings Catalog policies write to registry. Verify the expected registry key/value exists after a sync. Example: Password complexity policies write to HKLM\SOFTWARE\Microsoft\PolicyManager\current\device\DeviceLock.
Common Policy Issues
SymptomCauseFix
Policy shows "Not Applicable" Platform mismatch (e.g., iOS profile assigned to Windows device), or Assignment Filter excluded the device Verify platform target in profile. Check and test Assignment Filters against the device.
Policy shows "Conflict" Two or more policies setting the same value differently Use the conflict details in Intune to identify both conflicting policies. Consolidate settings into one authoritative policy or remove duplicate settings from one profile.
Policy shows "Succeeded" but setting not enforced on device Policy applied via MDM but superseded by a local Group Policy (GPO) or registry value set by another mechanism Check if GPO is applying the conflicting setting. In hybrid environments, GPO often wins over MDM for the same setting. Use ADMX-backed policies or move device to cloud-only management.
Policy "Pending" for a long time Device hasn't synced. Device offline, user hasn't logged in, or Intune Management Extension not running Force sync from portal. Check device is online. Check IME service is running: Services.msc → Microsoft Intune Management Extension.
Custom OMA-URI not applying Wrong OMA-URI path, wrong data type, or CSP not supported on device OS version Verify CSP path in Microsoft documentation. Check the exact data type (String, Integer, Boolean). Check Windows build requirements for the CSP.
⚠️ GPO vs MDM Conflict — Very Common in Hybrid Environments

When a device is Hybrid Azure AD Joined, both Group Policy and Intune MDM can apply settings. If a GPO sets the same registry key as an Intune policy, GPO usually wins (it writes to the same path later). The Intune portal may show "Succeeded" — because MDM successfully sent the setting — but the device shows the GPO value. Fix: block inheritance for the OU, move the setting to MDM-only, or use ADMX-backed Intune policies which write to the same "Policy" registry hive as GPO and therefore follow MDM/GPO precedence rules.

🎤 Interview Questions — Topic 111
A configuration profile shows "Succeeded" in Intune but the setting isn't enforced on the device. What are the possible causes?
Several possibilities: a Group Policy Object is overriding the MDM setting — especially common on Hybrid AADJ devices; a local registry value was manually set and isn't being overwritten; the setting requires a reboot to take effect; or there's a platform limitation where that particular CSP isn't fully supported on that OS build. I'd check Event Viewer's DeviceManagement channel, run dsregcmd /status, and check for conflicting GPOs using gpresult /h.
What does a "Conflict" status mean on a configuration profile setting?
It means two or more Intune policies are targeting the exact same setting with different values on the same device. When Intune detects this, it marks the setting as conflicted and doesn't apply either value — leaving the device in whatever state it was in before. The fix is to identify both conflicting profiles, decide which is authoritative, and remove the duplicate setting from the other profile.
TOPIC 112
Troubleshooting Autopilot Failures
Profile Not Applying, ESP Stuck, and Deployment Errors
The Autopilot Deployment Stages

Autopilot failures happen in distinct stages. Knowing which stage failed narrows your investigation dramatically: Hardware hash registration → Profile download (OOBE) → Azure AD join → Intune enrollment → Enrollment Status Page (ESP) — Device phase → ESP — Account/User phase → Desktop.

Stage 1: Device Doesn't Get Autopilot Profile
OOBE
No branded/custom OOBE — standard Windows setup instead
The device didn't download an Autopilot profile during OOBE. Usually means device isn't registered in Intune/ABM, or profile assignment isn't targeting the device.
Verify the hardware hash is imported in Intune → Devices → Windows → Windows enrollment → Devices. Check the Autopilot profile is assigned to a group containing this device. Confirm internet connectivity during OOBE (Autopilot requires connectivity to download profile).
Stage 2: ESP Gets Stuck or Times Out

The Enrollment Status Page (ESP) blocks the user from the desktop until all assigned apps and policies install. If it gets stuck or times out:

  1. 1
    Check which phase is stuck: ESP has two phases — Device Setup and Account Setup. The screen tells you which phase and often which app/policy is blocking.
  2. 2
    Collect logs during ESP: Press Ctrl+Shift+F3 (Audit mode shortcut — careful), or in newer Windows 11 builds, Shift+F10 opens Command Prompt during ESP. Run MDMDiagnosticsTool from there to capture state mid-deployment.
  3. 3
    Check IME log: App installs during ESP are tracked in the IME log. Find which app is stuck downloading or installing.
  4. 4
    Check ESP timeout settings: In Intune → Devices → Enrollment Status Page profile → review timeout (default 60 min). If complex deployments exceed this, increase the timeout or move non-critical apps to "not blocking" in ESP.
Common Autopilot Error Codes

🌏
Real-World Scenario — ESP Stuck at "Identifying"

Problem: 50 new devices deployed via Autopilot. ESP gets stuck at "Identifying" for 20+ minutes before progressing. Affects all devices.

Investigation: Shifted+F10 during ESP, ran MDMDiagnosticsTool. Event Viewer → Autopilot logs showed devices were waiting on TPM attestation which was timing out.

Root cause: Corporate proxy was blocking ekop.intel.com and ekcert.spserv.microsoft.com — required for TPM attestation during Autopilot.

Fix: Added TPM attestation endpoints to proxy allowlist. ESP "Identifying" phase reduced from 20 minutes to under 60 seconds.

🎤 Interview Questions — Topic 112
A new device goes through Autopilot but ESP gets stuck and eventually times out. How do you troubleshoot?
First, note which ESP phase is stuck — Device Setup or Account Setup — and which app/policy is shown blocking. During ESP, press Shift+F10 to get a command prompt, then run MDMDiagnosticsTool to capture state. Check the IME log for which app is failing. Common causes: an app install timing out, a required app has an install error, or a required policy can't be applied. I'd either fix the failing app or mark non-critical apps as non-blocking in the ESP profile.
A device goes through OOBE but doesn't show the company's branded Autopilot screen. What's wrong?
The device didn't receive an Autopilot deployment profile. Check: is the device's hardware hash registered in Intune? Is there an Autopilot profile created and assigned to a group? Is the device in that group? Is internet available during OOBE? Autopilot profile download requires connectivity to Microsoft endpoints at the very start of OOBE, before any user interaction.
TOPIC 113
Troubleshooting Conditional Access Blocks
Why Users Are Being Blocked and How to Diagnose CA Policies
How Conditional Access Blocks Work

When a user is blocked by CA, Azure AD evaluates all applicable CA policies at the time of sign-in and denies the token. The user sees an error page explaining why. The key is understanding which CA policy triggered the block and which condition wasn't met.

Step 1 — Read the Block Screen

When a user is blocked, the error page contains a Correlation ID and an explanation. Always capture this. The Correlation ID is the link between the user's experience and the Azure AD Sign-In logs entry for that exact authentication attempt.

Step 2 — Azure AD Sign-In Logs
  1. 1
    Go to Azure AD → Monitoring → Sign-in logs
  2. 2
    Filter by the user's UPN and the approximate time of the block
  3. 3
    Find the failed sign-in entry (Status: Failure, Conditional Access: Failure)
  4. 4
    Click the entry → Conditional Access tab. This shows every CA policy that evaluated — which passed, which failed, and exactly which condition failed (compliant device? MFA? approved app?)
Common CA Block Scenarios
Block ReasonCauseFix
Device not compliant CA policy requires compliant device. Device compliance status in Intune is Non-Compliant (grace period expired, compliance policy not met). Check compliance policy status on the device in Intune. Fix the non-compliant setting (e.g., enable BitLocker, set password). Sync the device. Allow up to 15 min for compliance state to refresh in CA.
Device not managed CA requires Hybrid AADJ or AADJ device. Device is only registered (BYOD) or not enrolled at all. Enroll the device in Intune. If personal device, verify the CA policy targets managed devices — should BYOD be excluded?
MFA required but not satisfied CA requires MFA, user doesn't have MFA registered or was denied step-up auth. Register MFA method at aka.ms/mfasetup. Check if user's MFA method was recently disabled or reset.
App not approved CA policy requires approved client app. User is using an app not on the approved list (e.g., native Mail app instead of Outlook). Use an approved app (Outlook, Teams, OneDrive). Or review the CA policy to confirm app restriction is intentional.
Location block CA policy blocks sign-in from specific countries/IP ranges. User is travelling or on a new network. Check if user is in a blocked location. If legitimate, temporarily add their IP to a Named Location exclude, or add the user to a CA exclusion group.
What If You Can't See Sign-In Logs?

In some organizations, only Global Admins or Security Readers can see Sign-In logs. If you're working helpdesk, use the Intune Troubleshooting blade instead. Search for the user → Troubleshoot → look at the "Conditional Access" section which shows a simplified view of CA policy states for that user's device.

💡
CA What-If Tool: Before creating or modifying CA policies, use the What If tool in Azure AD → Conditional Access → What If. Enter a user, app, device, and location — it simulates which CA policies would apply and what the result would be. Essential for testing before deploying CA changes in production.
🎤 Interview Questions — Topic 113
A user is being blocked accessing Exchange Online. They say nothing has changed on their end. How do you investigate?
I start with Azure AD Sign-In logs, filter by the user and time. I open the failed sign-in entry and go to the Conditional Access tab. This shows exactly which CA policy triggered the block and which condition failed. Common reasons: the device recently became non-compliant, MFA was reset, or the user is accessing from a new location. Once I know the exact failing condition, I fix it at the source — not by disabling the CA policy.
How long does it take for a device to go from Non-Compliant to Compliant in Azure AD after the issue is fixed?
After the compliance issue is fixed on the device, the device needs to sync with Intune (which can be forced), Intune then re-evaluates compliance (near real-time after sync), and then Azure AD receives the updated compliance signal — this can take 5 to 15 minutes. CA policies then evaluate the new compliance state on the next sign-in or token refresh. Tell users to wait 15 minutes after fixing the compliance issue and then try signing in again.
TOPIC 114
Using the Intune Troubleshooting Blade
The Underused Feature That Solves 80% of Helpdesk Tickets
What Is the Troubleshooting Blade?

The Intune Troubleshooting blade is a dedicated support view in the Intune portal that aggregates all relevant information about a specific user in one place. It was designed to help helpdesk technicians quickly diagnose issues without navigating through multiple menus.

Location: Intune portal → Troubleshooting + Support → Troubleshoot (or direct URL: endpoint.microsoft.com/#view/Microsoft_Intune_DeviceSettings/TroubleshootingDetailViewBlade)

What the Troubleshooting Blade Shows
👤
User Summary
User UPN, display name, Intune license status, last Intune activity. First check — if user has no Intune license, nothing else matters.
💻
Devices
All devices enrolled to this user — device name, OS, enrollment date, compliance status, last check-in. Click any device for deep details.
📦
App Installations
All apps assigned to the user and their install status per device — Installed, Failed, Not Applicable, Pending. Shows error codes for failures.
✅
Compliance Policies
All compliance policies assigned to the user, per device, with individual setting results. See exactly which setting caused Non-Compliance.
⚙️
Configuration Profiles
All configuration profiles applied to the user's devices with Succeeded/Error/Conflict/Not Applicable status for each.
🔐
App Protection Policies
MAM policy assignment and status — whether app protection policies are applied correctly to managed apps on personal devices.
How to Use It Effectively
  1. 1
    Search by user UPN. Type the user's email address. This is the fastest way to get a complete picture of one user's Intune state.
  2. 2
    Check license first. User summary section shows if the user has an Intune license. No license = nothing works. This is a 30-second first check that catches a surprisingly large number of issues.
  3. 3
    Check last check-in time. In the Devices section, check when the device last synced. If it was days ago, force a sync before investigating further — the issue may have resolved and just not been reported back.
  4. 4
    Use the app status table. App Installations section shows every assigned app with status per device. Filter by "Failed" to immediately jump to problem apps.
  5. 5
    Drill into compliance details. Click the compliance policy entry. It expands to show individual setting results — which specific setting is "Not compliant". This saves opening the MDMDiagReport for simple compliance questions.

Ticket: "User Jane Smith can't access SharePoint from her laptop — getting blocked."

Action: Open Troubleshooting blade → search "jane.smith@company.com".
→ Devices: Surface Pro, last check-in 2 days ago, compliance: Non-Compliant.

TOPIC 115
Common Real-World Tickets and Resolutions
10 Scenarios You'll Actually Face in Production

This topic brings everything together. These are real-world support tickets based on common enterprise Intune deployments, with structured resolution approaches.

Ticket 1: "My new laptop won't let me log in — it says device not recognized"
🎫
Ticket 1 — Device Not Recognized

Likely cause: Conditional Access policy requires compliant/managed device. New device is not yet enrolled or not yet compliant after Autopilot.
Resolution: Confirm Autopilot completed (ESP finished, desktop accessible). Check if device enrolled in Intune portal. Force sync. Wait 15 min for compliance to propagate to Azure AD. If Autopilot not used, manually enroll via Settings → Accounts → Access work or school.

Ticket 2: "Chrome was deployed to all devices last week but 12 devices still don't have it"
🎫
Ticket 2 — App Not On Some Devices

Investigation: Open Intune → Apps → Chrome → Device install status. Filter by "Failed" and "Pending." Check the 12 devices specifically.
Common findings: (a) 8 devices show "Pending" — haven't synced in 5+ days (offline devices or remote workers). Force sync or wait. (b) 4 devices show error 0x87D1FDE8 — open IME log on one device, find exit code 1603, discover those 4 devices have a corrupted VC++ redistributable that Chrome depends on. Fix prerequisite on those 4 devices.
Resolution: Different fix for different root causes — always segment the failing population first.

Ticket 3: "BitLocker key recovery option isn't working — can't find key in Azure AD"
🎫
Ticket 3 — BitLocker Key Not in Azure AD

Likely causes: (a) Device was encrypted before Intune enrollment — key backed up to old MDM or not backed up. (b) BitLocker policy configured to encrypt but not set to escrow recovery key to Azure AD. (c) Device encrypted manually and key never escrowed.
Resolution: Check Intune → Devices → [device] → Recovery keys. If not there: on the device run manage-bde -protectors -get C: to find the Key ID. Then force escrow: BackupToAAD-BitLockerKeyProtector -MountPoint C: -KeyProtectorId "{KEY-ID}". Review BitLocker policy to ensure "Save BitLocker recovery information to Azure Active Directory" is enabled.

Ticket 4: "PowerShell script deployed via Intune ran once but won't run again"
🎫
Ticket 4 — PowerShell Script Won't Re-run

Cause: By default, Intune PowerShell scripts run once per device. After successful execution, Intune marks it as "Success" and doesn't re-run it, even if you want it to run again.
Resolution: To force re-run: (a) Delete the script assignment and re-create it. (b) Or delete the device record from the script's device status — Intune will re-deliver next sync. (c) For scripts that need to run regularly, use Proactive Remediations (Detect + Remediate scripts) which run on a configurable schedule instead of once.

Ticket 5: "iPhone enrolled fine but Outlook shows 'Your organization is managing this device' — user upset"
🎫
Ticket 5 — User Concerned About Personal Device Management

Context: This is a communication/expectation issue more than a technical one. If the device is BYOD (user-enrolled), Intune only manages the work profile/apps — it cannot see personal data, apps, photos, or messages.
Resolution: Explain to the user what Intune can and cannot do on a BYOD device. On iOS BYOD: Intune manages only the apps deployed by IT and enforces MAM policies on those apps. It cannot wipe personal data — only a "selective wipe" that removes corporate apps and data. If the device was enrolled as ADE/DEP (supervised), then it is fully managed — confirm if this is appropriate or if user should be on BYOD instead.

Ticket 6: "Wi-Fi profile deployed but devices keep prompting for password"
🎫
Ticket 6 — Wi-Fi Profile Password Prompt

Likely cause: Certificate-based Wi-Fi authentication where the device certificate isn't deploying correctly, so it falls back to prompting for credentials. Or the SSID in the profile doesn't match the actual network name (case-sensitive).
Resolution: Check if a SCEP/PKCS certificate profile is assigned alongside the Wi-Fi profile and delivering successfully (check cert in device keystore). Verify SSID is exact match including case. For PSK Wi-Fi, re-enter the pre-shared key in the profile — the key could have been updated on the AP.

Ticket 7: "Compliance shows Non-Compliant for OS version but the device is up to date"
🎫
Ticket 7 — False Non-Compliance on OS Version

Cause: Compliance policy has a minimum OS version set (e.g., 22H2 10.0.19045.3803) but device reports a slightly different build. Also common when compliance policy was updated to require a newer build but device hasn't synced yet.
Resolution: Check exact OS build on device (winver) vs what the compliance policy requires. If device is actually compliant: force sync, allow 10 minutes. If the policy minimum version is too specific (patch Tuesday builds), consider setting the minimum version to the major feature update only (e.g., 10.0.19045 without the patch suffix).

Ticket 8: "Bulk enrolled devices show in Intune but policies aren't applying"
🎫
Ticket 8 — Bulk Enrolled Devices Without Policies

Cause: Devices bulk enrolled via Provisioning Package (PPKG) enroll without a user — they are "userless" device enrollments. If policies are assigned to user groups, they won't apply. Policies must be assigned to device groups.
Resolution: Move all policy and app assignments for bulk-enrolled devices to device groups (not user groups). Create a dynamic device group for these devices (filter by enrollment type or device name prefix). Reassign policies to the device group.

Ticket 9: "Android device enrolled but Google Play managed apps not showing"
🎫
Ticket 9 — Managed Google Play Apps Missing

Likely cause: Managed Google Play isn't approved/synced to Intune, or device is enrolled as a personal (BYOD) profile but app is set to require fully managed. Also, newly approved apps can take up to 24 hours to appear in the managed work profile.
Resolution: Check Intune → Apps → Android → the specific app → is it approved in Managed Google Play? Sync: Intune → Tenant admin → Connectors → Android Enterprise → Sync. Check the device's work profile (separate badged apps section) — apps appear there, not in the personal profile.

Ticket 10: "After a Windows Update, Defender ATP stopped reporting to Intune"
🎫
Ticket 10 — Defender ATP Integration Broken After Update

Cause: Windows Updates occasionally reset Defender for Endpoint's onboarding state, particularly on shared/reimaged devices. The device loses its MDE onboarding blob.
Resolution: Check Intune → Endpoint Security → Microsoft Defender for Endpoint → Connector status. If connector is healthy: on the device, check Services.msc for "Microsoft Defender Advanced Threat Protection Service" — is it running? Check Event Viewer → Microsoft → Windows → SENSE logs. If onboarding state is lost, re-deploy the MDE onboarding configuration profile from Intune (Endpoint Security → Endpoint Detection & Response). Device will re-onboard on next sync.

✅ The Pattern Behind All 10 Tickets

Every ticket above follows the same pattern: confirm what's actually happening (don't take the ticket at face value) → check assignment and sync → look at the specific error → fix at the root cause (not a workaround). The engineers who resolve tickets fastest are those who ask clarifying questions first and look at data (portal + logs) before touching anything.

🎤 Interview Questions — Topic 115
Tell me about a complex Intune issue you've resolved (or walk through how you'd handle one).
I walk through a structured approach: first clarify the exact symptom (which device, which user, what error, when started); check the Intune portal for assignment status and error codes; verify group membership and last sync time; read the relevant log (IME for apps, MDMDiagReport for policies, Event Viewer for enrollment); look up the error code; test in isolation; apply fix and verify. I always document the root cause and resolution so the team can learn from it.
A policy is assigned to a user group but isn't applying to bulk-enrolled devices. Why?
Bulk-enrolled devices (PPKG, or tokenless enrollment) don't have a user associated with them during enrollment. Policies assigned to user groups only apply when a user is associated with the device. For userless or shared devices, all policies must be assigned to device groups, not user groups. The fix is to create a device group for these devices and reassign the policies to that group.
How do you verify a BitLocker recovery key is escrowed to Azure AD from a Windows device?
Two ways: in the Intune portal, go to Devices → the device → Recovery keys — if the key is there, it's escrowed. Or on the device itself, run manage-bde -protectors -get C: to get the Key Protector ID, then use the PowerShell cmdlet BackupToAAD-BitLockerKeyProtector to force escrow. Confirm it appears in the portal after about 5 minutes.
