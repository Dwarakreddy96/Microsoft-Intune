Report Types in Intune
📋
Summary Reports
High-level dashboard views showing aggregate counts and percentages. Good for daily health checks and management briefings. Example: "750 compliant devices, 43 non-compliant, 12 not evaluated."
🔍
Detail Reports
Drilldown into specific devices, policies, or users. Filterable. Click from a summary view into the underlying detail. Example: from compliance summary → click Non-Compliant count → see every non-compliant device with the specific failing setting.
📤
Exportable Reports
Most Intune reports can be exported to CSV. Export → filter in Excel → share with management or compliance auditors. Schedule regular exports via Graph API for automated reporting pipelines.
⚡
Real-Time vs Cached
Some reports (device list) are near-real-time. Some reports (compliance aggregate) are cached and may lag 15–60 minutes. Be aware of this when investigating time-sensitive incidents — always trigger a device sync first.

🎤 Interview Questions — Topic 96
Where would you look in Intune to find which devices have BitLocker disabled?
Reports → Devices → Encryption report. This report shows every enrolled Windows device with its BitLocker protection status (Protected, Not Protected, Not Applicable) and FileVault status for macOS. You can filter to show only "Not Protected" devices, export to CSV for remediation follow-up, and click into any device to see the current encryption state and reason. Alternatively, Reports → Device Compliance → Setting compliance → filter by the BitLocker compliance setting shows devices failing the BitLocker compliance check.

🎤 Interview Questions — Topic 97
A manager asks: "What percentage of our devices have Outlook installed successfully?" How do you find this?
In Intune, navigate to Apps → find the Outlook app → click on it → Device install status (or User install status). This shows every device in the assignment group with the installation status: Installed, Failed, Pending, or Not Applicable. The summary at the top shows aggregate percentages. Export to CSV for the manager's report. If you want a fleet-wide app inventory view, Reports → Apps → App failure report shows aggregate installation success/failure rates across all deployed apps.
How do you use the "Setting compliance" report to prioritize remediation work?
The Setting compliance report shows each compliance setting individually with the count of devices that are compliant, non-compliant, and not applicable. Sort by non-compliant count descending to see which settings have the most failures across the fleet. If "BitLocker required" shows 200 non-compliant devices but "Password minimum length" shows only 5 non-compliant devices, you'd prioritize BitLocker remediation first since it affects more devices and has higher security impact. Export to CSV, sort by non-compliant count, and create a remediation prioritization list.
TOPIC 98
Log Analytics Integration — Sending Intune Logs to Azure Monitor
Extending beyond the Intune portal — custom queries, long-term retention, and real-time alerts
Why Integrate with Log Analytics?

The built-in Intune portal reports are good for daily operations but have limitations: 1-year data retention, limited filtering, no custom alerting, no cross-service correlation, and no way to build custom dashboards. Azure Monitor Log Analytics solves all of these. By streaming Intune diagnostic logs to a Log Analytics Workspace, you get:

  • KQL queries: Write custom queries against your Intune data — combine compliance data, device health, audit logs, and enrollment events in a single query.
  • Long-term retention: 90 days in Log Analytics hot storage, years in cold storage (Azure Storage). Required for many compliance frameworks.
  • Custom alerts: Alert when compliance drops below 90%, when a device is wiped, when a new admin role is assigned — real-time.
  • Cross-service correlation: Correlate Intune compliance data with Azure AD sign-in logs, Defender for Endpoint alerts, and network logs in a single query.
  • Power BI dashboards: Connect Log Analytics as a Power BI data source for executive-level dashboards (covered in Topic 105).
Setting Up Diagnostic Settings
  1. 1
    Intune → Tenant Administration → Diagnostic settings (or search "Diagnostic settings" in the portal)
  2. 2
    Click + Add diagnostic setting
  3. 3
    Select log categories to stream:
    • AuditLogs — All admin actions (create, delete, modify, device wipe)
    • OperationalLogs — Device check-ins, policy delivery status, enrollment events
    • DeviceComplianceOrg — Organization-level compliance aggregate data
    • Devices — Device inventory, hardware details, OS versions
    • IntuneDevices — Detailed per-device Intune management data
  4. 4
    Destination: Select Send to Log Analytics workspace → Choose or create a workspace → Save
  5. 5
    Logs begin flowing within 5–15 minutes. Query them in Log Analytics workspace → Logs blade.
Essential KQL Queries for Intune
// 1. All device wipe actions in the last 7 days IntuneAuditLogs | where TimeGenerated > ago(7d) | where OperationName contains "Wipe" | project TimeGenerated, Identity, ResultDescription, TargetDisplayNames | order by TimeGenerated desc // 2. Compliance trend — daily compliant % for last 30 days IntuneDeviceComplianceOrg | where TimeGenerated > ago(30d) | summarize Compliant = countif(ComplianceState == "compliant"), Total = count() by bin(TimeGenerated, 1d) | extend CompliancePct = round(100.0 * Compliant / Total, 1) | project TimeGenerated, CompliancePct | order by TimeGenerated asc // 3. Devices running old OS versions IntuneDevices | where TimeGenerated > ago(1d) | where OS == "Windows" | where OSVersion !startswith "10.0.22" // Not Windows 11 | project DeviceName, UserDisplayName, OSVersion, LastSyncDateTime | order by OSVersion asc // 4. BitLocker compliance by department (via AAD group) IntuneDeviceComplianceOrg | where TimeGenerated > ago(1h) | where SettingName == "bitLockerEnabled" | summarize Compliant = countif(State == "compliant"), NonCompliant = countif(State == "notCompliant") by PolicyName | extend PctCompliant = round(100.0 * Compliant / (Compliant + NonCompliant), 1) // 5. Failed enrollments with error codes IntuneOperationalLogs | where TimeGenerated > ago(7d) | where OperationName == "Enrollment" | where ResultType == "Failure" | parse Properties with * "ErrorCode:" ErrorCode "," * | summarize Count = count() by ErrorCode | order by Count desc

  1. 1
    In Log Analytics workspace → write the query you want to alert on (e.g., any wipe action)
  2. 2
    Click New alert rule → Configure condition: "Run every 5 minutes, alert if result count > 0"

🎤 Interview Questions — Topic 98
Why would you send Intune logs to Azure Monitor Log Analytics instead of just using the built-in Intune reports?
Built-in Intune reports have a 1-year retention limit, no custom alerting, limited filtering, and can't be correlated with data from other Microsoft services. Log Analytics provides: long-term retention (years) required by compliance frameworks, KQL queries to build custom analytics, real-time alerts when specific events occur, cross-service correlation (Intune + Azure AD + Defender in one query), and a data source for Power BI dashboards. For a mature enterprise, Log Analytics integration is essential for compliance auditing and proactive security monitoring.
Write a KQL query to find all device wipes in the last 24 hours.
IntuneAuditLogs | where TimeGenerated > ago(24h) | where OperationName contains "Wipe" | project TimeGenerated, Identity, ResultDescription, TargetDisplayNames | order by TimeGenerated desc. This queries the IntuneAuditLogs table (populated via Diagnostic Settings streaming), filters to the last 24 hours, matches any operation containing "Wipe", and shows the timestamp, who performed the action (Identity), the result, and the affected device names.

Microsoft Endpoint Analytics
Proactive device health insights — startup performance, reliability, and work-from-anywhere readiness
What Is Endpoint Analytics?

Endpoint Analytics is an Intune feature that collects and analyzes device health telemetry to give you proactive insights about your fleet's performance — before users call the helpdesk. It answers questions like "Why does John's laptop take 3 minutes to boot?" and "Which model of laptop crashes most in our fleet?" — and provides actionable recommendations.

Access it at: Intune → Reports → Endpoint Analytics

The Endpoint Analytics Scorecard

Endpoint Analytics gives your organization an overall score (0–100) and per-category scores based on your fleet's health vs industry benchmarks. The score appears on the Endpoint Analytics overview page and feeds into Microsoft Secure Score.

Score CategoryWhat It MeasuresTarget Score
Startup performanceTime from power-on to responsive desktop. Average boot time, sign-in time, and time to responsive desktop across your fleet.Above 50 (0–100 scale, higher = faster)
Application reliabilityApp crash and hang rates across managed devices. Which apps are causing the most user disruption.Above 50
Work from anywhereCloud management readiness — how many devices are Azure AD Joined, cloud-managed, VPN-independent. Measures cloud-native readiness.Above 60
Battery performanceBattery capacity vs designed capacity. How many devices have degraded batteries. Battery runtime predictions.Above 50
Resource performanceCPU and RAM utilization relative to device capabilities. Identifies underpowered devices for their workloads.Above 50
Startup Performance Deep Dive

The startup performance report is the most used — slow boot times are one of the top IT helpdesk complaints. Endpoint Analytics breaks startup time into components:

⚡
Core Boot Time
Time from power button to Windows logon screen. Includes firmware POST, Windows boot loader, kernel initialization. High core boot time → hardware issue (slow HDD vs SSD) or BIOS settings.
🔑
Group Policy + Login Scripts
Time to process Group Policy during sign-in. High GP processing time → too many GPOs, slow DC connectivity, complex login scripts. Review and optimize GPO count and scope.
🚀
Desktop Responsive Time
Time from sign-in complete to desktop being usable. High value → too many startup apps, Intune policies taking long to apply, antivirus full scan on boot. Review startup app list and AV scan scheduling.
Proactive Recommendations

Endpoint Analytics doesn't just show you the problem — it gives actionable recommendations. Examples:

  • "23 devices have HDDs — upgrade to SSD to reduce boot time by average 47 seconds" → Link to a report of specific devices to target for hardware refresh.
  • "Outlook is crashing on 85 devices — 78% are running version 16.0.xxx" → Update Outlook to current version. Link to deployment action.
  • "15 devices have less than 20% battery capacity remaining" → Battery replacement candidates. Export device list for procurement.
Enabling Endpoint Analytics
  • Intune-managed devices: Automatically enrolled in Endpoint Analytics when the device is enrolled in Intune. No additional policy needed.
  • SCCM-managed devices (without co-management): Deploy the Endpoint Analytics data collection script via SCCM to enroll non-Intune devices.
  • Data delay: New devices appear in Endpoint Analytics within 24–48 hours of enrollment. Scores update daily.
  • License: Included in Microsoft 365 E3/E5, Intune Plan 1.
🎤 Interview Questions — Topic 99
What is Endpoint Analytics and how does it help proactively improve device health?
Endpoint Analytics collects performance telemetry from managed devices and provides health scores across categories like startup performance, application reliability, battery health, and work-from-anywhere readiness. Unlike reactive monitoring (waiting for helpdesk tickets), Endpoint Analytics proactively identifies that 23 devices have HDDs causing slow boots, or that Outlook is crashing on 85 devices running a specific version — before users call the helpdesk. It provides specific recommendations with device lists for targeted remediation, quantifying the expected improvement so IT can prioritize budget and effort.
An executive laptop has a startup time of 4.5 minutes. How would you use Endpoint Analytics to investigate?
In Endpoint Analytics → Startup performance → Device performance tab → filter to the specific device or user. Review the breakdown: Core boot time (firmware/hardware), Group Policy processing time (GP and logon scripts), and Desktop responsive time (startup apps and post-login processing). If core boot time is high, likely an HDD vs SSD issue. If GP processing is high, too many GPOs or slow DC connectivity. If desktop responsive time is high, check startup apps list and whether AV is running a scan on login. Endpoint Analytics will also show comparisons to similar devices in the fleet, helping identify if this is device-specific or model-wide.
TOPIC 100
Proactive Remediation Scripts
Detect and automatically fix issues across thousands of devices without IT touching them
What Are Proactive Remediations?

Proactive Remediations (now called Remediations in the Intune portal) is a powerful automation feature that runs two paired PowerShell scripts on managed Windows devices on a schedule: a detection script that checks for an issue, and a remediation script that fixes it if found. The scripts run silently in the background — users never see anything.

This creates a self-healing fleet: instead of waiting for users to call the helpdesk about problems, Intune detects and fixes them automatically.

How It Works — The Detection/Remediation Pattern
🔍
Detection Script
Runs first. Checks if the issue exists. Must exit with code 0 (no issue found — no remediation needed) or 1 (issue found — run remediation script). Can also produce output text (shown in Intune reports).
🔧
Remediation Script
Only runs when detection exits with code 1. Fixes the detected issue. Must exit with code 0 (fixed successfully) or 1 (fix failed). Results reported back to Intune with success/failure per device.
Real-World Proactive Remediation Examples
################################ # Example 1: Detect and fix Windows Time Service ################################ # DETECTION SCRIPT (save as detect.ps1) $service = Get-Service -Name W32Time -ErrorAction SilentlyContinue if ($service.Status -eq "Running") { # Service is running — no issue exit 0 } else { Write-Output "Windows Time Service is not running" exit 1 # Issue found — run remediation } # REMEDIATION SCRIPT (save as remediate.ps1) try { Start-Service -Name W32Time Set-Service -Name W32Time -StartupType Automatic w32tm /resync /force Write-Output "Time service started and synced" exit 0 # Fixed successfully } catch { Write-Output "Failed to start time service: $($_.Exception.Message)" exit 1 # Fix failed }
################################ # Example 2: Detect and clear stuck Temp folder ################################ # DETECTION SCRIPT $tempSize = (Get-ChildItem $env:TEMP -Recurse -ErrorAction SilentlyContinue | Measure-Object -Property Length -Sum).Sum / 1GB if ($tempSize -gt 5) { # More than 5GB in Temp Write-Output "Temp folder is $([math]::Round($tempSize,2))GB — needs cleanup" exit 1 } else { exit 0 } # REMEDIATION SCRIPT Remove-Item "$env:TEMP\*" -Recurse -Force -ErrorAction SilentlyContinue $diskUsage = (Get-WmiObject Win32_LogicalDisk -Filter "DeviceID='C:'" | Select-Object @{n='FreeGB';e={[math]::Round($_.FreeSpace/1GB,1)}}).FreeGB Write-Output "Temp cleared. Free space now: ${diskUsage}GB" exit 0

These are production-tested, ready-to-deploy scripts. Use them as starting points and customize as needed — no need to write everything from scratch.

🎤 Interview Questions — Topic 100
What is Proactive Remediation in Intune and how does the detection/remediation pattern work?
Proactive Remediation runs two paired PowerShell scripts on Windows devices on a schedule. The detection script checks if an issue exists — it exits with code 0 if no issue, code 1 if an issue is found. If exit code 1, Intune runs the remediation script to fix the issue. The remediation script exits 0 for success, 1 for failure. Results per device are visible in Intune — you can see which devices had the issue detected, which were successfully remediated, and which failed remediation with the script's output text. This creates self-healing devices that fix common issues without helpdesk involvement.
Give a real-world example where Proactive Remediation would save significant IT time.
OneDrive sync stopping silently is extremely common — users don't notice until they realize files weren't backed up or accessible on another device. A Proactive Remediation runs every hour: detection script checks if OneDrive process is running and the last sync was within 2 hours. If either fails, the remediation script kills and restarts the OneDrive process. Instead of users calling the helpdesk because "my files aren't syncing," the issue is detected and fixed automatically within an hour — often before the user notices. For an organization with 1,000 users where this happens 5 times per user per year, that's 5,000 potential helpdesk tickets automated away.

Deploying a PowerShell Script via Intune
  1. 1
    Intune → Devices → Scripts (under Windows section) → Add → Windows 10 and later
  2. 2
    Upload your .ps1 script file
  3. 3
    Configure execution settings:
    • Run script as signed-in user: No (SYSTEM context — for device-wide config) / Yes (user context — for per-user settings)
    • Enforce script signature check: No (typical for internal scripts) / Yes (enterprise security requirement)
    • Run script in 64-bit PowerShell host: Yes (always recommended — avoids 32-bit compatibility issues)
  4. 4
    Assign to device or user groups → Deploy
  5. 5
    Monitor: Devices → Scripts → Find your script → Device status — shows per-device: Pending, Success, Failed. Click a failed device to see the error output.
Important Script Execution Notes
  • Script runs once by default: After successful execution, the script doesn't re-run. If you need recurring execution, use Proactive Remediations instead.
  • No user interaction: Scripts run silently in the background. Any Write-Host or Read-Host in the script won't display to users — use Write-Output for logging (visible in Intune output).
  • 30-minute timeout: Scripts that take longer than 30 minutes are terminated and marked as failed.
  • Output limit: Only the last 2KB of script output is stored in Intune. Use a log file for detailed output.
  • Execution context: SYSTEM context has elevated privileges but no network credentials for accessing network resources. User context has the user's network credentials.
# Best practice: Log script output to a file AND output to Intune $LogPath = "C:\ProgramData\Intune\Logs\ConfigScript.log" function Write-Log { param([string]$Message) $Timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss" $LogEntry = "$Timestamp - $Message" Add-Content -Path $LogPath -Value $LogEntry Write-Output $LogEntry } New-Item -Path (Split-Path $LogPath) -ItemType Directory -Force | Out-Null Write-Log "Script started" try { # Your configuration logic here Write-Log "Configuration applied successfully" exit 0 } catch { Write-Log "Error: $($_.Exception.Message)" exit 1 }
🎤 Interview Questions — Topic 101
What is the difference between PowerShell Script deployment and Proactive Remediations in Intune?
PowerShell Scripts run once (on first deployment, then only if it previously failed) and are best for one-time configuration tasks. Proactive Remediations run on a recurring schedule with two scripts — detect and remediate — and only fix issues when the detection script finds them. Use PowerShell Scripts for initial setup tasks (configuring registry keys, deploying config files, one-time security hardening). Use Proactive Remediations for ongoing health monitoring and auto-healing (OneDrive sync issues, Defender RTP getting disabled, disk cleanup needed).

Just like PowerShell scripts for Windows, Intune supports deploying Shell scripts (bash/zsh) to macOS devices. The same use cases apply: one-time configuration tasks, security hardening, software configuration, and post-enrollment setup that can't be done via configuration profiles alone.

#!/bin/bash # Example 1: Configure macOS time zone and NTP ################################ # Set timezone to Asia/Kolkata systemsetup -settimezone "Asia/Kolkata" # Enable NTP and set server systemsetup -setusingnetworktime on systemsetup -setnetworktimeserver "time.apple.com" echo "Timezone and NTP configured" exit 0
#!/bin/bash # Example 2: Install company root CA certificate ################################ CERT_URL="https://internal.company.com/RootCA.cer" CERT_PATH="/tmp/CompanyRootCA.cer" # Download and install certificate to System keychain curl -s -o "$CERT_PATH" "$CERT_URL" if [ -f "$CERT_PATH" ]; then security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain "$CERT_PATH" echo "Root CA certificate installed successfully" exit 0 else echo "Failed to download certificate" exit 1 fi

🎤 Interview Questions — Topic 102
How do macOS Shell scripts in Intune differ from Windows PowerShell scripts?
macOS Shell scripts natively support recurring execution frequency settings (daily, weekly, etc.) directly in the script deployment configuration — Windows PowerShell scripts only run once by default (use Proactive Remediations for recurring Windows tasks). Shell scripts run as root when "run as signed-in user" is disabled. The maximum execution time is 15 minutes for macOS vs 30 minutes for Windows. macOS scripts use bash or zsh; Windows scripts use PowerShell. Both are deployed silently through the Intune Management Extension (macOS) or the Intune Management Extension agent (Windows).

What Is the Microsoft Graph API?

The Microsoft Graph API is a unified REST API that provides programmatic access to all Microsoft 365 services — including Intune, Azure AD, Exchange, SharePoint, Teams, and more. Everything you can do in the Intune portal has a corresponding API endpoint. This means you can: query device data, create policies, assign apps, trigger remote actions, pull reports — all programmatically, without clicking through the UI.

Base URL for all Graph API calls: https://graph.microsoft.com/v1.0/ or https://graph.microsoft.com/beta/ (for preview features)

Graph Explorer — Test Before You Code

Graph Explorer (aka.ms/ge) is Microsoft's browser-based tool for testing Graph API calls. Sign in with your work account, and you can make live API calls against your tenant — no code needed. Perfect for learning the API and testing queries before building automation.

OData Filtering and Querying

Graph API supports OData query parameters for filtering, selecting, ordering, and paginating results:

# Filter: Get only non-compliant Windows devices GET https://graph.microsoft.com/v1.0/deviceManagement/managedDevices ?$filter=complianceState eq 'noncompliant' and operatingSystem eq 'Windows' # Select: Get only specific fields (reduces response size) GET https://graph.microsoft.com/v1.0/deviceManagement/managedDevices ?$select=deviceName,userDisplayName,complianceState,osVersion,lastSyncDateTime # Expand: Include related objects in response GET https://graph.microsoft.com/v1.0/deviceManagement/managedDevices ?$expand=deviceCompliancePolicyStates # Top + Skip: Pagination (Intune returns max 1000 per page) GET https://graph.microsoft.com/v1.0/deviceManagement/managedDevices ?$top=100&$skip=200 # OrderBy: Sort results GET https://graph.microsoft.com/v1.0/deviceManagement/managedDevices ?$orderby=lastSyncDateTime desc
Required Graph API Permissions

To call Intune Graph API endpoints from an app or script, you need an Azure AD app registration with the appropriate Microsoft Graph permissions:

  • DeviceManagementManagedDevices.Read.All — Read device data
  • DeviceManagementManagedDevices.ReadWrite.All — Read and write device data, trigger remote actions
  • DeviceManagementConfiguration.ReadWrite.All — Read and write configuration policies
  • DeviceManagementApps.ReadWrite.All — Read and write app management
  • DeviceManagementRBAC.ReadWrite.All — Read and write RBAC roles and assignments
  • Permission types: Application permissions (no user sign-in, uses app secret — for automation) vs Delegated permissions (uses signed-in user's permissions — for interactive tools)

Authentication — Getting an Access Token

Before calling Graph API from PowerShell, you need an access token. Two approaches:

################################ # Method 1: Microsoft Graph PowerShell SDK (Recommended) # Install: Install-Module Microsoft.Graph -Scope CurrentUser ################################ # Connect interactively (for scripts you run manually) Connect-MgGraph -Scopes "DeviceManagementManagedDevices.ReadWrite.All" # Connect with App Registration (for scheduled automation) $TenantId = "your-tenant-id" $ClientId = "your-app-client-id" $ClientSecret = "your-app-secret" # Store in Azure Key Vault in production! $SecureSecret = ConvertTo-SecureString $ClientSecret -AsPlainText -Force $Credential = New-Object System.Management.Automation.PSCredential($ClientId, $SecureSecret) Connect-MgGraph -TenantId $TenantId -ClientSecretCredential $Credential
Practical Automation Script Examples
################################ # Script 1: Get all non-compliant devices and export to CSV ################################ Connect-MgGraph -Scopes "DeviceManagementManagedDevices.Read.All" $NonCompliant = Get-MgDeviceManagementManagedDevice ` -Filter "complianceState eq 'noncompliant'" ` -Select "deviceName,userDisplayName,osVersion,complianceState,lastSyncDateTime,operatingSystem" ` -All # -All handles pagination automatically $NonCompliant | Select-Object DeviceName, UserDisplayName, OsVersion, LastSyncDateTime | Export-Csv "C:\Reports\NonCompliant_$(Get-Date -f 'yyyyMMdd').csv" -NoTypeInformation Write-Output "Exported $($NonCompliant.Count) non-compliant devices"
################################ # Script 2: Sync all devices that haven't checked in for 48 hours ################################ Connect-MgGraph -Scopes "DeviceManagementManagedDevices.ReadWrite.All" $CutoffTime = (Get-Date).AddHours(-48).ToUniversalTime() $StaleDevices = Get-MgDeviceManagementManagedDevice ` -Filter "lastSyncDateTime lt $($CutoffTime.ToString('yyyy-MM-ddTHH:mm:ssZ'))" ` -Select "id,deviceName,lastSyncDateTime" ` -All Write-Output "Found $($StaleDevices.Count) devices not synced in 48h" foreach ($Device in $StaleDevices) { try { Invoke-MgDeviceManagementManagedDeviceSyncDevice -ManagedDeviceId $Device.Id Write-Output "Synced: $($Device.DeviceName)" } catch { Write-Output "Sync failed for $($Device.DeviceName): $($_.Exception.Message)" } }
################################ # Script 3: Generate weekly compliance report email ################################ Connect-MgGraph -Scopes "DeviceManagementManagedDevices.Read.All" $AllDevices = Get-MgDeviceManagementManagedDevice -All $Compliant = ($AllDevices | Where-Object {$_.ComplianceState -eq "compliant"}).Count $NonCompliant = ($AllDevices | Where-Object {$_.ComplianceState -eq "noncompliant"}).Count $NotEvaluated = ($AllDevices | Where-Object {$_.ComplianceState -eq "unknown"}).Count $Total = $AllDevices.Count $PctCompliant = [math]::Round(($Compliant / $Total) * 100, 1) $Report = @" Weekly Intune Compliance Report - $(Get-Date -f 'dd-MMM-yyyy') Total Enrolled Devices : $Total Compliant : $Compliant ($PctCompliant%) Non-Compliant : $NonCompliant Not Evaluated : $NotEvaluated Top Non-Compliant OS Versions: $($AllDevices | Where-Object {$_.ComplianceState -eq 'noncompliant'} | Group-Object OsVersion | Sort-Object Count -Descending | Select-Object -First 5 | ForEach-Object {" $($_.Name): $($_.Count) devices"} | Out-String) "@ # Send via Microsoft Graph Mail API or save to SharePoint $Report | Out-File "C:\Reports\WeeklyCompliance_$(Get-Date -f 'yyyyMMdd').txt" Write-Output $Report
Scheduling Automation with Azure Automation or Logic Apps
  • Azure Automation Runbooks: Host PowerShell scripts in Azure Automation. Run on a schedule (daily, weekly). Uses Managed Identity for authentication — no stored credentials. Best for recurring Intune operations.
  • Azure Logic Apps: Visual workflow automation. Trigger on event (new device enrolled, compliance status changed) → run Graph API calls → send Teams notification or create ServiceNow ticket. No code required.
  • GitHub Actions / Azure DevOps: For IT teams using DevOps practices — deploy Intune policies as code, version-control your configuration, automate policy deployment via CI/CD pipelines.

How would you automate a weekly non-compliant device report using Graph API and PowerShell?
Create an Azure AD App Registration with DeviceManagementManagedDevices.Read.All permission. Write a PowerShell script using the Microsoft Graph PowerShell SDK that connects with the app's client credentials, calls Get-MgDeviceManagementManagedDevice -Filter "complianceState eq 'noncompliant'" -All to retrieve all non-compliant devices, formats the results, and exports to CSV or sends an email via Graph Mail API. What is the Microsoft Graph PowerShell SDK and why use it over raw REST calls?
The Microsoft Graph PowerShell SDK (Install-Module Microsoft.Graph) wraps all Graph API endpoints as native PowerShell cmdlets — Get-MgDeviceManagementManagedDevice instead of Invoke-RestMethod with manual URL construction, headers, and response parsing. Benefits: automatic pagination handling with -All parameter, strongly typed response objects with IntelliSense, consistent authentication with Connect-MgGraph, built-in error handling, and much cleaner code. For complex automation with many API calls, the SDK is significantly less code and more maintainable than raw REST calls.

Intune + Power BI Reporting
Building executive-grade dashboards from Intune and Log Analytics data
Why Power BI for Intune Reporting?

The Intune portal reports are operational — designed for IT admins to investigate specific issues. Power BI is for management and executive reporting — visualizing trends over time, comparing departments, and showing security posture improvements to leadership. A well-designed Power BI dashboard communicates in 30 seconds what takes 10 minutes of navigating Intune reports to explain.

Data Sources for Intune Power BI
📊
Log Analytics Workspace
Connect Power BI to your Log Analytics workspace (where Intune diagnostic logs stream). Write KQL queries in Power BI, schedule refresh. Best for real-time operational dashboards with complex filtering and cross-service data.
🔌
Intune Data Warehouse
The Intune Data Warehouse is a structured data store of historical Intune data — more stable schema than raw logs. Access via the Intune Data Warehouse OData feed URL. Best for trend analysis over months/years.
⚡
Graph API (Power Query)
Use Power BI's Power Query to call Graph API endpoints directly. Flexible — query any Intune data. Manual pagination handling required. Best for specific custom datasets not available via Log Analytics or Data Warehouse.
📁
CSV Exports
Export CSVs from Intune reports and import into Power BI. Simplest approach. Not real-time — manual refresh needed. Good for one-off analysis or when automation isn't yet set up.

What are the data source options for building Intune Power BI dashboards?
Four main options: (1) Intune Data Warehouse via OData feed — structured historical data, best for trend analysis, most stable schema. What Intune metrics would you include in a CISO executive dashboard?

Each metric should have a trend direction indicator and comparison to previous period — executives need context not just current numbers.
